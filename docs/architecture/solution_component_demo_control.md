# C9: демонстрация, контроль и защитные материалы `SolutionComponent`

## Назначение

Документ фиксирует результат финального этапа C9 из `solution_component_editor_roadmap.md`: редактор компонентов теперь можно показать как завершённый и объяснимый сценарий дипломного проекта, а не как отдельную экспериментальную вкладку.

C9 не добавляет новый аналитический метод. Он связывает уже реализованные части C1–C8 в проверяемый демонстрационный маршрут:

```text
SolutionComponent editor
  → нормализация и warnings
  → runtime-хранение
  → CandidateConfiguration
  → TCO / energy / NPV basis
  → AHP / GA / GA + AHP adapters
  → DecisionReport / JSON / Markdown / CSV
```

## Демонстрационная фикстура

Финальный набор лежит в `data/fixtures/solution_component_c9_demo_dataset.json`. Он отделён от основного `demo_dataset.json`, чтобы защитный сценарий редактора можно было проверять отдельно и не смешивать с учебными примерами AHP/GA.

В набор входят четыре показательных компонента.

| Компонент | Что демонстрирует | Ожидаемое поведение |
|---|---|---|
| `c9-mixed-server-hypervisor-setup` | Составной комплект «сервер + гипервизор + базовая настройка» | Попадает в TCO и DecisionReport как strict mixed-компонент; энергетическая часть разрешена через `technical_part` |
| `c9-software-subscription-support` | Программная подписка с сопровождением | Участвует в ПО-профиле и TCO; не участвует в электроэнергии |
| `c9-peripheral-mfu-no-client-seat` | Периферия, которая не занимает клиентское место | Участвует в техническом профиле, но сохраняет `client_seats = 0` |
| `c9-draft-unclassified-training` | Черновик с неполной классификацией | Сохраняется и показывается в отчёте, но не попадает в `CandidateConfiguration` |

## Контрольный сценарий

Проверка запускается без `pytest` и не должна создавать bytecode-кэш:

```bash
PYTHONDONTWRITEBYTECODE=1 python -B scripts/run_solution_component_c9_demo.py
```

Сценарий проверяет, что:

- все четыре демонстрационных компонента читаются из fixture;
- три strict-компонента попадают в общий кандидатный набор;
- черновик исключён из `CandidateConfiguration` и имеет warnings/errors;
- mixed-комплект может участвовать в энергетической части;
- программная подписка не считается энергетическим компонентом;
- МФУ не превращается в клиентское рабочее место;
- `DecisionReport` показывает валидные и исключённые компоненты;
- отдельный CSV компонентов покрывает все строки редактора.

Ожидаемые значения вынесены в `data/fixtures/regression/solution_component_c9_invariants.json`.

## Защитная формулировка

Коротко идею редактора можно объяснять так:

> В проекте сохранены быстрые вкладки ПО и ТО для типовых строк. Редактор `SolutionComponent` нужен для нестандартных и составных компонентов, например комплекта «сервер + гипервизор + базовая настройка». Такой компонент не создаёт вторую модель данных: он проходит нормализацию, получает `scope`, `component_type`, warnings и только после допуска попадает в уже существующие `CandidateConfiguration`, TCO, AHP/GA и `DecisionReport`. Черновики сохраняются, но не влияют на расчёты.

## Почему это закрывает C9

Финальный этап считается закрытым, потому что теперь есть:

- отдельная демонстрационная fixture для редактора;
- воспроизводимый smoke-сценарий без `pytest`;
- документация для защиты и объяснения роли редактора;
- обновлённая навигация по demo-материалам;
- явная фиксация того, что `SolutionComponent` развивает связанную архитектуру проекта, а не возвращает её к набору независимых вкладок.
