# Генетический алгоритм подбора конфигурации

Этот документ описывает текущее состояние генетического алгоритма в проекте и объясняет, как он связан с задачей подбора конфигурации ИТ-инфраструктуры.

Важно: ядро алгоритма сделано универсальным. Оно не знает о серверах, рабочих станциях, сетевом оборудовании, лицензиях, CAPEX или OPEX. Предметный смысл задаётся снаружи через критерии, ограничения и прикладной слой.

## Назначение

Генетический алгоритм используется для автоматического поиска допустимой комбинации элементов из заданного набора кандидатов.

В рамках дипломного приложения это означает:

1. пользователь заполняет или загружает данные об оборудовании;
2. прикладной слой превращает runtime-записи в универсальные элементы оптимизации;
3. пользователь задаёт ограничения и веса критериев;
4. ГА ищет несколько допустимых конфигураций;
5. лучший результат и top-N кандидатов показываются в блоке **GA** внутри рабочей области ПО/ТО;
6. пользователь явно передаёт top-N в общий пул, после чего AHP, Pareto и Hybrid работают с теми же альтернативами.

При этом сам алгоритм можно использовать и для других предметных областей: достаточно передать другой набор item-ов, критериев и ограничений.

## Постановка задачи

Есть множество элементов:

```text
I = {i1, i2, ..., in}
```

Нужно выбрать подмножество элементов:

```text
S ⊆ I
```

Такое, чтобы:

- все жёсткие ограничения были выполнены;
- агрегированная оценка по критериям была как можно выше;
- результат был воспроизводимым при фиксированном `seed`;
- пользователь мог увидеть не только один победивший вариант, но и несколько лучших кандидатных решений.

Для задачи ИТ-инфраструктуры подмножество `S` интерпретируется как состав конфигурации: серверы, клиентские устройства, сетевое оборудование, лицензии и другие позиции. Но эта интерпретация появляется только в прикладном слое, а не в ядре ГА.

## Универсальный item

Элемент оптимизации — это объект с произвольными свойствами. Для совместимости используется модель `Item` из `domain.optimization.mech`, а application layer использует лёгкий объект `RuntimeOptimizationItem` с полем `properties`.

Пример:

```python
Item(
    name="Server_A",
    properties={
        "cost": 4000,
        "energy": 500,
        "perf": 700,
        "category": "server",
    },
)
```

Схема свойств не фиксирована. Можно передать любые поля, если критерии и ограничения знают, как их читать.

## Структура хромосомы

Хромосома представлена бинарной маской:

```text
(1, 0, 1, 0, 1)
```

Каждый бит соответствует одному item-у:

- `1` — элемент включён в конфигурацию;
- `0` — элемент не выбран.

Например, если есть пять кандидатов, маска `(1, 0, 1, 0, 1)` означает выбор первого, третьего и пятого элемента.

Такое представление подходит для задачи выбора подмножества и не зависит от конкретной предметной области.

## Критерии

Новый контракт ГА принимает список критериев:

```python
criteria = [
    {
        "name": "performance",
        "func": lambda subset: sum(item.properties["perf"] for item in subset),
        "direction": "max",
    },
    {
        "name": "capital_cost",
        "func": lambda subset: sum(item.properties["cost"] for item in subset),
        "direction": "min",
    },
]
```

Каждый критерий содержит:

- `name` — имя критерия для отчёта и UI;
- `func` — функция оценки выбранного подмножества;
- `direction` — направление оптимизации: `max` или `min`.

Для обратной совместимости сохранён старый контракт:

```python
criteria_funcs = [performance, total_energy, total_cost]
optimize_directions = [1, -1, -1]
```

## Веса критериев

ГА поддерживает три источника весов:

| Источник | Когда используется |
|---|---|
| `weights` | Если веса заданы напрямую |
| `pairwise_matrix` | Если веса нужно получить из AHP-матрицы попарных сравнений |
| равные веса | Если ни `weights`, ни `pairwise_matrix` не переданы |

Веса нормализуются так, чтобы их сумма была равна `1`.

## Функция приспособленности

Главная методическая правка текущей версии — отказ от сложения сырых значений разных размерностей.

Раньше критерии вроде стоимости, мощности и производительности могли смешиваться напрямую. Это приводило к перекосу: критерий с большим числовым масштабом мог подавлять остальные.

Сейчас расчёт выполняется так:

1. вычисляются исходные значения критериев;
2. для критериев минимизации значение умножается на `-1`, чтобы после преобразования для всех критериев действовало правило «чем больше — тем лучше»;
3. значения нормализуются в диапазон `0..1`;
4. итоговая fitness-оценка считается как взвешенная сумма нормализованных значений.

Упрощённо:

```text
fitness(S) = Σ weight_j * normalized_j(S)
```

Итоговое поле результата:

```text
fitness_scale = "normalized_0_1"
```

Благодаря этому стоимость, энергопотребление, производительность и другие критерии можно сравнивать совместно без грубого масштабного перекоса.

## Ограничения

ГА поддерживает список произвольных ограничений:

```python
constraints = [
    {
        "name": "budget",
        "func": lambda subset: sum(item.properties["cost"] for item in subset),
        "operator": "<=",
        "bound": 10000,
    },
    {
        "name": "has_server",
        "func": lambda subset: count_category(subset, "server"),
        "operator": ">=",
        "bound": 1,
    },
]
```

Поддерживаются операторы:

```text
<=, >=, <, >, ==, !=
```

Также можно передать `predicate`, если ограничение удобнее выразить как булеву функцию.

Старый контракт `constraint_func + max_constraint` сохранён и автоматически превращается в ограничение вида:

```text
legacy_constraint <= max_constraint
```

## Инициализация популяции

Поддерживаются режимы:

| Режим | Смысл |
|---|---|
| `random_init` | Стартовая популяция собирается случайно |
| `heuristic_init` | Используются одиночные допустимые решения, простые эвристики и случайное заполнение |
| `pareto_seeded_init` | Для совместимых малых сценариев старт можно обогатить результатами переборного механизма |

Режим по умолчанию:

```text
heuristic_init
```

`KnapsackMechanic` больше не является обязательной частью запуска. Он используется только как опциональный источник стартовых решений в совместимом режиме `pareto_seeded_init`.

## Операторы генетического алгоритма

В текущей реализации используются:

- **отбор** — турнирный отбор;
- **скрещивание** — одноточечное скрещивание;
- **мутация** — инверсия бита с вероятностью `mutation_rate`;
- **repair** — восстановление недопустимой маски до ближайшего допустимого решения;
- **элитизм** — перенос лучших решений в следующее поколение;
- **ранняя остановка** — остановка по `stagnation_limit`, если лучший score долго не улучшается.

Причина остановки возвращается в поле:

```text
termination_reason
```

Возможные значения:

- `max_generations`;
- `stagnation`;
- `no_feasible_solution`.

## История поколений

Результат содержит `history` — список записей по поколениям:

```text
generation
best_score
generation_best_score
mean_score
feasible_count
best_mask
generation_best_normalized_scores
```

Эти данные используются в блоке **GA** внутри рабочих областей ПО/ТО для отображения сходимости.

## Несколько лучших решений

ГА возвращает не только одно лучшее решение, но и список лучших кандидатных решений:

```text
top_solutions
candidate_solutions
```

`top_solutions` формируется в ядре ГА, а `candidate_solutions` — в прикладном сервисе, где к решениям добавляются totals, группировка по категориям и данные, удобные для UI/экспорта.

Это нужно для связки с AHP: AHP ранжирует не все возможные комбинации, а несколько лучших допустимых кандидатов, найденных ГА.

## Связь с AHP, Pareto и Hybrid после рефакторинга

Текущая пользовательская цепочка больше не выглядит как одна кнопка `GA + AHP` внутри GA-панели. Роли разделены:

```text
runtime-данные → GA → общий пул CandidateConfiguration → AHP / Pareto → Hybrid → DecisionReport
```

Разделение ответственности такое:

- **GA** ищет допустимые и перспективные конфигурации внутри выбранной области ПО/ТО;
- **AHP** самостоятельно ранжирует общий пул альтернатив по профильным критериям области;
- **Pareto** проверяет компромиссы и недоминируемые варианты;
- **Hybrid** не запускает GA или AHP заново, а согласует последние результаты методов;
- `DecisionReport` фиксирует источник пула и отдельные результаты методов.

В UI это выглядит так: пользователь запускает GA, нажимает передачу `top-N` в общий пул, затем при необходимости рассчитывает AHP, Pareto и обновляет Hybrid-витрину. Старый `GeneticAhpRankingService` сохранён как совместимый слой для тестов и старых JSON-экспортов, но не является основной пользовательской формулой.

### Независимая AHP-оценка GA-кандидатов

AHP получает уже сформированный пул `CandidateConfiguration`. Для кандидатов сохраняются исходные `totals`, `metrics`, `metadata`, а также источник пула. Это позволяет объяснить, что AHP не генерирует инфраструктуру и не является частью GA: он только ранжирует переданные альтернативы.

### Гибридная итоговая оценка

Гибридная оценка строится после независимых результатов. Базовая формула остаётся простой:

```text
H_i = λ × Ĝ_i + (1 - λ) × Â_i
```

Где `Ĝ_i` и `Â_i` — нормированные GA-score и AHP-score внутри одного пула. Pareto не добавляется как скрытый вес: он отображается как статус или предупреждение о доминировании.

Такой подход проще защищать: итоговая рекомендация не маскирует конфликт методов, а показывает, совпали ли поисковый, экспертный и компромиссный взгляды.

## Прикладной сценарий для ИТ-инфраструктуры

`GeneticOptimizationService` читает runtime-сущности из репозитория и превращает строки оборудования в универсальные кандидаты.

Оригинальные поля записей сохраняются. Дополнительно добавляются служебные поля:

```text
source_category
source_index
total_cost
capital_cost
unit_price
selected_count
```

Во вкладке **«Генетический подбор»** пользователь может задать:

- размер популяции;
- число поколений;
- вероятность мутации;
- `seed`;
- бюджет;
- максимальную мощность;
- минимальное число пользователей/клиентских мест или лицензий;
- трёхсостоянийный фильтр категорий `+ / − / нейтрально`;
- веса оставшихся soft-критериев.

Текущие UI-критерии для ТО:

| Критерий | Направление |
|---|---|
| `total_ram_gb` | max |
| `total_cpu_cores` | max |
| `total_storage_gb` | max |
| `total_power_watts` | min |
| `capital_cost` | min |

Текущие UI-критерии для ПО:

| Критерий | Направление |
|---|---|
| `functionality_score` | max |
| `support_score` | max |
| `capital_cost` | min |

`category_coverage` больше не участвует в fitness-функции как soft-критерий. Категории теперь работают только как ограничения: обязательная категория должна присутствовать, исключённая не должна попадать в решение, нейтральная не влияет на допустимость.

Мощность берётся из поля `max_power` у строк оборудования и из данных вкладки **«Электроэнергия»**, если пользователь вручную уточнил соответствующие значения.

Для клиентской ёмкости приоритетно используется отдельное поле `client_seats`. Это нужно, чтобы не путать количество единиц оборудования с количеством рабочих мест: например, МФУ или принтер может находиться в категории `client`, но иметь `client_seats = 0` и не увеличивать число клиентских мест. Если поле `client_seats` отсутствует, старое поведение сохраняется: для строк категории `client` используется `quantity`.

## Демонстрационный сценарий

Демонстрационный knapsack-пайплайн находится в:

```text
src/domain/optimization/pipeline_knapsack.py
```

Он использует набор серверного, сетевого и клиентского оборудования, бюджетное ограничение и три критерия:

- производительность — max;
- энергопотребление — min;
- стоимость — min.

Регрессионный тест фиксирует, что после перехода на нормализованную fitness-функцию ГА не должен деградировать к выбору одиночного дешёвого `Switch_A`, а должен находить более полноценную конфигурацию в рамках бюджета.

## Ограничения текущей реализации

Текущая версия уже пригодна для демонстрационного сценария, но остаётся MVP-реализацией. Основные ограничения:

1. Нормализация выполняется по стартовой/рабочей области поиска, а не по заранее известному полному пространству решений.
2. В `pareto_seeded_init` переборный механизм применим только к совместимым сценариям с одним ограничением `<=` и AHP-матрицей.
3. Нет отдельного визуального графика сходимости, пока история показывается таблично.
4. UI-сценарий использует ограниченный набор прикладных критериев, хотя ядро поддерживает произвольные критерии.
5. ГА не гарантирует глобальный оптимум, так как это эвристический метод.
6. Экспорт результата ГА подготовлен как payload, но не полностью объединён с итоговым CSV-отчётом всех вкладок.
7. Качество результата зависит от параметров `pop_size`, `generations`, `mutation_rate`, ограничений и качества исходных данных.

## Итог

ГА в проекте выполняет роль универсального поискового ядра для подбора комбинаций. Для дипломной логики он закрывает задачу автоматического формирования кандидатных конфигураций, а AHP дополняет его объяснимым ранжированием. Такая связка позволяет показать не только расчёт стоимости инфраструктуры, но и движение проекта к системе поддержки принятия решений.

## Диагностическая проверка полным перебором

Для малых наборов данных ГА теперь формирует диагностический блок `quality_check`.
Это не альтернативный алгоритм выбора конфигурации, а самопроверка качества:
система перебирает все возможные маски, оставляет только допустимые решения и
сравнивает точный результат с тем, что нашёл ГА.

Проверка включается автоматически только если число item-ов не превышает
`quality_check_max_items` (по умолчанию 18). Это ограничение нужно потому, что
полный перебор растёт экспоненциально: для 10 item-ов нужно проверить 1024
маски, а для 30 item-ов — уже больше миллиарда.

В `quality_check` фиксируются:

- доступность проверки (`available`);
- число проверенных масок (`total_masks_checked`);
- число допустимых конфигураций (`feasible_count`);
- совпал ли `GA best` с точным оптимумом (`ga_best_matches_exact_best`);
- ранг найденного ГА решения в точном переборе (`ga_best_exact_rank`);
- пересечение `GA top-N` с точным `exact top-N` (`top_overlap_count`, `top_overlap_ratio`);
- предупреждения, если лучший результат или top-N отличаются от эталона.

Для дипломной демонстрации это удобно объяснять как «ответ на последней
странице учебника»: полный перебор не заменяет ГА, но на маленьком примере
показывает, что эвристический алгоритм ищет корректно.

## Роль GA после завершения roadmap

После завершения roadmap генетический алгоритм занимает место генератора и поисковика допустимых кандидатных конфигураций. Он не заменяет AHP, NPV или DecisionReport.

Общая логика теперь такая:

```text
AnalysisScopeProfile + runtime components → GA search → CandidateConfiguration → AHP / Pareto / Hybrid / TCO / DecisionReport
```

Это уточняет границы ответственности:

| Элемент | Роль |
|---|---|
| `AnalysisScopeProfile` | Определяет, какие критерии и ограничения релевантны для ПО или ТО. |
| GA | Ищет допустимые сочетания компонентов и возвращает несколько сильных вариантов. |
| `CandidateConfiguration` | Делает результат GA совместимым с AHP, анализом важности, TCO и отчётом. |
| AHP | Проверяет, как поисковые кандидаты ранжируются экспертным методом. |
| Pareto | Показывает компромиссы и недоминируемые варианты. |
| Hybrid | Сводит GA/AHP и Pareto-статус в итоговую витрину без повторного запуска методов. |
| `DecisionReport` | Показывает не только победителя, но и параметры запуска, предупреждения и интерпретацию результата. |

Поэтому GA в проекте лучше описывать как механизм поиска альтернатив, а не как окончательный единоличный выбор. Итоговый выбор появляется только после сопоставления с критериями, стоимостью, финансовой интерпретацией и ограничениями.
