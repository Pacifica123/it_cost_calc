# Runtime-хранение `SolutionComponent`

## Назначение

Этот документ фиксирует результат этапа C3 roadmap универсального редактора компонентов. После C2 в проекте уже есть доменная модель `SolutionComponent` и сервис нормализации. Этап C3 добавляет безопасное runtime-хранение таких компонентов, демонстрационные fixture-данные и совместимость экспорта без перехода к UI-редактору.

Главная идея: компоненты редактора хранятся рядом с существующими runtime-данными, но не смешиваются с обычными строками CAPEX/OPEX. Это позволяет открыть старый проект без компонентов редактора и не выполнять принудительную миграцию при первом запуске.

## Формат хранения

Компоненты редактора находятся в отдельной секции runtime-entities:

```text
entities.solution_components[]
```

Для секции используется собственная версия схемы:

```text
SOLUTION_COMPONENT_ENTITY = "solution_components"
SOLUTION_COMPONENT_SCHEMA_VERSION = 1
```

Минимальная строка хранения содержит пользовательские поля компонента и служебные признаки:

- `schema_version` — версия схемы строки `SolutionComponent`;
- `storage_entity` — имя runtime-секции;
- `editor_status` — `strict` или `draft`;
- `normalization_state` — состояние нормализации;
- `candidate_eligible`, `tco_eligible`, `energy_eligible`, `npv_eligible`, `analysis_ready` — признаки допуска;
- `validation_warnings` и `blocking_errors` — объяснение, почему компонент можно или нельзя включать в строгую аналитику;
- `excluded_from_analysis_reason` — короткая пользовательская причина исключения.

## Черновики и строгие компоненты

Черновики не удаляются и не превращаются в нули. Если у компонента нет `scope`, `component_type`, стоимости или профильных метрик, он всё равно может храниться, но получает предупреждения и не попадает в строгий кандидатный набор.

Правило этапа C3:

```text
draft / blocked компонент можно хранить и экспортировать,
но нельзя молча включать в CandidateConfiguration, TCO, NPV или строгий вывод.
```

За это отвечает `SolutionComponentRuntimeService`:

- `list_components()` возвращает все компоненты после актуальной нормализации;
- `list_strict_components()` возвращает только компоненты, готовые к строгой аналитике;
- `list_draft_components()` возвращает черновики, заблокированные и исключённые записи;
- `build_candidate_configuration()` формирует альтернативу только из строгих компонентов;
- `export_snapshot()` отдаёт пользовательский снимок без debug-полей.

## Совместимость со старыми данными

Старые runtime-файлы могут не иметь секции `solution_components`. Это нормальное состояние. В таком случае сервис возвращает пустой список компонентов и не требует миграции.

Legacy-песочница остаётся отдельной областью. Записи с префиксом `legacy_infrastructure:` не становятся `SolutionComponent` автоматически. Их будущая конвертация должна быть ручной и явной.

## Совместимость экспорта

`DecisionReport` теперь умеет отличать компоненты редактора от обычных runtime-строк. В JSON/Markdown-экспорте они получают:

- `source_format = solution_component`;
- `role = solution_component_editor`;
- статус нормализации;
- признаки допуска;
- предупреждения и blocking errors.

Markdown-экспорт получил отдельный раздел `Компоненты редактора`. CSV-экспорт кандидатных конфигураций остаётся совместимым: он продолжает показывать пул альтернатив и не превращается в полный выгрузчик всех runtime-строк.

## Demo-fixture

`data/fixtures/demo_dataset.json` обновлён до `schema_version = 3` и содержит секцию `entities.solution_components`:

- технический компонент;
- программный компонент;
- смешанный компонент-комплект;
- неполный черновик для проверки warnings.

Регрессионные инварианты без `pytest` расширены проверками количества компонентов, строгих компонентов, черновиков и компонентов редактора в `DecisionReport`.

## Граница этапа

Этап C3 не добавляет пользовательскую форму и не переименовывает вкладку `ИТ-песочница`. Это граница этапа C4. После C3 проект умеет хранить и экспортировать компоненты редактора программно, но UI-сценарий создания ещё не реализован.
