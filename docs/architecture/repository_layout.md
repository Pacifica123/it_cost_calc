# Структура репозитория

## Принцип верхнего уровня

В корне репозитория лежат только сущности верхнего уровня:
- `README.md`;
- `pyproject.toml`;
- `requirements/`;
- `docs/`;
- `data/`;
- `scripts/`;
- `src/`;
- `tests/`;
- `tools/`.

Это позволяет сразу понять, где находится:
- код продукта;
- документация;
- данные;
- утилиты;
- тесты.

## Назначение основных каталогов

### `src/`
Основной код приложения.

### `docs/`
Инженерная, продуктовая, исследовательская и дипломная документация.

### `data/`
Фикстуры, примеры и генерируемые артефакты.

### `scripts/`
Точки входа для запуска приложения и отдельных сценариев.

### `tests/`
Автотесты и связанные с ними материалы.

### `tools/`
Вспомогательные технические и исследовательские инструменты, не являющиеся ядром продукта.

## Что не должно лежать в корне

В корень репозитория не должны попадать:
- `__pycache__`;
- временные логи;
- случайные HTML/JSON/CSV-дампы;
- профили браузера;
- промежуточные debug-артефакты;
- одноразовые исследовательские файлы без понятного места в структуре.

## Почему это важно

Чистый корень репозитория создаёт два эффекта:
1. проект легче читать и объяснять;
2. репозиторий визуально выглядит зрелым и аккуратным.

Такая структура помогает привести проект к более профессиональному виду без слома рабочей логики.

## Каталог оборудования как отдельная граница

- `tools/catalog_parser/` — вспомогательный инструмент построения каталога;
- `data/catalog/` — описание стабильной границы данных;
- `data/generated/catalog/` — рабочий выход инструмента;
- `data/examples/catalog/` — пример нормализованного каталога.
