# Патч 8: конкретные метрики ТО вместо абстрактной производительности

## Назначение

Патч переводит технический профиль с условной `performance`-логики на явные измеримые характеристики оборудования. Цель остаётся MVP: не строить сложную экспертную модель ПК/серверов/сетей, а сделать так, чтобы GA, AHP и Pareto работали с понятными полями и показывали предупреждения при неполных данных.

## Что изменено

### Runtime-нормализация

`runtime_entity_normalization_service.py` теперь нормализует технические характеристики из ручных, demo и будущих parser-полей:

| Каноническое поле | Fallback-алиасы |
|---|---|
| `ram_gb` | `ram_score`, `memory_gb` |
| `cpu_cores` | `cpu_score`, `cores` |
| `storage_gb` | `disk_gb`, `ssd_gb`, `hdd_gb` |
| `max_power_watts` | `max_power`, `power_watts`, `energy` |
| `lan_ports` | `lan_port_count`, `ethernet_ports` |
| `lan_speed_mbps` | `ethernet_speed_mbps`, `port_speed_mbps` |
| `wifi_total_mbps` | `wifi_speed_mbps`, `wireless_speed_mbps` |
| `ipv6_support` | `ipv6` |

Старое поле `max_power` сохранено как совместимый алиас, потому что оно уже используется вкладкой электроэнергии и существующими демо-данными.

### Профиль ТО

`AnalysisScopeProfileService` теперь включает в технический профиль не только вычислительные метрики, но и сетевые:

- `total_ram_gb`;
- `total_cpu_cores`;
- `total_storage_gb`;
- `total_power_watts`;
- `lan_ports`;
- `lan_speed_mbps`;
- `wifi_total_mbps`;
- `ipv6_support_count`.

Стоимость по-прежнему не является soft-критерием: она остаётся бюджетным фильтром и tie-breaker из Патча 7.

### GA/AHP/Pareto

GA получает новые поля через нормализованные `RuntimeOptimizationItem`. В итоговых `totals` теперь сохраняются вычислительные и сетевые агрегаты, чтобы общий пул альтернатив мог передать их в AHP и Pareto.

AHP-агрегация поддерживает сетевые поля и сохраняет `metric_warnings`, поэтому неполные характеристики становятся видимыми в pipeline/экспорте, а не превращаются в незаметный ноль.

`ScopedCandidatePoolService` добавляет fallback-алиасы для сетевых критериев при сборке Pareto-case из общего пула.

## Предупреждения о неполных данных

Пустые поля не блокируют расчёт. Вместо этого нормализация добавляет `metric_warnings`:

- для серверов/рабочих станций проверяются `ram_gb`, `cpu_cores`, `storage_gb`, `max_power_watts`;
- для сетевых устройств проверяются `lan_ports`, `lan_speed_mbps`, `wifi_total_mbps`, `ipv6_support`, `max_power_watts`.

Это не идеальная модель качества данных, но она выполняет важное методическое правило: система не притворяется, что отсутствие характеристики — это корректный ноль без пояснений.

## Демо-данные

`data/fixtures/demo_dataset.json` дополнен базовыми характеристиками:

- серверы: RAM, CPU, storage, power;
- рабочие станции: RAM, CPU, storage, power;
- сеть: LAN-порты, LAN-скорость, Wi-Fi скорость, IPv6, power.

Этого достаточно, чтобы на защите показать, что ТО сравнивается не по абстрактному баллу, а по измеримым характеристикам.

## Что не входит

- нет полноценного парсера DNS-Shop;
- нет сложной модели типов накопителей/RAID/SLA;
- нет автоматического разделения роутеров и коммутаторов по разным формулам;
- нет требования заполнять все характеристики перед запуском.

Эти темы остаются для Патча 9 и дальнейших улучшений.

## Риски

| Риск | Снижение |
|---|---|
| Неполные данные могут искажать ранжирование | warnings явно попадают в payload и totals |
| Старые JSON используют `max_power`, а не `max_power_watts` | оба поля поддерживаются как алиасы |
| Сетевые характеристики неоднородны | пока используются простые агрегаты, а парсер будет best-effort |
| UI может выглядеть насыщеннее из-за новых критериев | веса оставлены умеренными, без сложного rule engine |
