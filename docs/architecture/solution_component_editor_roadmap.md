# Roadmap реализации универсального редактора `SolutionComponent`

## Назначение roadmap

Этот документ развивает идею из `future_solution_component_editor_variant_c.md` и переводит её в пошаговый план реализации. Идея варианта C состоит в том, чтобы развить старую `ИТ-песочницу` не в ещё одну свободную таблицу, а в универсальный редактор нормализованных компонентов ИТ-решения.

Roadmap учитывает итоговую сводку `conceptual_cohesion_completion_summary.md`: базовая дорожная карта концептуальной связанности уже завершена, а основная цепочка проекта считается стабилизированной:

```text
runtime-записи
  → scope / component_type
  → AnalysisScopeProfile
  → CandidateConfiguration
  → CostModel / TCO
  → NPV
  → AHP / GA / GA + AHP
  → DecisionReport
```

Поэтому новая инициатива не должна заново изобретать модель данных. Она должна аккуратно подключить новый пользовательский сценарий к уже существующей цепочке.

Документ намеренно оформлен как иерархический чеклист. Корневые блоки можно использовать как будущие devctl-патчи, а вложенные пункты — как критерии готовности и основу для проверок.

---

## Исходная позиция

В текущем состоянии проекта старая вкладка `ИТ-инфраструктура` стабилизирована как `ИТ-песочница`. Это безопасное решение: свободные записи не попадают в строгую аналитику без нормализации и не создают вторую скрытую модель рядом с вкладками `ПО`, `ТО`, `OPEX`, `Электроэнергия`, `NPV`, `AHP`, `GA`, `GA + AHP` и `Экспорт`.

Вариант C предлагает следующий шаг: превратить эту песочницу в advanced-редактор компонентов решения. Такой редактор нужен не вместо обычных вкладок, а рядом с ними:

- `ПО` остаётся быстрым вводом типовых лицензий, подписок и программных сервисов;
- `ТО` остаётся быстрым вводом оборудования и технического контура;
- `OPEX` остаётся вводом регулярных расходов;
- `Электроэнергия` остаётся специализированным расчётом технической нагрузки;
- редактор `SolutionComponent` закрывает нестандартные, смешанные и составные случаи.

Ключевая граница: редактор должен создавать не произвольную строку, а компонент, который понятен проектной модели.

```text
SolutionComponent = нормализованный компонент ИТ-решения,
который имеет область анализа, тип, стоимость, метрики,
ограничения, происхождение и правила участия в аналитике.
```

---

## Целевое место `SolutionComponent` в общей модели

`SolutionComponent` должен стать входным объектом между пользовательским вводом и уже существующими аналитическими сущностями.

```text
Редактор компонентов
  → SolutionComponent
  → нормализация и предупреждения
  → CandidateConfiguration / CostModel
  → TCO / NPV / AHP / GA / GA + AHP
  → DecisionReport
```

Это означает:

- компонент не хранит стоимость как одну неясную сумму, а раскладывает её на CAPEX, внедрение, миграцию, сопровождение, подписки, электроэнергию и прочие применимые части;
- компонент не сам решает, как участвовать в ПО/ТО-анализе, а проходит через `AnalysisScopeProfile`;
- компонент не порождает отдельный формат альтернатив, а включается в `CandidateConfiguration`;
- компонент не уходит в отчёт как непроверенная заметка, а передаёт происхождение, допущения и warnings в `DecisionReport`.

## Общие правила реализации

- [ ] **Не ломать завершённую conceptual cohesion roadmap.**
  - [ ] Не менять смысл уже введённых `scope`, `component_type`, `AnalysisScopeProfile`, `CandidateConfiguration`, `CostModel` и `DecisionReport` без отдельного обоснования.
  - [ ] Не возвращать проект к состоянию набора независимых вкладок.
  - [ ] Не создавать второй формат альтернатив рядом с `CandidateConfiguration`.

- [ ] **Сохранять вкладки `ПО` и `ТО` как быстрые сценарии.**
  - [ ] Не заменять типовые формы одной большой универсальной формой.
  - [ ] Не заставлять обычного пользователя заполнять расширенный редактор для простых лицензий, серверов, рабочих мест и сетевого оборудования.
  - [ ] Использовать редактор только для смешанных, составных и нестандартных компонентов.

- [ ] **Держать предметную логику вне UI.**
  - [ ] UI выбирает тип компонента, показывает поля и предупреждения.
  - [ ] Проверки обязательных полей выполняются в application/domain-слое.
  - [ ] Решение о допуске компонента в аналитику принимает сервис нормализации, а не экран формы.

- [ ] **Сохранять обратную совместимость.**
  - [ ] Старые sandbox-записи остаются читаемыми.
  - [ ] Конвертация старых записей выполняется вручную и явно.
  - [ ] Неполные компоненты могут храниться как черновики, но не участвуют в строгой аналитике.

- [ ] **Каждый будущий патч должен быть воспроизводимым.**
  - [ ] Кодовые патчи должны иметь проверки без генерации служебных кэш-каталогов.
  - [ ] Для регресса предпочтительны `python -c`, `ast.parse`, проектные demo/control-сценарии или отдельные скрипты без кэширующего тестового раннера.
  - [ ] Документационные патчи должны проверять наличие файлов, читаемость Markdown и баланс code fences.

## Термины готовности компонента

Чтобы редактор не подмешивал неполные данные в расчёты, компоненту нужны понятные состояния.

| Состояние | Смысл | Можно хранить | Можно включать в аналитику | Можно показывать в отчёте |
|---|---|---:|---:|---:|
| `draft` | Черновик, не хватает обязательных полей | да | нет | да, как черновик/исключение |
| `normalized` | Есть `scope`, `component_type`, базовая стоимость и метрики | да | частично | да |
| `candidate_eligible` | Компонент может входить в `CandidateConfiguration` | да | да | да |
| `tco_eligible` | Финансовые поля достаточно структурированы для TCO | да | да, по стоимости | да |
| `analysis_ready` | Компонент прошёл профильные проверки ПО/ТО/сводного сценария | да | да | да |
| `excluded` | Компонент сохранён, но сознательно исключён из строгой аналитики | да | нет | да, с причиной |

Эти состояния не обязаны быть отдельным enum в первой реализации, но их нужно поддержать хотя бы через поля допуска и предупреждения.

## Этап C5. Интеграция стоимости, TCO, энергии и NPV

> Цель: убедиться, что редактор не создаёт самостоятельный калькулятор стоимости, а отдаёт данные в уже существующую финансовую цепочку.

- [ ] **C5.1. Подключить финансовые поля к `CostModel`.**
  - [ ] `purchase_cost` передавать в капитальную часть.
  - [ ] `implementation_cost`, `migration_cost`, `testing_cost` передавать в разовые расходы.
  - [ ] `monthly_cost` передавать в регулярные ежемесячные расходы.
  - [ ] `annual_cost` приводить к годовому или месячному горизонту по правилам TCO.
  - [ ] Не смешивать пользовательскую оценку эффекта с расходами.

- [ ] **C5.2. Подключить энергию только для применимых компонентов.**
  - [ ] Для `server`, `workstation`, `network_device`, `peripheral` использовать мощность и профиль работы.
  - [ ] Для `software_license` и `software_subscription` не требовать мощность.
  - [ ] Для `mixed` требовать явного указания технической части или предупреждать о невозможности расчёта энергии.
  - [ ] Показывать, является ли energy cost введённой суммой или производным расчётом.

- [ ] **C5.3. Передать финансовую базу в NPV.**
  - [ ] Стартовые расходы компонента попадут в initial investment.
  - [ ] Регулярные расходы попадут в annual outflows.
  - [ ] Ожидаемый эффект/экономия останется отдельным вводом или отдельным блоком допущений.
  - [ ] NPV не должен становиться положительным без объяснения источника эффекта.

- [ ] **C5.4. Обновить предупреждения финансовой модели.**
  - [ ] Предупреждать, если все расходы введены одной суммой.
  - [ ] Предупреждать, если компонент участвует в TCO, но не имеет горизонта или регулярности.
  - [ ] Предупреждать, если энергетическая часть указана вручную и не связана с мощностью.
  - [ ] Предупреждать, если NPV получает расходы, но не получает эффекта.

- [ ] **Критерий завершения C5.**
  - [ ] Компонент редактора проходит финансовую цепочку `стоимость → TCO → NPV` без ручного повторного ввода уже известных расходов.

## Этап C6. Интеграция с `CandidateConfiguration`, AHP, GA и GA + AHP

> Цель: подключить валидные компоненты редактора к аналитике через общий пул альтернатив, не создавая отдельный сценарий сравнения.

- [ ] **C6.1. Определить правила включения в альтернативы.**
  - [ ] Одиночный компонент может стать частью ручной альтернативы.
  - [ ] Смешанный комплект может стать компонентом альтернативы только после нормализации состава или явных допущений.
  - [ ] Неполный `draft` не включается в альтернативы.
  - [ ] Исключённый компонент показывается как источник данных, но не влияет на рейтинг.

- [ ] **C6.2. Подключить профильные метрики.**
  - [ ] Технические компоненты используют технический `AnalysisScopeProfile`.
  - [ ] Программные компоненты используют программный `AnalysisScopeProfile`.
  - [ ] Смешанные компоненты требуют сводного профиля или разбиения.
  - [ ] Отсутствующие метрики дают warnings, а не неявные нули.

- [ ] **C6.3. Проверить AHP.**
  - [ ] Компоненты редактора могут попасть в альтернативы AHP через общий формат.
  - [ ] Контрольный пример AHP не меняет результат.
  - [ ] Пояснения критериев берутся из профиля, а не из формы редактора.

- [ ] **C6.4. Проверить GA.**
  - [ ] Компоненты редактора могут быть кандидатными items при генерации конфигурации.
  - [ ] Ограничения бюджета, клиентских мест, лицензий и мощности сохраняют смысл.
  - [ ] Периферия не превращается в рабочее место только из-за принадлежности к клиентскому контуру.
  - [ ] Seed и параметры запуска сохраняются для воспроизводимости.

- [ ] **C6.5. Проверить GA + AHP.**
  - [ ] Гибридная оценка получает те же `CandidateConfiguration`, что и остальные методы.
  - [ ] Расхождение между финансово дешёвой и качественно сильной альтернативой объясняется в отчёте.
  - [ ] Компоненты редактора не получают скрытого преимущества из-за неполных метрик.

- [ ] **Критерий завершения C6.**
  - [ ] AHP, GA и GA + AHP работают с компонентами редактора только через `CandidateConfiguration`, без нового ad-hoc-формата альтернатив.

## Итоговая формулировка

Редактор `SolutionComponent` имеет смысл реализовывать только как продолжение уже связанной архитектуры. Его задача — не заменить существующие вкладки, а дать controlled advanced-режим для нестандартных компонентов, которые после нормализации проходят тот же путь, что и остальные данные проекта:

```text
компонент → профиль → альтернатива → стоимость → аналитика → отчёт
```

Если каждый будущий патч будет сохранять эту цепочку, вариант C усилит проект: он позволит описывать более реалистичные ИТ-решения, не возвращая систему к разрозненным таблицам и не разрушая уже выстроенную концептуальную связанность.
