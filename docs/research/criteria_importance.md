# Исследование: анализ важности критериев

## Назначение модуля

Этот модуль нужен для задачи **обоснования выбора ИТ-решения**.  
Он полезен в тех ситуациях, когда требуется не просто выбрать лучший вариант, а показать, как на выбор повлияли отношения важности между критериями.

## Основная идея

Пользователь или эксперт задаёт:

- набор критериев;
- набор альтернатив;
- отношения важности между критериями;
- оценки альтернатив по критериям.

После этого система:
- строит модель относительной важности;
- определяет кратности критериев;
- сравнивает альтернативы;
- выделяет недоминируемые варианты;
- делает итоговый выбор и формирует объяснение результата.

## Где расположен код

- `src/domain/decision/criteria_importance/analysis.py`
- `src/domain/decision/criteria_importance/pipeline.py`
- `src/ui/tabs/criteria_importance/`

## Что содержит результат

На выходе формируется структура с:
- `ranking`
- `weights`
- `winner_id`
- `winner_name`
- `explanation`
- вспомогательной `metadata`

То есть модуль даёт не только числовой ответ, но и текстовое объяснение, почему был выбран именно этот вариант.

## Почему это важно для диплома

В дипломной постановке этот модуль важен как мост между:
- сухим расчётом показателей;
- и содержательным объяснением, почему решение считается предпочтительным.

Это особенно важно для кейсов, где требуется не просто “минимальная цена”, а формализованное обоснование выбора ИТ-решения по нескольким критериям.

## Связь с академическими материалами

Тематика этого модуля связана с дипломной постановкой выбора готовых ИТ-решений и объяснением, почему одно решение предпочтительнее другого по нескольким критериям.

## Связь с общим пулом альтернатив после roadmap

После завершения roadmap анализ важности критериев связан с тем же пулом `CandidateConfiguration`, что и AHP/GA-сценарии. Это снижает риск, что разные вкладки сравнивают разные сущности под одинаковым названием «альтернатива».

Текущая роль модуля:

```text
candidate_configurations → отношения важности критериев → недоминируемые варианты → объяснение → DecisionReport
```

Модуль остаётся самостоятельным по методике: он строит собственную модель отношений критериев и может выделять недоминируемые варианты. Но его вход и выход теперь должны быть связаны с общей предметной цепочкой, чтобы итоговое объяснение не противоречило AHP, GA, TCO и NPV.

Для защиты это можно формулировать так: анализ важности критериев показывает, какие факторы сильнее всего влияют на выбор и почему один вариант выглядит предпочтительным не только по цене.
