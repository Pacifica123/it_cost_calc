# Роль старой вкладки ИТ-инфраструктуры

## Назначение

Старая вкладка `ИТ-инфраструктура` сохранена как вспомогательная песочница для пользовательских статей. Она нужна для черновых заметок и свободных записей, которые ещё не готовы стать частью строгой модели CAPEX/OPEX/энергии/аналитики.

Выбранный вариант стабилизации из roadmap — **вариант A: оставить как песочницу пользовательских статей**. Полный перевод вкладки в универсальный редактор `SolutionComponent` отложен, потому что основной сценарий уже покрыт вкладками `ТО`, `ПО`, `Операционные затраты`, `Электроэнергия`, `NPV`, `GA/AHP` и `Экспорт`.

## Почему вкладка не становится второй моделью данных

До стабилизации вкладка позволяла создавать произвольные статьи с произвольными названиями. Такие статьи могли выглядеть как полноценные категории затрат, но не имели предметных признаков `scope`, `component_type`, стоимости владения, энергетических метрик и связи с профилями анализа. Из-за этого появлялся риск второй скрытой модели данных рядом с нормализованными вкладками ПО/ТО.

После стабилизации новые статьи вкладки сохраняются в runtime-хранилище с префиксом:

```text
legacy_infrastructure:
```

Также каждая запись получает служебные поля:

| Поле | Назначение |
|---|---|
| `_legacy_sandbox` | явно помечает строку как вспомогательную sandbox-запись |
| `_legacy_article_title` | хранит человекочитаемое название статьи |
| `_legacy_expense_type` | фиксирует тип свободной статьи: периодическая, разовая или без стоимости |
| `_legacy_note` | объясняет, что запись не участвует в строгой аналитике без отдельной нормализации |

## Участие в аналитике

Sandbox-записи не должны автоматически попадать в строгий пул `CandidateConfiguration`. Прикладной слой пропускает группы с префиксом `legacy_infrastructure:` при построении runtime-кандидата. Это защищает GA/AHP, TCO, NPV и `DecisionReport` от случайного смешивания свободных заметок с нормализованными компонентами.

При этом `DecisionReport` может показать такие строки как исходные компоненты с признаком:

```text
strict_analysis_participation = false
```

Такой подход сохраняет прозрачность: пользователь видит, что вспомогательные записи существовали в runtime-данных, но не получает ложного ощущения, что они уже прошли предметную нормализацию.

## Что использовать вместо старой вкладки

| Сценарий | Основная вкладка |
|---|---|
| Серверы, рабочие места, сеть | `ТО` |
| Лицензии и программные позиции | `ПО` |
| Подписки, миграция, тестирование, администрирование | `Операционные затраты` |
| Мощность и стоимость электроэнергии | `Электроэнергия` |
| Сравнение альтернатив | `AHP`, `Генетический подбор`, `ГА + AHP` |
| Финансовая интерпретация | `NPV-анализ` |
| Итоговое объяснение выбора | `Экспорт` / `DecisionReport` |

## Будущее развитие

Если вкладку решат развивать дальше, безопасный следующий шаг — не расширять свободные статьи, а превратить её в универсальный редактор `SolutionComponent`. Тогда форма должна явно показывать `scope`, `component_type`, стоимость, метрики, участие в CAPEX/OPEX/энергии и правила нормализации. До такого решения вкладка остаётся именно песочницей, а не методическим центром проекта.

Отдельная будущая идея варианта C зафиксирована в документе `future_solution_component_editor_variant_c.md`. Там вариант C рассматривается уже после завершения всей roadmap, когда `scope`, `component_type`, профили анализа, `CandidateConfiguration`, TCO, NPV и `DecisionReport` стали единой рабочей цепочкой.
