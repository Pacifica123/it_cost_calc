# Синхронизация AHP-критериев с профилями ПО/ТО

## Что изменено

AHP-панель больше не обязана жить на старом универсальном наборе критериев `надёжность / производительность / стоимость / энергия / срок службы` внутри рабочих областей `ПО` и `ТО`.

Теперь при работе внутри scope AHP берёт список критериев из того же `AnalysisScopeProfile`, который используется GA и Pareto:

- для ТО — явные технические метрики: ОЗУ, CPU, накопитель, энергопотребление;
- для ПО — функциональная пригодность и сопровождение;
- направления критериев (`max`/`min`) передаются в AHP-пайплайн явно.

## Зачем это нужно

До этого после удаления `category_coverage` всё ещё оставался методический разрыв: GA работал по профильным критериям ПО/ТО, а AHP визуально и расчётно продолжал использовать старую модель. Это мешало объяснить формулу `(GA + AHP + Pareto + Hybrid) × ПО/ТО`, потому что методы сравнивали альтернативы не по одному и тому же смысловому набору показателей.

Патч закрывает именно этот разрыв: AHP остаётся самостоятельным методом ранжирования, но его критерии теперь берутся из той же предметной модели области анализа.

## Как это работает

1. AHP-панель определяет текущий scope: `technical` или `software`.
2. Из `AnalysisScopeProfileService` берётся список критериев текущего профиля.
3. UI строит чекбоксы критериев и матрицу попарных сравнений по этому списку.
4. При запуске AHP в пайплайн передаются не только имена критериев, но и направления `max`/`min`.
5. Для `min`-критериев AHP использует положительную utility-шкалу, где меньшее значение становится лучшим, а не худшим.

## Поведение машинного режима

Для старых учебных AHP-критериев сохранены прежние экспертные матрицы.

Для новых профильных критериев ПО/ТО исторической экспертной матрицы ещё нет. Поэтому машинный режим использует нейтральную матрицу единичных предпочтений: все выбранные профильные критерии считаются равноважными, пока пользователь явно не переключится в экспертный режим и не задаст свои попарные сравнения.

Такой подход не переусложняет MVP и не создаёт ложного ощущения, что экспертные веса уже теоретически обоснованы.

## Основные файлы

| Файл | Роль |
|---|---|
| `src/ui/tabs/ahp_analysis/tab.py` | AHP UI строит список критериев из текущего scope-профиля |
| `src/ui/tabs/ahp_analysis/matrix_utils.py` | нейтральные матрицы для новых профильных критериев |
| `src/domain/decision/ahp/pipeline.py` | поддержка направлений `max`/`min` при построении матриц альтернатив |
| `src/domain/decision/ahp/aggregation.py` | агрегирование профильных метрик ТО/ПО для AHP |
| `src/application/services/candidate_configuration_service.py` | сохранение `totals` и `metrics` при передаче общего пула в AHP |

## Ограничения шага

- Экспертные матрицы для новых критериев пока нейтральные, а не методически выведенные.
- Стоимость после следующего шага больше не является базовым AHP-критерием профиля; она остаётся бюджетным фильтром, отчётной величиной и GA tie-breaker между близкими вариантами.
- Для сетевого оборудования пока нет отдельного набора AHP-критериев по LAN/Wi-Fi/IPv6; это логично делать вместе с развитием парсинга характеристик.

## Критерий готовности

Критерии AHP, GA и Pareto теперь приходят из одного scope-профиля. Пользователь видит, что AHP ранжирует тот же пул альтернатив и по тем же предметным критериям области, а стоимость не дублируется как второй главный критерий поверх бюджета.
