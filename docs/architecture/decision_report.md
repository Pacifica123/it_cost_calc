# DecisionReport: единый итоговый отчёт выбора

## Назначение

`DecisionReport` завершает цепочку концептуальной связанности проекта: отдельные расчёты CAPEX, OPEX, электроэнергии, AHP, GA, GA + AHP и NPV собираются в один воспроизводимый артефакт выбора ИТ-решения.

До этого этапа приложение уже умело:
- нормализовать сущности через `scope` и `component_type`;
- применять профили анализа ПО/ТО;
- приводить альтернативы к `CandidateConfiguration`;
- рассчитывать TCO и передавать финансовую базу в NPV.

Этап 6 добавляет слой, который не запускает методы заново, а фиксирует состояние данных на момент экспорта и объясняет итоговый выбор.

## Структура отчёта

| Блок | Содержание | Назначение |
|---|---|---|
| `project` | исходный проект выбора, цель, дата формирования | показывает, для какого решения сформирован отчёт |
| `components` | текущие runtime-компоненты из CAPEX/OPEX/ПО/ТО | связывает вывод с исходными пользовательскими данными |
| `cost_model` | итоги CAPEX/OPEX/электроэнергии и блок `tco` | показывает стоимость владения и финансовую базу |
| `candidate_configurations` | общий пул альтернатив | даёт единый объект сравнения для AHP, GA и отчёта |
| `solution_component_report` | отдельное объяснение компонентов редактора | показывает валидные компоненты, черновики, warnings и связь с альтернативами |
| `analysis_results` | результаты AHP, GA, GA + AHP, критериального анализа | хранит машинно-воспроизводимые результаты методов |
| `npv_interpretation` | NPV, денежные потоки, предупреждения | объясняет финансовую часть выбора |
| `winner_explanation` | лидеры методов, согласованность, рекомендуемый вариант | превращает набор результатов в итоговое объяснение |
| `risks` и `warnings` | ограничения, неполные данные, методические предупреждения | не скрывает слабые места вывода |
| `metadata` | версия схемы, тип отчёта, источник экспорта | поддерживает машинный экспорт и будущую совместимость |

## Машинный и пользовательский экспорт

Этап разделяет два уровня представления.

### JSON

JSON считается полным машинным артефактом. Он сохраняет:
- исходные компоненты;
- полный пул `CandidateConfiguration`;
- отдельный блок `solution_components` для строк редактора;
- признаки допуска, warnings, origin и связь компонентов с альтернативами;
- результаты методов;
- TCO и NPV;
- предупреждения;
- metadata схемы.

Файл предназначен для повторного анализа, диагностики и будущей автоматической генерации дипломных материалов.

### Markdown

Markdown является пользовательским и защитным представлением. Он не пытается показать все debug-поля, а отвечает на вопросы:
- какое решение рассматривается;
- какой вариант рекомендован;
- почему методы пришли к такому выводу;
- какая стоимость владения получилась;
- какие `SolutionComponent` участвовали в выборе;
- какие черновики или исключённые компоненты не влияли на выбор и почему;
- что показывает NPV;
- какие ограничения и риски остаются.

### CSV

CSV кандидатов остаётся табличным срезом пула альтернатив. Он удобен для просмотра в таблицах и не заменяет JSON-отчёт. Для C7 добавлен отдельный CSV компонентов редактора, чтобы не смешивать финансовые поля и профильные метрики с таблицей альтернатив.


## Компоненты редактора SolutionComponent

После C7 `DecisionReport` отдельно объясняет строки редактора компонентов. Валидные компоненты показываются с `scope`, `component_type`, происхождением, статусом допуска, финансовым вкладом, использованными метриками и связью с `CandidateConfiguration`. Черновики и исключённые компоненты выводятся отдельной таблицей с причинами, поэтому неполные записи не выглядят как скрытые расчётные данные.

Это особенно важно для защиты: можно показать, что advanced-редактор не создаёт параллельный формат альтернатив, а отдаёт данные в ту же цепочку `component → CandidateConfiguration → TCO/AHP/GA → DecisionReport`.

## Объяснение победителя

`DecisionReportService` собирает лидеров из доступных методов:
- AHP;
- GA;
- GA + AHP / гибридной оценки;
- анализа важности критериев.

Если лидеры совпали, отчёт помечает вывод как согласованный. Если лидеры различаются, отчёт не скрывает конфликт: рекомендуемый вариант выбирается по приоритету гибридной/AHP-оценки, а причины расхождения остаются в отчёте.

## Защитный сценарий

Для дипломной защиты поток можно объяснить так:

1. Пользователь вводит компоненты ИТ-решения.
2. Прикладной слой нормализует их по области анализа и типу компонента.
3. Из компонентов формируется общий пул альтернатив `CandidateConfiguration`.
4. TCO связывает CAPEX, OPEX и электроэнергию.
5. AHP, GA и GA + AHP анализируют один пул альтернатив.
6. NPV использует рассчитанную финансовую базу.
7. `DecisionReport` показывает валидные компоненты редактора, исключённые черновики и связь компонентов с альтернативами.
8. `DecisionReport` фиксирует исходные данные, результаты методов, победителя, риски и ограничения.

Главная мысль: проект больше не является набором отдельных калькуляторов, а работает как связанная система поддержки выбора ИТ-решения.


## Роль C9-демонстрации

Финальный C9-сценарий использует отдельную fixture `data/fixtures/solution_component_c9_demo_dataset.json`, чтобы показать именно редактор компонентов. В отчёте должны быть видны:

- mixed-комплект «сервер + гипервизор + базовая настройка» как валидный компонент с TCO-вкладом;
- программная подписка как ПО-компонент без энергетического участия;
- МФУ как техническая периферия с `client_seats = 0`;
- черновик как исключённая запись с причинами.

Это делает `DecisionReport` не только экспортом расчёта, но и защитным материалом: он показывает, какие данные участвовали в выборе, какие были исключены и почему.

## Границы этапа 6

Этап 6 не заменяет старый CSV-экспорт затрат и не требует полного прохождения всех аналитических вкладок перед экспортом. Если часть методов ещё не запускалась, отчёт формируется как частичный и явно добавляет предупреждения.

Такой подход безопасен для текущего GUI: пользователь может получить отчёт уже после ввода затрат, а по мере запуска AHP, GA, GA + AHP и NPV отчёт становится богаче.
