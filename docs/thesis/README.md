# Дипломные материалы

Этот каталог хранит только те академические материалы, которые нужны в рабочем репозитории после инженерной переработки источников.

## Политика хранения

Раньше в репозитории лежал каталог `raw/` с исходными офисными файлами: черновыми отчётами, оглавлением, таблицей по парсингу и другими материалами. Эти файлы были полезны как временный источник контекста, но они заметно раздували архив проекта и не участвовали в запуске приложения, тестах, демонстрационных сценариях или архитектурной сборке.

Чтобы рабочий архив оставался лёгким для передачи в чатах и devctl-пайплайне, исходные `raw`-материалы больше не хранятся внутри проекта. Их содержательная роль перенесена в человекочитаемые документы:

- `docs/thesis/project_academic_overview.docx` — академическое описание проекта;
- `docs/research/` — исследовательские пояснения по AHP, важности критериев, генетическому алгоритму и парсингу;
- `docs/architecture/` — инженерная и архитектурная фиксация решений.

## Что остаётся в `docs/thesis/`

- `README.md` — это описание политики хранения академических материалов;
- `project_academic_overview.docx` — компактный оформленный материал, который можно использовать как основу для текста диплома.

## Важное правило

Исходные дипломные материалы не должны подменять собой инженерную документацию репозитория. Если тяжёлые черновики снова понадобятся, их лучше хранить вне рабочего архива проекта и переносить в репозиторий только переработанные выдержки или компактные итоговые документы.

## Краткая академическая формулировка текущего результата

После завершения roadmap проект можно описывать в дипломной работе не как набор отдельных расчётных вкладок, а как прототип системы поддержки выбора ИТ-решения.

Рабочая формулировка:

> Разработанное приложение позволяет описать состав ИТ-решения, нормализовать компоненты по области анализа и типу, сформировать сопоставимые кандидатные конфигурации, рассчитать стоимость владения, провести многокритериальную и оптимизационную оценку, а затем собрать результаты в единый итоговый отчёт выбора.

В этой формулировке:

- CAPEX, OPEX и электроэнергия образуют финансовую основу TCO;
- GA используется для поиска допустимых конфигураций;
- AHP и анализ важности критериев объясняют многокритериальный выбор;
- NPV оценивает финансовую сторону сценария;
- `DecisionReport` связывает входные данные, методы анализа, ограничения, риски и итоговое объяснение.

## Как объяснять различие ПО и ТО на защите

Различие ПО и ТО теперь задаётся не только названием вкладки. Оно проходит через:

- `scope` — область анализа;
- `component_type` — тип компонента;
- `AnalysisScopeProfile` — набор категорий, критериев, ограничений и подсказок;
- `CandidateConfiguration` — общий формат альтернатив, который позволяет сравнивать результаты методов без дублирования форматов.

Коротко это можно сказать так: технические и программные решения анализируются разными профилями, но сводятся к единому формату альтернатив и итоговому отчёту.
