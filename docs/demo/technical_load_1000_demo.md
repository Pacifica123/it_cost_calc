# Нагрузочный demo-профиль ТО на 1000 устройств

## Назначение

Этот сценарий нужен не для ручной проверки формул, а для проверки скорости и устойчивости программы на большом массиве исходных данных.

Расчётный профиль `calculation_control` отвечает на вопрос: «правильно ли методики считают маленький пример?».  
Нагрузочный профиль `technical_load_1000` отвечает на другой вопрос: «не разваливается ли программа, когда в разделе ТО много строк?».

## Что лежит в профиле

Файл: `data/fixtures/demo_technical_load_1000_dataset.json`.

Состав технического обеспечения:

| Категория | Количество строк | Смысл |
|---|---:|---|
| `server` | 334 | синтетические серверы разных уровней |
| `client` | 333 | синтетические рабочие станции |
| `network` | 333 | синтетические сетевые устройства |
| Итого ТО | 1000 | большой пул для GA |

ПО и OPEX-разделы в этом профиле намеренно пустые. Так нагрузка не размазывается по разным областям, а проверяет именно большой массив технического обеспечения.

## Как показать в GUI

1. Запустить приложение.
2. Нажать `Ещё` / вторичные действия в верхней панели.
3. Выбрать `Демо: нагрузочный ТО`.
4. Открыть раздел `Техническое обеспечение`.
5. Проверить карточку `Позиций`: должно быть `1000`.
6. Перейти на шаг `GA`.
7. Запустить GA с параметрами по умолчанию или с рекомендуемыми параметрами ниже.
8. После GA перейти к AHP и Pareto: они должны работать по Top-N пулу GA-кандидатов.

## Рекомендуемые параметры для headless-проверки

Скрипт:

```bash
python -B scripts/run_technical_load_demo.py --check-only
```

Параметры по умолчанию в скрипте:

| Параметр | Значение | Почему так |
|---|---:|---|
| Бюджет | 2 400 000 | даёт GA возможность собрать непустую конфигурацию |
| Минимум мест | 3 | оставляет быстрый и воспроизводимый smoke-запуск |
| Популяция | 24 | совпадает с обычным GUI-запуском |
| Поколений | 30 | совпадает с обычным GUI-запуском |
| Top-N | 8 | AHP/Pareto ранжируют общий пул из 8 GA-кандидатов |
| Полный перебор | выключен | для 1000 item полный перебор невозможен |

## Что именно проверяется

Скрипт проверяет инварианты, которые не зависят от скорости конкретного компьютера:

- загружено 1000 строк ТО;
- категории распределены как 334/333/333;
- GA увидел 1000 кандидатов;
- полный перебор GA пропущен намеренно;
- GA вернул непустую лучшую конфигурацию;
- в общий пул передано 8 кандидатов;
- AHP успешно ранжировал GA-пул;
- Pareto успешно обработал тот же GA-пул.

В отчёте скрипта также печатаются времена `load_seconds`, `ga_seconds`, `ahp_seconds`, `pareto_seconds`. Это не жёсткие требования, а ориентиры для демонстрации производительности на конкретной машине.

## Ограничение сценария

Этот профиль не доказывает математическую оптимальность результата. Для этого есть маленький расчётный пример `calculation_control`, где ответы можно проверить вручную.

Нагрузочный профиль доказывает другое: текущая архитектура способна загрузить большой fixture, превратить 1000 строк ТО в кандидатов GA, выполнить эвристический поиск и затем передать компактный Top-N пул в AHP/Pareto.
