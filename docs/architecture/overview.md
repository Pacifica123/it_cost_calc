# Обзор архитектуры

## Общая картина

Проект состоит из четырёх основных инженерных слоёв и нескольких вспомогательных зон:
- `ui` — пользовательский интерфейс;
- `application` — сценарии использования и прикладные сервисы;
- `domain` — предметная логика и алгоритмы;
- `infrastructure` — технические адаптеры хранения, экспорта и логирования;
- `tools` — вспомогательные исследовательские утилиты;
- `docs` и `data` — документация и данные.

## Главные пользовательские модули

### CAPEX / OPEX
Модули капитальных и операционных затрат позволяют ввести состав инфраструктуры и затраты по категориям.

Результат:
- структурированные записи по категориям;
- сводные суммы;
- данные для экспорта и последующего анализа.

### Электроэнергия
Модуль электроэнергии использует данные о составе оборудования и параметры режима работы.

Результат:
- расчёт потребления;
- оценка затрат на электроэнергию;
- включение этих затрат в общую сводку.

### NPV
Модуль NPV не зависит от GUI-таблиц напрямую и формирует отдельный отчёт по инвестиционному сценарию.

Результат:
- табличный NPV-отчёт;
- накопленные значения по годам;
- итоговая оценка инвестиционной привлекательности.

### AHP-анализ конфигураций
Этот модуль предназначен для сравнения альтернативных конфигураций оборудования по мягким критериям после предварительной фильтрации по жёстким ограничениям.

Результат:
- агрегирование конфигураций;
- фильтрация недопустимых вариантов;
- расчёт весов критериев;
- ранжирование альтернатив;
- выделение победителя и top-группы.

### Обоснование выбора ИТ-решения
Модуль анализа важности критериев строит модель относительной важности критериев и применяет её к набору альтернатив.

Результат:
- группировка и упорядочивание критериев;
- расширенная модель оценки;
- множество недоминируемых альтернатив;
- итоговый выбор и текстовое объяснение результата.

### Каталог оборудования
Инструмент `tools/catalog_parser` вынесен за пределы основного runtime.
Он собирает или нормализует данные по реально существующему оборудованию и формирует каталог, который затем может использоваться как внешний справочник.

## Слои системы

### `src/ui`
Отвечает за:
- вкладки;
- формы ввода;
- диалоги;
- таблицы;
- отображение результатов.

UI не должен быть местом хранения предметной логики.

### `src/application`
Отвечает за:
- orchestration;
- use case-сценарии;
- координацию между UI, доменом и репозиториями;
- сервисы подготовки итоговых данных.

Именно здесь находятся сценарии вроде:
- `PrepareCostSummaryUseCase`;
- `ExportCostReportUseCase`;
- `LoadDemoDatasetUseCase`;
- `BuildNpvReportUseCase`.

### `src/domain`
Отвечает за:
- предметные модели;
- финансовые формулы;
- AHP-алгоритмы;
- анализ важности критериев;
- оптимизационные процедуры.

Это слой, который должен оставаться максимально независимым от `Tkinter`, JSON-хранилищ и конкретных экранов.

### `src/infrastructure`
Отвечает за:
- JSON storage;
- runtime-репозитории;
- CSV-экспорт;
- логирование.

Это техническая прослойка, а не место для продуктовой логики.

## Поток данных в системе

Упрощённо поток выглядит так:
1. пользователь работает с вкладкой в `ui`;
2. вкладка вызывает прикладной сервис или use case из `application`;
3. сервис берёт или сохраняет данные через `infrastructure`;
4. при необходимости используется предметная логика из `domain`;
5. результат возвращается в `ui` для отображения пользователю.

## Каталог оборудования и граница с парсером

Основное приложение не занимается HTML-разбором и сетевым обходом магазинов напрямую.
Вместо этого вспомогательный инструмент обновляет нормализованный каталог оборудования, а приложение работает уже с готовым справочником.
