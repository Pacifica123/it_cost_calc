# Граница между приложением и каталогом оборудования

## Базовый принцип

В проекте используется следующая модель:
- **основное приложение** отвечает за расчёты, анализ, сравнение конфигураций и формирование отчётов;
- **инструмент обновления каталога** отвечает за получение и нормализацию данных о реально существующем оборудовании;
- **каталог оборудования** выступает промежуточным артефактом между этими двумя зонами.

Иными словами, GUI не должен тащить в себя Playwright, сетевой обход магазинов и HTML-разбор.
Он должен работать уже с подготовленным каталогом.

## Где проходит эта граница в репозитории

### Основное приложение
- `src/it_cost_calc/`

### Вспомогательный инструмент обновления каталога
- `tools/catalog_parser/`
- `scripts/update_equipment_catalog.py`

### Сырые и нормализованные данные
- `data/examples/parser/` — сохранённые примеры сырых снимков;
- `data/examples/catalog/` — пример нормализованного каталога;
- `data/generated/catalog/` — рабочий выход инструмента.

## Практический поток данных

1. запускается `scripts/update_equipment_catalog.py`;
2. инструмент либо нормализует уже имеющиеся example-снимки, либо использует источник, который получает сырые данные;
3. на выходе формируется `equipment_catalog.json`;
4. приложение читает этот каталог как внешний справочник оборудования.

## Почему это лучше, чем встраивать парсер в GUI

Если парсер живёт внутри runtime приложения, то туда попадают:
- браузерные профили;
- сетевые сбои;
- нестабильные селекторы;
- антибот-ограничения;
- длинные операции и зависания UI.

Такая граница позволяет сохранить проект аккуратным:
- приложение остаётся настольным аналитическим инструментом;
- парсер остаётся отдельным техническим инструментом наполнения справочника.
