# Обзор архитектуры

## Общая картина

Проект состоит из четырёх основных инженерных слоёв и нескольких вспомогательных зон:
- `ui` — пользовательский интерфейс;
- `application` — сценарии использования и прикладные сервисы;
- `domain` — предметная логика и алгоритмы;
- `infrastructure` — технические адаптеры хранения, экспорта и логирования;
- `tools` — вспомогательные исследовательские утилиты;
- `docs` и `data` — документация и данные.

## Главные пользовательские модули

### CAPEX / OPEX
Модули капитальных и операционных затрат позволяют ввести состав инфраструктуры и затраты по категориям.

Результат:
- структурированные записи по категориям;
- сводные суммы;
- данные для экспорта и последующего анализа.

### Электроэнергия
Модуль электроэнергии использует данные о составе оборудования и параметры режима работы.

Результат:
- расчёт потребления;
- оценка затрат на электроэнергию;
- включение этих затрат в общую сводку.

### NPV
Модуль NPV не зависит от GUI-таблиц напрямую и формирует отдельный отчёт по инвестиционному сценарию.

Результат:
- табличный NPV-отчёт;
- накопленные значения по годам;
- итоговая оценка инвестиционной привлекательности.

### AHP-анализ конфигураций
Этот модуль предназначен для сравнения альтернативных конфигураций оборудования по мягким критериям после предварительной фильтрации по жёстким ограничениям.

Результат:
- агрегирование конфигураций;
- фильтрация недопустимых вариантов;
- расчёт весов критериев;
- ранжирование альтернатив;
- выделение победителя и top-группы.

### Обоснование выбора ИТ-решения
Модуль анализа важности критериев строит модель относительной важности критериев и применяет её к набору альтернатив.

Результат:
- группировка и упорядочивание критериев;
- расширенная модель оценки;
- множество недоминируемых альтернатив;
- итоговый выбор и текстовое объяснение результата.

### Каталог оборудования
Инструмент `tools/catalog_parser` вынесен за пределы основного runtime.
Он собирает или нормализует данные по реально существующему оборудованию и формирует каталог, который затем может использоваться как внешний справочник.

## Слои системы

### `src/ui`
Отвечает за:
- вкладки;
- формы ввода;
- диалоги;
- таблицы;
- отображение результатов.

UI не должен быть местом хранения предметной логики.

### `src/application`
Отвечает за:
- orchestration;
- use case-сценарии;
- координацию между UI, доменом и репозиториями;
- сервисы подготовки итоговых данных.

Именно здесь находятся сценарии вроде:
- `PrepareCostSummaryUseCase`;
- `ExportCostReportUseCase`;
- `LoadDemoDatasetUseCase`;
- `BuildNpvReportUseCase`.

### `src/domain`
Отвечает за:
- предметные модели;
- финансовые формулы;
- AHP-алгоритмы;
- анализ важности критериев;
- оптимизационные процедуры.

Это слой, который должен оставаться максимально независимым от `Tkinter`, JSON-хранилищ и конкретных экранов.

### `src/infrastructure`
Отвечает за:
- JSON storage;
- runtime-репозитории;
- CSV-экспорт;
- логирование.

Это техническая прослойка, а не место для продуктовой логики.

## Поток данных в системе

Упрощённо поток выглядит так:
1. пользователь работает с вкладкой в `ui`;
2. вкладка вызывает прикладной сервис или use case из `application`;
3. сервис берёт или сохраняет данные через `infrastructure`;
4. при необходимости используется предметная логика из `domain`;
5. результат возвращается в `ui` для отображения пользователю.

## Каталог оборудования и граница с парсером

Основное приложение не занимается HTML-разбором и сетевым обходом магазинов напрямую.
Вместо этого вспомогательный инструмент обновляет нормализованный каталог оборудования, а приложение работает уже с готовым справочником.

## Состояние после завершения roadmap концептуальной связанности

После завершения roadmap основной сценарий проекта описывается не набором отдельных калькуляторов, а единой цепочкой поддержки выбора ИТ-решения:

```text
ввод компонентов → нормализация → профиль анализа → альтернативы → стоимость владения → методы анализа → DecisionReport
```

Эта цепочка связывает прежние вкладки следующим образом:

| Шаг | Что происходит | Где находится ответственность |
|---|---|---|
| Ввод компонентов | Пользователь вводит CAPEX/OPEX, энергетические параметры или загружает демо-набор | `src/ui`, `EquipmentService`, runtime-репозитории |
| Нормализация | Старые и новые записи получают `scope`, `component_type`, `client_seats` и признаки участия в анализе | `RuntimeEntityNormalizationService` |
| Профиль анализа | Для ПО/ТО выбираются свои категории, критерии, ограничения и объяснения | `AnalysisScopeProfileService` |
| Пул альтернатив | Runtime-записи, демо-конфигурации, GA-решения и AHP-альтернативы приводятся к `CandidateConfiguration` | `CandidateConfigurationService` |
| Стоимость владения | CAPEX, OPEX, внедрение, сопровождение и электроэнергия собираются в TCO | `TCOModelService`, `CostModel` |
| Финансовая оценка | Рассчитанные затраты могут стать финансовой базой NPV | `NpvReportService`, вкладка NPV |
| Итоговый отчёт | Компоненты, альтернативы, аналитика, NPV, риски и предупреждения собираются в единый артефакт | `DecisionReportService`, exporters |

## Почему методы анализа не дублируют друг друга

В текущей архитектуре каждый метод занимает своё место:

- **GA** ищет допустимые комбинации компонентов и формирует кандидатов;
- **AHP** ранжирует общий пул альтернатив по экспертным критериям;
- **анализ важности критериев** объясняет влияние отношений между критериями и выделяет недоминируемые варианты;
- **гибридная GA + AHP-оценка** показывает согласованность поискового и экспертного взгляда;
- **NPV** оценивает финансовый сценарий после того, как затраты уже собраны в TCO;
- **DecisionReport** не заменяет расчёты, а связывает их в объяснимый итог выбора.

## Редактор компонентов `SolutionComponent`

После завершения C1–C9 в проекте появился расширенный сценарий ввода нестандартных компонентов решения. Он не заменяет быстрые вкладки `ПО`, `ТО`, `OPEX` и `Электроэнергия`, а дополняет их для составных случаев: комплектов, сервисов, смешанных расходов и ручной конвертации старых sandbox-записей.

Роль редактора в общей архитектуре:

| Шаг | Что делает редактор | Как защищается аналитика |
|---|---|---|
| Ввод | Пользователь задаёт `scope`, `component_type`, стоимость, метрики и допущения | Неполные записи сохраняются как draft |
| Нормализация | `SolutionComponentNormalizationService` рассчитывает warnings и признаки допуска | `candidate_eligible=false` исключает черновик из строгой цепочки |
| Хранение | `SolutionComponentRuntimeService` пишет данные в `entities.solution_components` | Legacy-записи не мигрируют автоматически |
| Финансы | C5-мост передаёт CAPEX/OPEX/энергию в TCO и NPV-базу | Эффект не подставляется искусственно |
| Аналитика | C6-мост отдаёт strict-компоненты в AHP/GA через `CandidateConfiguration` | Параллельный формат альтернатив не создаётся |
| Отчёт | C7/C9 показывают валидные и исключённые компоненты в `DecisionReport` | Черновики видны, но не влияют на выбор |

Финальный C9-сценарий зафиксирован в `docs/architecture/solution_component_demo_control.md` и `docs/demo/solution_component_editor_defense_guide.md`.

## Границы legacy-частей

Старая вкладка ИТ-инфраструктуры больше не конкурирует с вкладками ПО/ТО. Она описана как `ИТ-песочница`: свободные записи остаются видимыми, но не попадают в строгий пул `CandidateConfiguration` без явной конвертации в `SolutionComponent`. Если пользователь выбирает ручное преобразование, новая запись получает `origin = converted_sandbox`, сохраняет metadata исходной строки и проходит те же warnings/статусы допуска, что и компонент, созданный напрямую в редакторе.
