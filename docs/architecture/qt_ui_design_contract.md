# UX-контракт миграции на PySide6/Qt

## 1. Назначение

Этот документ фиксирует обязательные правила для новой Qt-версии интерфейса калькулятора стоимости ИТ-инфраструктуры. Контракт нужен, чтобы миграция с Tkinter/ttkbootstrap не повторила старую проблему: много равноправных панелей, длинные пояснения прямо на экране, перегруженные таблицы и неочевидный порядок действий.

Контракт действует для всех будущих файлов `src/ui_qt/**` и для любых изменений старого `src/ui/**`, пока legacy-слой ещё остаётся в проекте.

## 2. Главный принцип

Один экран решает одну смысловую задачу.

Пользователь должен видеть:

- где он находится;
- что уже готово;
- что нужно сделать сейчас;
- какой следующий шаг оптимален.

Если для понимания экрана нужен длинный текст, значит экран спроектирован неверно. Длинный текст переносится в tooltip, справку, раскрываемую секцию `Подробнее` или отдельный диалог.

## 3. Ограничения видимого текста

Обязательные правила:

- visible label: не больше 10 слов;
- рекомендуемая длина label: 1–4 слова;
- заголовки групп: 1–5 слов;
- кнопки: 1–3 слова;
- пункты меню: 1–4 слова;
- статусные сообщения: короткая причина и короткое действие;
- длинные пояснения запрещены в обычных label;
- длинные пояснения разрешены только в tooltip/help/details/export/docs/log.

Примеры:

| Плохо | Хорошо |
|---|---|
| `Задайте ограничения и веса критериев, затем нажмите кнопку запуска` | `Параметры GA` + tooltip |
| `Гибридная панель не запускает GA и AHP заново` | `Гибридный итог` + tooltip |
| `Не удалось выполнить расчёт из-за отсутствия набора данных` | `Нет данных` |

## 4. Минимум элементов

На каждом экране должен быть один главный action. Остальные действия получают меньший визуальный вес или скрываются.

Правила:

- главный action находится внизу справа или в верхней action-панели;
- вторичные действия уходят в `Ещё`, context-menu или раскрываемый блок;
- опасные действия требуют подтверждения;
- одинаково важные кнопки рядом запрещены;
- пустой экран показывает одно действие восстановления;
- таблица не должна быть первым и единственным ответом интерфейса.

## 5. Навигация и маршрут

Корневые разделы сохраняются:

1. `Программное обеспечение`.
2. `Техническое обеспечение`.
3. `Редактор компонентов`.
4. `Электроэнергия`.
5. `NPV-анализ`.
6. `Экспорт`.
7. `Настройки`.

Внутри `Программного обеспечения` и `Технического обеспечения` используется маршрут:

```text
Данные → GA → AHP → Pareto → Гибрид
```

Правила маршрута:

- активный шаг всегда виден;
- следующий шаг визуально понятен;
- недоступный шаг объясняет причину коротко;
- шаг не должен требовать ручного поиска нужной панели;
- результаты предыдущих шагов не пересчитываются молча.

## 6. Моноэкранный формат

Базовая компоновка нового UI:

```text
верхняя панель: раздел, статус, основные действия
маршрут: текущий шаг внутри области
основной экран: одна смысловая задача
нижний статус: короткие события и ошибки
```

Запрещено делать основой интерфейса:

- постоянные split-панели;
- Adobe-подобные доки;
- ручное растягивание смысловых зон;
- несколько равноправных экранов внутри одного экрана;
- длинные инструкции поверх рабочих данных.

Разрешено:

- раскрывать необязательные параметры;
- открывать детали по требованию;
- лениво загружать тяжёлые экраны;
- показывать таблицы только после выбора действия или в зоне результата.

## 7. Темы и визуальный стиль

Обязательные темы:

- мягкая светлая;
- мягкая тёмная.

Стиль:

- flat design как база;
- лёгкий неоморфизм через форму и поверхности;
- без тяжёлых теней;
- без декоративной перегрузки;
- спокойный accent-color;
- одинаковые отступы;
- высокий контраст текста;
- предсказуемые размеры кнопок и полей.

Базовый шрифт:

- Windows: `Segoe UI`;
- fallback: `Arial`;
- основной текст: 10–11 pt;
- заголовки: 14–18 pt;
- таблицы: 9–10 pt.

## 8. Адаптивность

Интерфейс должен выдерживать изменение размера окна без поломки сценария.

Правила:

- корневая навигация не ломается при сужении;
- вторичные действия сворачиваются в `Ещё`;
- таблицы получают scroll, а не ломают layout;
- карточки перетекают в 1–2 колонки;
- вспомогательные блоки уходят вниз или скрываются;
- критически важные actions не скрываются;
- иконки без текста обязаны иметь tooltip.

## 9. Производительность

Qt UI должен стартовать без видимой долгой прорисовки.

Правила:

- не строить все экраны на старте;
- использовать lazy-loading root-разделов;
- тяжёлые расчёты не выполнять в GUI-потоке;
- не пересчитывать GA/AHP/Pareto при переключении шагов;
- не заполнять большие таблицы синхронно без необходимости;
- показывать короткий статус выполнения;
- избегать полной перерисовки экрана без причины.

Минимальные метрики для поздних патчей:

- время импорта UI;
- время создания main window;
- время открытия первого экрана;
- время загрузки demo dataset.

## 10. Состояния экранов

Каждый экран должен явно поддерживать состояния:

- `Нет данных`;
- `Готово`;
- `Идёт расчёт`;
- `Результат готов`;
- `Результат устарел`;
- `Нужен предыдущий шаг`;
- `Ошибка`.

Сообщение состояния должно быть коротким. Подробности уходят в `Подробнее`.

## 11. Граница UI и логики

Новый Qt UI не должен переносить расчётную логику в виджеты.

Разрешённые зависимости UI:

- application services;
- use cases;
- repositories;
- view-model/presenter слой.

Запрещено:

- дублировать GA/AHP/Pareto/NPV расчёты в `src/ui_qt/**`;
- обращаться к Tkinter-виджетам из Qt-кода;
- использовать `TreeviewCrudRepository` в Qt-коде;
- хранить бизнес-правила в обработчиках кнопок.

## 12. Проверки каждого UI-патча

Каждый будущий UI-патч должен проходить минимум:

```text
python -B tools/check_ui_text_density.py
python -B -c "import ast,pathlib; ..."
```

Для кодовых Qt-патчей дополнительно:

- smoke-импорт PySide6;
- создание QApplication в тестовом режиме;
- проверка запуска без построения всех тяжёлых экранов;
- проверка, что расчётная логика импортируется без GUI;
- проверка ключевого сценария изменённого экрана.

## 13. Критерий приёмки UI-патча

Патч считается приемлемым, если:

- он не добавляет видимых label длиннее 10 слов;
- у экрана есть один очевидный главный action;
- длинные объяснения не занимают рабочую область;
- пользователь видит следующий шаг;
- расчётная логика не переехала в UI;
- старт приложения не стал заметно тяжелее;
- изменения можно откатить без потери данных.
