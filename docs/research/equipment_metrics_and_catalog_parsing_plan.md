# План развития количественных метрик оборудования и связи с парсингом каталога

## Назначение документа

Документ выделяет отдельную тему, которую не стоит смешивать с перераспределением вкладок GA/AHP/Pareto: постепенный переход от абстрактной `производительности` к конкретным характеристикам оборудования.

Цель — подготовить реалистичную и не переусложнённую модель метрик для ТО, которую можно частично заполнять вручную, частично получать из каталога и частично извлекать из названий товаров.

---

## 1. Почему нельзя оставлять абстрактную производительность

Общее поле `performance` удобно для демонстрации, но плохо подходит для объяснения результата:

| Проблема | Последствие |
|---|---|
| непонятная размерность | пользователь не знает, что именно означает число |
| разные типы оборудования | ПК, сервер и роутер нельзя честно сравнить одним score |
| сложность защиты | преподаватель может спросить, откуда взята производительность |
| слабая связь с парсингом | из карточки магазина проще извлечь RAM, CPU, порты и Wi-Fi, чем абстрактный score |

Поэтому `performance` лучше заменить набором конкретных метрик, которые зависят от типа компонента.

---

## 2. Общий принцип

Для каждого `component_type` должен быть свой минимальный набор метрик. Если метрика не заполнена, метод не должен делать вид, что она равна нулю и ухудшает вариант. Лучше показывать предупреждение `нет данных`.

| Правило | Смысл |
|---|---|
| метрики зависят от типа компонента | ПК сравнивается с ПК, роутер — с роутером |
| часть метрик может быть пустой | MVP не требует полного каталога |
| парсер является помощником, а не источником истины | пользователь может исправить или заполнить поле вручную |
| энергия остаётся отдельным мостом к OPEX/TCO | БП/мощность влияет не только на оптимизацию, но и на эксплуатационные расходы |

---

## 3. Рабочие станции и ПК

Минимальные поля:

| Поле | Тип | Использование |
|---|---|---|
| `ram_gb` | число | мягкий критерий качества |
| `cpu_cores` | число | мягкий критерий вычислительной ёмкости |
| `storage_gb` | число | мягкий критерий объёма хранения |
| `storage_type` | строка | ручной признак HDD/SSD/NVMe |
| `storage_type_rank` | число | нормализованный ранг типа накопителя |
| `client_seats` | число | жёсткий фильтр рабочих мест |
| `max_power_watts` | число | критерий энергии и вход для вкладки электроэнергии |

Для базового MVP достаточно `ram_gb`, `cpu_cores`, `storage_gb`, `client_seats`, `max_power_watts`. Тип накопителя можно добавить позже.

---

## 4. Серверы

Минимальные поля:

| Поле | Тип | Использование |
|---|---|---|
| `ram_gb` | число | мягкий критерий |
| `cpu_cores` | число | мягкий критерий |
| `storage_gb` | число | мягкий критерий |
| `drive_bays` | число | будущий признак масштабируемости |
| `redundancy_score` | число | будущий признак отказоустойчивости |
| `max_power_watts` | число | энергия и OPEX |

Не нужно сразу вводить сложную серверную модель с RAID, SLA и виртуализацией. Для дипломного MVP достаточно показать, что серверы оцениваются по явным техническим параметрам, а не по абстрактному score.

---

## 5. Сетевое оборудование

Для роутеров, коммутаторов и сетевых устройств нужен отдельный набор метрик.

| Поле | Тип | Использование |
|---|---|---|
| `lan_ports` | число | количество проводных портов |
| `lan_speed_mbps` | число | скорость LAN |
| `wan_speed_mbps` | число | скорость WAN, если распознана |
| `wifi_total_mbps` | число | суммарная заявленная скорость Wi-Fi |
| `wifi_generation_rank` | число | условный ранг Wi-Fi 4/5/6/7 |
| `ipv6_support` | булево | функциональный признак |
| `max_power_watts` | число | энергия, если доступна |

Для коммутаторов Wi-Fi-поля будут пустыми, а для роутеров — важными. Поэтому профиль критериев должен учитывать `component_type`.

---

## 6. Связь с названием товара DNS-Shop

Название товара может быть частичным источником характеристик. Пример:

`Wi-Fi роутер TP-Link Archer Air R5 [1 LAN, 1000 Мбит/с, 100 Мбит/с, 4 (802.11n), 5 (802.11ac), 6 (802.11ax), Wi-Fi 2976 Мбит/с, IPv6]`

Потенциальные извлечения:

| Фрагмент | Возможное поле |
|---|---|
| `1 LAN` | `lan_ports = 1` |
| `1000 Мбит/с` | `lan_speed_mbps = 1000` или другое поле после уточнения позиции |
| `100 Мбит/с` | `wan_speed_mbps = 100` или fallback-speed |
| `802.11n`, `802.11ac`, `802.11ax` | список стандартов Wi-Fi |
| `Wi-Fi 2976 Мбит/с` | `wifi_total_mbps = 2976` |
| `IPv6` | `ipv6_support = true` |

Важно: извлечение из названия должно быть best-effort. Если парсер ошибся, пользователь должен видеть исходную строку и иметь возможность поправить распознанные поля.

---

## 7. Минимальный парсерный контракт

Парсер не должен сразу пытаться понимать весь рынок. Достаточно контракта:

| Поле | Смысл |
|---|---|
| `raw_name` | исходное название |
| `parsed_metrics` | распознанные метрики |
| `parse_warnings` | предупреждения о неоднозначности |
| `confidence` | условная уверенность распознавания |
| `source` | источник данных: DNS-Shop, ручной ввод, demo |

Если характеристика распознана с низкой уверенностью, её можно показывать в UI, но не использовать в оптимизации без подтверждения пользователя.

---

## 8. Связь с энергопотреблением и БП

Блок питания и максимальная мощность не должны быть просто ещё одним критерием в GA. Они связаны сразу с несколькими частями проекта:

| Где участвует | Как |
|---|---|
| ТО-оптимизация | как мягкий критерий `меньше энергопотребление лучше` или как лимит мощности |
| вкладка электроэнергии | как вход для расчёта эксплуатационных расходов |
| TCO/NPV | как регулярный OPEX через стоимость электричества |
| DecisionReport | как пояснение эксплуатационной нагрузки |

Поэтому `max_power_watts` должен быть общим полем технического компонента, а не локальным полем одной вкладки.

---

## 9. Очередность внедрения

### Этап E1. Документировать поля и fallback

- описать минимальные поля для ПК, серверов и сети;
- зафиксировать, что пустая метрика означает `нет данных`, а не `0`;
- добавить предупреждения в отчёт.

### Этап E2. Добавить поля в нормализацию компонента

- расширить runtime-нормализацию без ломки старых JSON;
- поддержать ручной ввод или demo-поля;
- сохранить старые `properties` как fallback.

### Этап E3. Подключить метрики к профилю ТО

- заменить абстрактную производительность на конкретные метрики;
- не включать все метрики сразу в UI, а выбрать 2–3 самые понятные;
- мощность оставить связанной с энергией.

### Этап E4. Добавить best-effort извлекатель из названия

- начать с сетевых устройств, потому что пример DNS-Shop хорошо структурирован;
- возвращать предупреждения при неоднозначности;
- не использовать сомнительные поля как обязательные.

### Этап E5. Синхронизировать с экспортом

- DecisionReport показывает, какие метрики были заполнены;
- если победитель выбран при неполных технических данных, отчёт содержит предупреждение;
- demo-сценарий показывает хотя бы один пример распознанных сетевых характеристик.

---

## 10. Защитная формулировка

В проекте техническая производительность не задаётся произвольным баллом. Для разных типов оборудования используются конкретные измеримые характеристики: память, ядра, накопитель, мощность, сетевые порты, скорость и поддерживаемые стандарты. Если данных недостаточно, система показывает предупреждение и не скрывает неопределённость.

---

## Статус реализации Патча 8

Базовая часть этапов E2–E3 реализована отдельным devctl-патчем:

- runtime-нормализация поддерживает `ram_gb`, `cpu_cores`, `storage_gb`, `max_power_watts` и сетевые поля `lan_ports`, `lan_speed_mbps`, `wifi_total_mbps`, `ipv6_support`;
- `max_power` сохранён как алиас для совместимости с вкладкой электроэнергии;
- профиль ТО использует явные вычислительные и сетевые метрики вместо абстрактной производительности;
- GA/AHP/Pareto получают эти метрики через общий профиль и общий пул альтернатив;
- при неполных характеристиках добавляются `metric_warnings`;
- demo dataset содержит несколько заполненных примеров для серверов, рабочих станций и сетевых устройств.

Не реализовано в этом патче: best-effort парсер DNS-Shop. Он остаётся отдельным Патчем 9, чтобы не смешивать модель метрик и распознавание характеристик из строк каталога.


## Статус реализации Патча 9

Best-effort часть этапа E4 реализована отдельным devctl-патчем:

- добавлен `tools/catalog_parser/dns_network_metrics.py`;
- из названий сетевых устройств извлекаются LAN-порты, LAN-скорость, Wi-Fi стандарты, Wi-Fi total Mbps и IPv6;
- результат содержит `parsed_metrics`, `parse_warnings`, `confidence` и `parse_source`;
- ручные поля из `specs` имеют приоритет над распознанными;
- неизвестный формат возвращает предупреждение, а не exception;
- `data/examples/parser/dns_routers.json` содержит demo-роутеры для проверки.

Парсер остаётся вспомогательным инструментом каталога, а не частью runtime GUI.


## Статус реализации этапа E5

Экспорт технических метрик и их качества реализован отдельным devctl-патчем:

- staging сохраняет `review`, parser warnings/confidence/source и `field_provenance`;
- runtime-строки каталога переносят эти сведения в `catalog_metadata`;
- `DecisionReport` содержит блок `catalog_data_quality` с заполненными и отсутствующими метриками;
- неполные данные и parser warnings попадают в общие warnings/risks отчёта;
- JSON сохраняет полный provenance, Markdown показывает таблицу качества, CSV кандидатов содержит агрегированные `technical_metrics` и `metric_warnings`;
- `scripts/check_catalog_decision_report.py` проверяет путь от DNS HTML-снимка до трёх форматов экспорта.

Таким образом, этапы E1–E5 закрыты: parser остаётся помощником, а неопределённость данных видна пользователю до итогового обоснования выбора.
