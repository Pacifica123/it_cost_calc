# Roadmap развития концептуальной связанности проекта

## Назначение roadmap

Этот документ развивает идеи из `conceptual_cohesion_manifest.md` и переводит их в последовательный план дальнейших работ. Манифест фиксирует целевую логику проекта: перейти от набора отдельных расчётных вкладок к единой системе поддержки выбора ИТ-решения. Roadmap нужен для того, чтобы будущие патчи можно было планировать не как разрозненные доработки, а как цепочку связанных архитектурных шагов.

Главный ориентир roadmap:

```text
ИТ-решение → компоненты → стоимость и метрики → альтернативы → методы анализа → отчёт выбора
```

Документ намеренно оформлен как иерархический чеклист. Корневые блоки можно использовать как крупные этапы работ, а вложенные пункты — как основу для отдельных devctl-патчей.

---

## Общие правила выполнения roadmap

- [ ] **Сохранять обратную совместимость пользовательских данных.**
  - [ ] Не ломать существующие JSON-файлы в `data/generated/` и демо-фикстуры.
  - [ ] Любое новое поле вводить сначала как необязательное.
  - [ ] Для старых записей сохранять вывод значений из текущих категорий.
  - [ ] В документации явно помечать, где используется переходная логика.
- [ ] **Не переносить предметную логику в UI.**
  - [ ] UI должен выбирать сценарий, показывать форму и отображать результат.
  - [ ] Правила ПО/ТО, критерии, ограничения и объяснения должны постепенно уходить в application/domain.
  - [ ] При каждом изменении вкладки проверять, не появляется ли в ней новая скрытая предметная модель.
- [ ] **Собирать изменения малыми воспроизводимыми патчами.**
  - [ ] Один патч должен иметь один понятный архитектурный смысл.
  - [ ] Документационные патчи должны предшествовать рискованным изменениям кода.
  - [ ] Для кодовых патчей обязательно сохранять проверки без `__pycache__` и без cacheprovider тестового раннера.
- [ ] **Укреплять связность, а не добавлять изолированные функции.**
  - [ ] Перед новой вкладкой или расчётом фиксировать, какую сущность она создаёт или анализирует.
  - [ ] Проверять, участвует ли результат в общей модели стоимости, аналитике или отчёте.
  - [ ] Не добавлять новый формат альтернатив, если его можно выразить через общий формат.

## Минимальный целевой результат всей дорожной карты

После выполнения roadmap проект должен восприниматься не как набор отдельных калькуляторов, а как связанная система поддержки решения:

```text
1. Пользователь описывает состав ИТ-решения.
2. Компоненты получают область, тип, стоимость и метрики.
3. Из компонентов формируются альтернативные конфигурации.
4. CAPEX, OPEX и электроэнергия собираются в TCO.
5. AHP, анализ важности критериев, GA и гибридная оценка сравнивают один пул альтернатив.
6. NPV использует рассчитанные финансовые потоки.
7. DecisionReport объясняет итоговый выбор и ограничения анализа.
```

Такой результат напрямую продолжает манифест концептуальной связанности: отдельные расчёты сохраняются, но начинают работать как части одного процесса выбора ИТ-решения.
