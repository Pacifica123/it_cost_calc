# Интеграция SolutionComponent с DecisionReport и экспортом

## Назначение

Этот документ фиксирует результат этапа C7 roadmap редактора компонентов. До C7 компоненты редактора уже могли храниться, нормализоваться, попадать в финансовую цепочку и использоваться в AHP/GA через `CandidateConfiguration`. Этап C7 делает этот путь объяснимым в итоговом отчёте и экспортных файлах.

Главная цель: пользователь и проверяющий должны видеть не только победителя, но и то, какие `SolutionComponent` участвовали в выборе, какие были исключены и почему.

## Что добавлено в DecisionReport

`DecisionReportService` теперь формирует отдельный блок `solution_component_report`. Он не заменяет общий список `components`, а собирает поверх него человеко- и машинно-читаемое объяснение роли редактора.

| Поле | Смысл |
|---|---|
| `role_explanation` | кратко объясняет, зачем нужен редактор компонентов и почему черновики отделяются от строгой аналитики |
| `counts` | количество всех, валидных, исключённых и связанных с альтернативами компонентов |
| `valid_components` | компоненты, прошедшие нормализацию и допуск к аналитике |
| `excluded_components` | черновики, blocked-компоненты и записи, не влияющие на выбор |
| `alternative_links` | связь компонента с `CandidateConfiguration` |
| `metrics_used_by_method` | ориентир по метрикам, которые были использованы аналитическими методами |
| `export_notes` | пояснения по JSON/Markdown/CSV-экспорту |

Такой блок нужен для дипломной защиты: он показывает, что редактор не создаёт скрытую вторую модель, а продолжает общую цепочку проекта.

## Валидные компоненты

Валидный компонент в отчёте содержит:

- `id`, название, `scope`, `component_type`, `origin` и статус;
- признаки допуска `candidate_eligible`, `analysis_ready`, `tco_eligible`, `energy_eligible`, `npv_eligible`;
- финансовый вклад в первый год: разовые расходы, ежемесячный OPEX, годовой OPEX и суммарную стоимость первого года;
- метрики, которые важны для AHP/GA: например мощность, рабочие места, лицензии, функциональность, поддержка, надёжность;
- ссылки на альтернативы, куда компонент попал через `CandidateConfiguration`.

Это закрывает проблему, когда пользователь видит итоговую рекомендацию, но не понимает, какие строки редактора реально повлияли на расчёт.

## Исключённые компоненты и черновики

Исключённые компоненты не скрываются. Они попадают в `excluded_components` с причинами:

- отсутствует `scope`;
- отсутствует `component_type`;
- компонент сохранён как `draft`;
- компонент не прошёл профиль ПО/ТО;
- mixed-компонент требует разбиения или явных допущений;
- есть warnings по стоимости, энергии или регулярности расходов.

В Markdown-отчёте для них выводится отдельная таблица. После таблицы явно указано, что неполные компоненты не влияли на итоговый выбор, потому что не получили `candidate_eligible`/`analysis_ready` и не были связаны с `CandidateConfiguration`.

## JSON-экспорт

JSON сохраняет обратную совместимость через старую обёртку `report`, но дополнительно содержит блок `solution_components`.

Это даёт два уровня чтения:

- старые потребители могут продолжать читать полный `report`;
- новые инструменты могут сразу брать `solution_components` как машинный snapshot редактора.

В JSON не теряются:

- полный snapshot компонентов;
- признаки допуска;
- warnings и blocking errors;
- происхождение `origin`;
- связь с альтернативами;
- финансовый вклад;
- метрики, использованные для объяснения.

## Markdown-экспорт

Markdown-отчёт получил расширенный раздел `Компоненты редактора SolutionComponent`.

Раздел состоит из трёх частей:

1. краткое объяснение роли редактора;
2. таблица валидных компонентов, участвующих в выборе;
3. таблица исключённых компонентов и черновиков.

Это делает отчёт пригодным для демонстрации: можно показать, что редактор поддерживает сложные компоненты, но не даёт неполным строкам незаметно исказить AHP/GA/TCO.

## CSV-экспорт

CSV кандидатов сохранён совместимым: он по-прежнему выгружает срез `candidate_configurations` и не перегружается метриками компонентов.

Для компонентов добавлена отдельная выгрузка `decision_report_solution_components.csv`. Она содержит:

- секцию `valid` или `excluded_or_draft`;
- идентификацию компонента;
- `scope`, `component_type`, `origin`, статус;
- признаки допуска;
- финансовый вклад;
- связанные `candidate_ids`;
- метрики в компактном JSON-тексте;
- warnings или причины исключения.

Разделение CSV важно: финансовые поля и профильные метрики не смешиваются с таблицей альтернатив и не ломают старый сценарий просмотра кандидатных конфигураций.

## Граница C7

C7 не меняет алгоритмы AHP, GA, GA + AHP, TCO или NPV. Он только делает уже построенную цепочку объяснимой в отчёте и экспорте.

Не входит в C7:

- ручная конвертация старых sandbox-записей;
- новые UI-действия миграции;
- новая модель альтернатив;
- изменение правил ранжирования.

Эти задачи остаются для следующих этапов roadmap.

## Проверка

Для проверки добавлен smoke-сценарий `scripts/run_solution_component_report_export_demo.py`. Он выполняет без `pytest`:

- собирает `DecisionReport` из demo-fixture;
- проверяет наличие `solution_component_report`;
- проверяет JSON-блок `solution_components`;
- проверяет Markdown-разделы валидных и исключённых компонентов;
- проверяет, что candidate CSV остался совместимым;
- проверяет отдельный component CSV;
- экспортирует файлы во временный каталог и перечитывает их.

## Формулировка для защиты

Редактор компонентов не просто добавляет ещё одну вкладку. Он позволяет описывать нестандартные элементы ИТ-решения, но допускает их в расчёты только после нормализации. В `DecisionReport` видно, какие компоненты участвовали в выборе, какой финансовый вклад они дали, какие метрики были использованы и какие черновики были исключены. Поэтому итоговый выбор остаётся объяснимым и методически защищённым.
