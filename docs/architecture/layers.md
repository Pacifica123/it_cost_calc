# Слои приложения

## Зачем понадобилось явное разбиение по слоям

Изначально проект рос как прикладной прототип: интерфейс, данные, расчёты и промежуточные утилиты находились слишком близко друг к другу.
Сейчас ответственность распределена по слоям так, чтобы код было проще сопровождать и развивать.

## Слой `ui`

Каталог: `src/ui/`

Здесь находятся:
- вкладки интерфейса;
- диалоги;
- переиспользуемые визуальные компоненты;
- код, связанный с отображением и обработкой действий пользователя.

### Что разрешено
- собирать форму;
- читать значения из виджетов;
- показывать сообщения и таблицы;
- вызывать use case или сервис прикладного слоя.

### Чего здесь быть не должно
- сложной предметной математики;
- хранения бизнес-состояния как главного источника истины;
- прямой логики сериализации и экспорта.

## Слой `application`

Каталог: `src/application/`

Это слой сценариев использования.
Он координирует взаимодействие между `ui`, `domain` и `infrastructure`.

Здесь находятся:
- use case-модули;
- прикладные сервисы;
- compat-обвязка для перехода от legacy-структуры к новой архитектуре;
- интерфейсы портов для репозиториев.

### Что делает этот слой
- получает команды из UI;
- забирает или обновляет данные в репозитории;
- вызывает доменные расчёты;
- подготавливает результат в форме, удобной для интерфейса.

## Слой `domain`

Каталог: `src/domain/`

Это ядро предметной логики проекта.

Здесь находятся:
- `dataclass`-модели;
- финансовые расчёты;
- AHP;
- анализ важности критериев;
- оптимизационные алгоритмы.

### Требование к слою
Этот слой должен как можно меньше зависеть от внешних деталей:
- `Tkinter`;
- файловой системы;
- JSON;
- CSV;
- конкретных экранов.

## Слой `infrastructure`

Каталог: `src/infrastructure/`

Здесь находятся технические адаптеры:
- JSON storage;
- runtime-репозитории;
- CSV-экспорт;
- логирование.

Это не предметная логика и не UI, а техническая обвязка проекта.

## Переходные зоны

В проекте ещё присутствуют участки переходного характера:
- compatibility wrapper’ы;
- `TreeviewCrudRepository`;
- часть legacy-вкладок, которые ещё не до конца очищены от старой связки UI и локального состояния.

Они сохранены ради совместимости и постепенного развития кода.

## Принцип взаимодействия слоёв

Нормальный поток вызовов выглядит так:

`UI -> Application -> Domain / Infrastructure -> Application -> UI`

Это означает:
- UI не должен сам хранить предметную модель как единственный источник истины;
- domain не должен знать о `Treeview` и messagebox;
- infrastructure не должен принимать архитектурные решения за предметную логику.

## Практический смысл такого разбиения

Такое разбиение делает проект:
- понятнее для чтения;
- безопаснее для развития;
- удобнее для тестирования;
- пригоднее для дальнейшего роста.
