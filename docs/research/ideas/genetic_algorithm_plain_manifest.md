# Простое объяснение генетического алгоритма

Статус документа: **архивно-пояснительный манифест**.  
Идея ГА уже встроена в общий сценарий `CandidateConfiguration → TCO → DecisionReport`, а этот документ сохранён как простое объяснение для защиты и первичного чтения. Он не является техническим заданием, планом реализации или математическим доказательством корректности.

## Суть в одном абзаце

Генетический алгоритм в проекте нужен для того, чтобы автоматически собрать подходящую конфигурацию ИТ-инфраструктуры из уже имеющихся позиций оборудования и расходов. Он не придумывает оборудование сам, а выбирает подмножество из текущих данных проекта: серверы, рабочие места, сетевое оборудование, лицензии и другие элементы. Затем алгоритм проверяет, проходит ли конфигурация ограничения, и оценивает, насколько она хороша по заданным критериям.

## Какую задачу решает ГА

Обычный пользователь может вручную посчитать стоимость уже выбранного решения. Но если вариантов оборудования много, появляется другая задача: не просто посчитать готовую конфигурацию, а помочь найти удачную комбинацию.

В этом смысле генетический алгоритм отвечает на вопрос:

```text
Какие позиции из доступного набора стоит включить в итоговую конфигурацию,
чтобы она была допустимой, не слишком дорогой и достаточно качественной?
```

Для дипломного приложения это важно, потому что проект постепенно уходит от простого калькулятора затрат к системе поддержки принятия решений.

## Что подаётся на вход

На вход ГА поступает не абстрактная формула, а набор данных приложения:

- список доступных элементов инфраструктуры;
- стоимость элементов;
- категории элементов;
- данные о мощности, если они заполнены;
- ограничения пользователя;
- веса критериев;
- параметры запуска алгоритма.

Пример ограничений:

- максимальный бюджет;
- максимальное энергопотребление;
- минимальное число клиентских мест;
- обязательные категории оборудования.

Пример критериев:

- чем ниже стоимость, тем лучше;
- чем ниже мощность, тем лучше;
- чем больше покрытие нужных категорий, тем лучше;
- чем больше клиентская ёмкость, тем лучше.

## Что такое конфигурация

Конфигурация — это один возможный набор выбранных элементов.

Например, есть список кандидатов:

```text
1. Сервер A
2. Сервер B
3. Рабочие места
4. Сетевое оборудование
5. Лицензии
```

Одна конфигурация может выглядеть так:

```text
Сервер A + рабочие места + сетевое оборудование
```

Другая так:

```text
Сервер B + рабочие места + лицензии
```

Задача ГА — попробовать много таких комбинаций и оставить наиболее удачные.

## Почему алгоритм называется генетическим

ГА работает по аналогии с искусственным отбором.

Сначала создаётся набор случайных или частично подготовленных решений. Этот набор называется популяцией.

```text
Поколение 1:
- вариант A
- вариант B
- вариант C
- вариант D
```

Затем каждый вариант получает оценку качества. Более удачные варианты сохраняются, комбинируются друг с другом и немного случайно изменяются. Так появляется новое поколение решений.

В проектной терминологии это выглядит так:

| Понятие | Простой смысл |
|---|---|
| Хромосома | Маска выбора: брать элемент или не брать |
| Популяция | Набор разных конфигураций |
| Fitness | Итоговая оценка качества конфигурации |
| Отбор | Сохранение более удачных решений |
| Скрещивание | Комбинирование двух решений |
| Мутация | Небольшое случайное изменение решения |
| Поколение | Очередной шаг улучшения набора решений |

## Что такое хромосома в этом проекте

Хромосома — это простая бинарная маска.

```text
(1, 0, 1, 0, 1)
```

Каждая позиция соответствует одному элементу из списка:

- `1` — элемент включён в конфигурацию;
- `0` — элемент не выбран.

Если в списке пять элементов, маска `(1, 0, 1, 0, 1)` означает: взять первый, третий и пятый элемент.

Такой формат удобен, потому что задача подбора инфраструктуры сводится к выбору подмножества из общего набора кандидатов.

## Как ГА понимает, что вариант хороший

Каждая конфигурация получает итоговую оценку — `fitness`.

Смысл fitness простой:

```text
чем выше fitness, тем лучше конфигурация с точки зрения заданных критериев и весов
```

Но критерии измеряются в разных единицах:

- стоимость — в рублях;
- мощность — в ваттах;
- клиентские места — в штуках;
- покрытие категорий — в количестве категорий.

Их нельзя честно сложить напрямую. Поэтому значения приводятся к сопоставимой шкале, а затем объединяются с учётом весов.

Упрощённая логика такая:

```text
итоговая оценка = вес стоимости × оценка стоимости
                 + вес мощности × оценка мощности
                 + вес клиентских мест × оценка клиентских мест
                 + вес покрытия категорий × оценка покрытия категорий
```

Если для критерия действует правило «меньше — лучше», например для стоимости, он предварительно приводится к общей логике «чем лучше, тем выше оценка».

## Что возвращает ГА

ГА возвращает не только один победивший вариант. Важный результат — это пул лучших кандидатных конфигураций.

В результате можно получить:

- лучшее найденное решение;
- состав выбранной конфигурации;
- значения критериев;
- проверку ограничений;
- историю сходимости по поколениям;
- несколько лучших альтернатив для дальнейшего анализа.

Это важно для связки с AHP: AHP не обязан сравнивать весь огромный набор возможных комбинаций, он может работать с уже найденным пулом сильных кандидатов.

## Как ГА связан с AHP

В текущей логике ГА и AHP выполняют разные роли.

```text
ГА ищет конфигурации.
AHP объяснимо ранжирует найденные конфигурации.
```

ГА хорошо подходит для поиска, потому что он может автоматически собрать варианты из большого набора элементов.

AHP хорошо подходит для объяснения выбора, потому что он позволяет сравнить уже сформированные альтернативы по критериям и весам.

Вместе это можно понимать так:

```text
ГА формирует кандидатов → AHP помогает выбрать и объяснить предпочтительный вариант
```

## Что ГА не гарантирует

Генетический алгоритм не нужно объяснять как магический способ найти абсолютный идеал.

Корректная формулировка:

```text
ГА ищет хорошее допустимое решение в большом пространстве вариантов.
```

Некорректная формулировка:

```text
ГА всегда гарантированно находит единственное оптимальное решение.
```

ГА является эвристическим методом. Это значит, что он обычно помогает быстро найти качественные варианты, но не доказывает глобальную оптимальность результата.

## Как это объяснять пользователю

Пользователю не нужно знать внутреннюю математику. Достаточно такой логики:

1. Система берёт текущие данные об оборудовании.
2. Пользователь задаёт ограничения и важность критериев.
3. ГА собирает разные возможные конфигурации.
4. Недопустимые варианты отбрасываются.
5. Допустимые варианты оцениваются по качеству.
6. Лучшие варианты показываются пользователю.
7. При необходимости AHP дополнительно ранжирует найденные конфигурации.

## Короткая формулировка для защиты

> Генетический алгоритм используется как поисковый механизм. Он берёт текущий набор оборудования, формирует разные допустимые конфигурации, проверяет ограничения и оценивает варианты по нескольким критериям. Это позволяет перейти от простого расчёта стоимости уже выбранного решения к автоматическому подбору и обоснованию конфигурации.

## Граница ответственности

ГА отвечает за поиск и первичную оценку конфигураций. Он не должен подменять AHP, экспертный выбор или финансовый анализ. Его роль — сформировать качественный пул допустимых вариантов, который затем можно объяснять, сравнивать и дорабатывать другими методами.
