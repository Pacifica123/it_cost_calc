# Staging-импорт каталога оборудования

## Назначение

Внешний каталог содержит варианты для анализа, а не готовый план закупки. Поэтому
записи из JSON, CSV и XLSX не попадают напрямую в runtime-категории ТО. Сначала
они сохраняются в `data/generated/catalog/catalog_staging.json`, проходят
валидацию, при необходимости исправляются и получают явный статус пользователя.

## Поток

```text
источник -> schema v1/v2 или плоская таблица -> staging -> подтверждение
         -> server/client/network -> GA -> AHP -> Pareto -> Hybrid
         -> выбранная конфигурация -> CAPEX/OPEX/энергия/NPV
```

Парсер и импорт не рассчитывают стоимость всей исходной выдачи. До Hybrid каждая
импортированная строка остаётся только альтернативой.

## Статусы

| Статус | Значение |
|---|---|
| `pending` | запись загружена и ожидает решения |
| `approved` | пользователь разрешил импорт |
| `rejected` | запись исключена пользователем |
| `blocked` | есть ошибка категории, цены или валюты |
| `imported` | запись уже перенесена в runtime ТО |

## Поддерживаемый маппинг

| Каталог | Runtime-категория | Тип компонента |
|---|---|---|
| server, rack_server, tower_server | `server` | `server` |
| prebuilt_pc, workstation, desktop, laptop | `client` | `workstation` |
| peripheral, printer, monitor | `client` | `peripheral` |
| router, switch, access_point, network_device | `network` | `network_device` |

CPU, GPU, RAM, материнские платы, накопители и корпуса намеренно блокируются.
Текущий оптимизатор выбирает готовые серверные, клиентские и сетевые устройства,
а не собирает совместимый компьютер из деталей. Пользователь может вручную
исправить ошибочную исходную категорию или назначение, но такое изменение
сохраняется в `manual_overrides` и показывается в проверке.

## Staging v2 и ручная проверка

Staging v2 хранит две версии карточки:

- `source_catalog_item` — неизменные нормализованные данные источника;
- `catalog_item` — эффективные данные после ручных исправлений.

Разница хранится отдельно в `manual_overrides`. Редактор позволяет изменить:

- название и исходную категорию;
- цену и валюту;
- целевую runtime-категорию и тип компонента;
- количество и клиентские места;
- RAM, CPU, накопитель, мощность и сетевые метрики.

После сохранения запись возвращается в `pending` и требует повторного
подтверждения. Импортированная запись блокируется для редактирования: её следует
изменять уже в данных ТО.

Если при повторной загрузке изменились исходная цена или характеристики,
ручные исправления сохраняются, но подтверждение сбрасывается. Для уже
импортированной позиции показывается предупреждение, что runtime ТО автоматически
не обновлялся.

Экран поддерживает фильтрацию по статусу, множественный выбор, массовое
подтверждение и отклонение. Готовность отображается отдельно:

| Готовность | Значение |
|---|---|
| заблокировано | есть ошибки, импорт запрещён |
| нужна проверка | запись ещё не подтверждена |
| готово к импорту | запись подтверждена, но ещё не перенесена в ТО |
| в ТО / готово к GA | запись импортирована как runtime-альтернатива |
| источник обновлён | после импорта изменились исходные данные; runtime ТО оставлен без изменений |

## Каталог v2

Схема v2 сохраняет старые верхнеуровневые поля и добавляет:

- `source_product_id`;
- `identity` с `brand`, `model`, `mpn`, `gtin`;
- `offer` с ценой, валютой, наличием, регионом, URL и временем наблюдения;
- `field_provenance` для происхождения полей;
- `review` с исходными предупреждениями.

Staging также читает старые v1 JSON-каталоги. Это позволяет постепенно обновлять
источники без массовой миграции сохранённых примеров.

## Ограничения

- XLSX читает первый лист и обычные значения ячеек;
- цена для расчёта должна быть положительной и заданной в RUB;
- CSV поддерживает UTF-8 и Windows-1251;
- staging не запускает Playwright и не обходит антибот-защиту;
- импортированные строки не синхронизируются автоматически с обновлённым
  каталогом, чтобы не менять уже анализируемые альтернативы без решения пользователя.
