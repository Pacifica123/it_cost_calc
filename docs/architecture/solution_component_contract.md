# Контракт `SolutionComponent`

## Назначение документа

Этот документ фиксирует контракт будущей сущности `SolutionComponent` до изменения runtime-кода. Контракт нужен, чтобы последующие кодовые патчи не придумывали поля заново внутри UI и не создавали второй формат альтернатив рядом с уже введёнными `AnalysisScopeProfile`, `CandidateConfiguration`, `CostModel` и `DecisionReport`.

`SolutionComponent` рассматривается как нормализованный компонент ИТ-решения: он создаётся пользователем, демо-сценарием, каталогом, импортом или явной конвертацией старой sandbox-записи, затем проходит проверку и только после этого может участвовать в строгой аналитике.

Главное правило: компонент не заменяет вкладки `ПО`, `ТО`, `OPEX` и `Электроэнергия`. Он нужен для нестандартных, смешанных и составных случаев, где обычной строки в одной вкладке недостаточно для корректного анализа.

## Место компонента в общей цепочке

Компонент должен подключаться к уже завершённой архитектурной цепочке проекта, а не обходить её.

```text
Пользовательский ввод / демо / каталог / импорт / sandbox-конвертация
  → SolutionComponent
  → нормализация, допуски и предупреждения
  → CandidateConfiguration.components
  → CostModel / TCO
  → NPV / AHP / GA / GA + AHP
  → DecisionReport
```

Это означает, что `SolutionComponent` не хранит собственную независимую аналитику. Он отдаёт стоимость, метрики, происхождение и предупреждения существующим прикладным сущностям.

## 1. Обязательные поля идентификации

| Поле | Тип / значения | Кто задаёт | Обязательность | Смысл |
|---|---|---|---:|---|
| `id` | строка | система | да | Стабильный идентификатор компонента. Не должен зависеть от названия, порядка в таблице или текущего UI-состояния. |
| `name` | строка | пользователь / импорт | да | Человекочитаемое название компонента, которое можно показывать в таблицах, альтернативах и отчёте. |
| `description` | строка или пусто | пользователь / импорт | нет | Краткое пояснение назначения компонента. Не используется как источник расчётов. |
| `scope` | `technical`, `software`, `implementation`, `common`, `mixed` | пользователь или нормализация | да | Область анализа. По ней выбирается профиль проверки и интерпретации. |
| `component_type` | строковый тип из предметного словаря | пользователь или нормализация | да | Конкретный тип: сервер, рабочее место, периферия, лицензия, подписка, сервис, комплект и т.д. |
| `origin` | `manual`, `demo`, `catalog`, `imported`, `converted_sandbox` | система / источник данных | да | Происхождение компонента. Нужно для объяснимости и отчёта. |
| `strict_analysis_participation` | `true` / `false` | пользователь + нормализация | да | Разрешение рассматривать компонент как участника строгой аналитики. Само по себе не гарантирует готовность компонента. |

### Правила для `scope`

| Значение | Когда использовать | Пример |
|---|---|---|
| `technical` | Физическое оборудование, периферия, сетевой контур, вычислительная инфраструктура | сервер, рабочее место, МФУ, маршрутизатор |
| `software` | Лицензии, подписки, программные сервисы, пользовательское ПО | офисный пакет, CRM-подписка, антивирус |
| `implementation` | Разовые работы по внедрению, настройке, миграции и пилотированию | настройка сервера, перенос данных, обучение пользователей |
| `common` | Общий расход или компонент, не привязанный только к ПО или ТО | проектное сопровождение, консультация, общий резерв |
| `mixed` | Составной компонент, включающий несколько контуров | комплект «сервер + гипервизор + базовая настройка» |

`scope=mixed` не должен скрывать внутреннюю структуру. Для смешанного компонента нужно либо описывать вложенные части, либо явно указывать, какие финансовые поля и метрики относятся к какому контуру.

### Правила для `component_type`

`component_type` должен быть машинно-читаемым значением, а не свободным текстом. На уровне документационного контракта допускаются следующие группы:

| Группа | Возможные типы | Связанный профиль |
|---|---|---|
| Технические компоненты | `server`, `workstation`, `network_device`, `peripheral`, `storage`, `infrastructure_bundle` | `technical` или `mixed` |
| Программные компоненты | `software_license`, `software_subscription`, `software_service`, `security_software`, `business_application` | `software` или `mixed` |
| Сервисные и внедренческие компоненты | `implementation_service`, `migration_service`, `testing_service`, `support_service`, `training_service` | `implementation`, `common` или `mixed` |
| Общие компоненты | `consulting`, `reserve`, `integration_bundle`, `other` | `common` или `mixed` |

Если импорт или конвертация не могут подобрать точный тип, компонент должен сохраняться как черновик с предупреждением, а не автоматически попадать в строгую аналитику.

## 2. Финансовые поля

Финансовая часть компонента должна быть разложена по смысловым полям, чтобы её можно было передать в `CostModel` и TCO без ручного угадывания.

| Поле | Тип | Кто задаёт | Передача в `CostModel` | Смысл |
|---|---|---|---|---|
| `purchase_cost` | число, руб. | пользователь / каталог / импорт | `purchase_cost` | Покупка, капитальная стоимость, разовый платёж за компонент. |
| `implementation_cost` | число, руб. | пользователь / импорт | `implementation_cost` | Разовые расходы на внедрение, настройку и первичную установку. |
| `migration_cost` | число, руб. | пользователь / импорт | `migration_cost` | Перенос данных, переход на новую систему, миграционные работы. |
| `testing_cost` | число, руб. | пользователь / импорт | `testing_cost` | Тестирование, пилотирование, опытная эксплуатация. |
| `monthly_cost` | число, руб./мес. | пользователь / каталог / импорт | `subscription_cost` или `support_cost` | Подписка, аренда, сопровождение, регулярное обслуживание. |
| `annual_cost` | число, руб./год | пользователь / импорт | преобразуется в годовую регулярную часть TCO | Ежегодные платежи, продления, регламентное обслуживание. |
| `energy_cost` | число, руб./мес. или вычисляемое значение | энергетический расчёт / пользователь | `electricity_cost` | Стоимость электроэнергии, если она задана напрямую или рассчитана по мощности и профилю использования. |
| `cost_assumptions` | список строк | пользователь / система | `warnings` / `metadata` | Пояснения к оценочным суммам, допущениям и неполным данным. |

### Правила интерпретации стоимости

- Нулевое значение и отсутствие значения не равны друг другу. Ноль означает осознанно указанное отсутствие затрат, а пустое значение означает, что данных нет.
- Оценочная сумма допустима, но должна сопровождаться записью в `cost_assumptions`.
- Если стоимость введена одной общей суммой без структуры, компонент может храниться, но `tco_eligible` должен оставаться `false` до уточнения структуры.
- Для `scope=software` энергетические поля не являются обязательными.
- Для `scope=technical` энергетические поля могут быть необязательными на уровне черновика, но должны формировать warning, если компонент участвует в TCO или сравнении технических альтернатив.
- `annual_cost` не должен без пояснения дублироваться с `monthly_cost * 12`. Если оба поля заполнены, нормализация должна либо различать их смысл, либо показать предупреждение.

## 3. Метрики и ограничения

Метрики не должны быть одинаковым набором полей для всех компонентов. Их смысл определяется через `scope`, `component_type` и будущую связку с `AnalysisScopeProfile`.

### Технические компоненты

| Поле / метрика | Смысл | Использование |
|---|---|---|
| `power_watts` | Потребляемая мощность | Энергетический расчёт, TCO, предупреждения по техническому профилю |
| `client_seats` | Количество рабочих мест или пользователей, которых обслуживает компонент | GA/AHP, проверка ёмкости технической альтернативы |
| `performance_score` | Условная оценка производительности | Многокритериальное сравнение |
| `reliability_score` | Условная оценка надёжности | AHP/GA и отчётное объяснение |
| `scalability_score` | Запас развития | Сравнение альтернатив и риск роста нагрузки |
| `maintainability_score` | Удобство сопровождения | Пояснение эксплуатационных рисков |
| `usage_profile` | Режим использования: часы, дни, коэффициент нагрузки | Расчёт электроэнергии |

### Программные компоненты

| Поле / метрика | Смысл | Использование |
|---|---|---|
| `license_quantity` | Количество лицензий, пользователей или рабочих мест | Проверка ПО-профиля и сравнение альтернатив |
| `functional_coverage_score` | Условное покрытие требований | AHP/GA и отчётное объяснение |
| `compatibility_score` | Совместимость с текущей инфраструктурой | Аналитика и риски внедрения |
| `support_score` | Наличие поддержки, обновлений, сопровождения | AHP/GA, DecisionReport |
| `implementation_complexity_score` | Сложность внедрения | Сравнение, предупреждения, NPV/TCO-допущения |
| `security_score` | Условная оценка защищённости | Профильные критерии и риски |

### Сервисные и внедренческие компоненты

| Поле / метрика | Смысл | Использование |
|---|---|---|
| `labor_hours` | Трудозатраты | Объяснение внедренческой стоимости |
| `duration_days` | Длительность выполнения работ | Планирование, риски, отчёт |
| `periodicity` | Разовость или периодичность услуги | TCO и эксплуатационные расходы |
| `vendor_dependency_score` | Зависимость от внешнего поставщика | Риски и сравнение альтернатив |
| `deliverable` | Результат работы: настройка, миграция, обучение, отчёт | Пояснение в DecisionReport |

### Смешанные компоненты

Смешанный компонент должен иметь один из двух вариантов структуры:

| Вариант | Когда подходит | Требование |
|---|---|---|
| Вложенные части | Компонент реально состоит из отдельных технических, программных и сервисных элементов | Каждая часть получает свой `scope`, `component_type`, стоимость и метрики. |
| Явное распределение полей | Компонент агрегирован, но его стоимость и метрики можно разделить по контурам | В `cost_assumptions` и metadata нужно указать, какая часть относится к ТО, ПО, внедрению и сопровождению. |

Если смешанный компонент невозможно разделить, он может быть сохранён как черновик или вспомогательная запись, но не должен автоматически попадать в строгую аналитику.

## 4. Признаки допуска в аналитику

Признаки допуска нужны для защиты от методической ошибки: неполный, но красиво оформленный компонент не должен автоматически влиять на результат выбора.

| Поле | Кто вычисляет | Условие `true` | Что означает `false` |
|---|---|---|---|
| `candidate_eligible` | нормализация | Есть `id`, `name`, `scope`, `component_type`, компонент не исключён и имеет достаточные метрики для альтернативы | Компонент нельзя включать в `CandidateConfiguration` как полноценный элемент |
| `tco_eligible` | нормализация / TCO-сервис | Стоимость структурирована по разовым и регулярным полям | Компонент не должен менять TCO без ручного уточнения |
| `energy_eligible` | энергетический сервис / нормализация | Есть мощность и профиль использования или корректно задан `energy_cost` | Энергетическая часть не считается или показывается как пропуск |
| `npv_eligible` | TCO/NPV-мост | Финансовая часть может быть переведена в инвестиции и регулярные потоки | Компонент не должен автоматически попадать в NPV без пояснения |
| `analysis_ready` | профильная проверка | Компонент прошёл обязательные проверки своего `scope` и `component_type` | Компонент хранится, но не считается полностью готовым для строгого сценария |
| `validation_warnings` | нормализация и прикладные сервисы | Список может быть пустым или содержать некритичные предупреждения | Если есть критичные предупреждения, они должны блокировать соответствующий допуск |

`strict_analysis_participation=true` означает намерение пользователя включить компонент в строгую аналитику. Итоговое право участия всё равно определяется вычисляемыми признаками допуска.

### Рекомендуемые состояния готовности

| Состояние | Условие | Поведение |
|---|---|---|
| `draft` | Не хватает обязательных полей или классификации | Хранить можно, в строгую аналитику не включать. |
| `normalized` | Есть базовая классификация и понятная структура данных | Можно показывать как нормализованный компонент, но не обязательно включать во все расчёты. |
| `candidate_eligible` | Достаточно данных для включения в альтернативу | Можно передавать в `CandidateConfiguration.components`. |
| `tco_eligible` | Стоимость структурирована для TCO | Можно передавать финансовую часть в `CostModel`. |
| `analysis_ready` | Пройдены профильные проверки | Можно использовать в целевом аналитическом сценарии. |
| `excluded` | Пользователь или нормализация исключили компонент | Хранить и показывать можно, учитывать в строгих результатах нельзя. |

Эти состояния могут быть отдельным enum в будущем коде, но на уровне контракта достаточно поддержать их через поля допуска и предупреждения.

## 5. Связь с существующими сущностями

### `SolutionComponent → CandidateConfiguration.components`

Валидный компонент становится элементом `components` внутри `CandidateConfiguration`. При этом:

- `id`, `name`, `scope`, `component_type` и `origin` должны сохраняться без потери;
- исходные пользовательские поля можно передавать в metadata, если они не нужны алгоритмам напрямую;
- компоненты с `candidate_eligible=false` не должны попадать в строгий набор альтернатив, но могут показываться в отчёте как исключённые.

### `SolutionComponent.costs → CostModel`

Финансовые поля компонента должны преобразовываться в `CostModel` по смыслу:

| Поле компонента | Поле `CostModel` / TCO |
|---|---|
| `purchase_cost` | `purchase_cost`, стартовые CAPEX |
| `implementation_cost` | `implementation_cost`, разовые внедренческие расходы |
| `migration_cost` | `migration_cost`, разовые миграционные расходы |
| `testing_cost` | `testing_cost`, разовые расходы на проверку и пилот |
| `monthly_cost` | `subscription_cost` или `support_cost`, регулярные ежемесячные расходы |
| `annual_cost` | годовая регулярная часть TCO, с явным пояснением пересчёта |
| `energy_cost` | `electricity_cost`, если значение задано или рассчитано корректно |

Если поле невозможно однозначно сопоставить, оно должно перейти в metadata и warning, а не молча попасть в расчёты.

### `SolutionComponent.metrics → AnalysisScopeProfile`

`AnalysisScopeProfile` должен определять, какие метрики обязательны и как они интерпретируются для выбранного `scope`.

| Профиль | Обязательная логика |
|---|---|
| `technical` | Не считать программные лицензии рабочими местами; учитывать ёмкость, мощность и технические ограничения только там, где они применимы. |
| `software` | Не требовать физическое оборудование и мощность; считать лицензии, пользователей, функциональное покрытие и поддержку. |
| `implementation` | Проверять длительность, трудозатраты, миграцию, тестирование и зависимость от поставщика. |
| `common` | Не подменять общий расход техническим или программным критерием без явной классификации. |
| `mixed` | Требовать разделение вложенных частей или понятное распределение стоимости и метрик. |

### `SolutionComponent.origin/warnings → DecisionReport`

Итоговый отчёт должен показывать не только итоговые суммы, но и объяснимость компонента:

- происхождение компонента;
- был ли он создан вручную, импортирован, взят из демо, каталога или конвертирован из sandbox;
- какие допущения использовались в стоимости;
- какие warnings остались после нормализации;
- почему компонент был включён или исключён из строгой аналитики.

### `SolutionComponent.power/usage_profile → ElectricityCostService`

Для технических компонентов энергетическая часть должна строиться по данным мощности и профиля использования, если они есть. Минимальный набор для расчёта:

| Поле | Смысл |
|---|---|
| `power_watts` | мощность одного компонента |
| `quantity` или `client_seats` | количество единиц, если применимо |
| `hours_per_day` | часов работы в день |
| `working_days` | рабочих дней в месяце или году |
| `cost_per_kwh` | тариф за кВт·ч |

Если компонент относится к ПО или сервису без технической нагрузки, `energy_eligible=false` не является ошибкой. Для технического компонента отсутствие энергетических данных должно становиться предупреждением, но не всегда блокировать хранение черновика.

## 6. Связь со старой `ИТ-песочницей`

Старые sandbox-записи остаются безопасными вспомогательными строками до явной конвертации. Автоматически превращать их в `SolutionComponent` нельзя, потому что в них может не быть `scope`, `component_type`, структурированной стоимости и метрик.

| Действие | Правило |
|---|---|
| Чтение старых записей | Поддерживается без изменения смысла. |
| Конвертация | Только явным действием пользователя или отдельным импортным сценарием. |
| Перенос названия | Старое название переносится в `name`; исходная категория сохраняется в metadata. |
| Перенос суммы | Сумма не должна автоматически считаться CAPEX, OPEX или TCO без выбора пользователем. |
| Происхождение | Для таких компонентов ставится `origin=converted_sandbox`. |
| Неполные данные | Компонент получает состояние `draft` и `validation_warnings`. |
| Участие в аналитике | Запрещено до прохождения нормализации и профильных проверок. |

## 7. Что является пользовательским вводом, вычислением и предупреждением

| Группа | Поля | Источник |
|---|---|---|
| Пользовательский ввод | `name`, `description`, `scope`, `component_type`, финансовые поля, метрики, `strict_analysis_participation`, `cost_assumptions` | Форма, импорт, каталог, демо или ручная конвертация |
| Системные поля | `id`, `origin`, `source_category`, `created_at`, `updated_at`, служебная metadata | Система или адаптер источника данных |
| Вычисляемые признаки | `candidate_eligible`, `tco_eligible`, `energy_eligible`, `npv_eligible`, `analysis_ready`, состояние готовности | Нормализация, профили и прикладные сервисы |
| Предупреждения | `validation_warnings`, финансовые warnings, warnings профиля, warnings отчёта | Нормализация, TCO/NPV, Energy, DecisionReport |

Пользователь может запросить участие компонента в строгой аналитике, но не должен вручную выставлять итоговые признаки допуска как произвольные галочки. Иначе UI сможет обойти предметные проверки.

## 8. Минимальный критерий готовности C1

Этап C1 считается завершённым, если по этому документу понятно:

- какие поля обязаны быть у `SolutionComponent`;
- какие поля являются пользовательским вводом;
- какие поля вычисляются нормализацией и прикладными сервисами;
- какие данные передаются в `CandidateConfiguration`, `CostModel`, `AnalysisScopeProfile`, `ElectricityCostService` и `DecisionReport`;
- почему старые sandbox-записи не попадают в строгую аналитику автоматически;
- почему редактор компонентов не заменяет быстрые вкладки `ПО`, `ТО`, `OPEX` и `Электроэнергия`.

Следующий этап после C1 может добавлять доменную модель и сервис нормализации, не меняя уже зафиксированный смысл полей.

## 9. Статус реализации после C2

Кодовый этап C2 закрепляет этот контракт в domain/application-слое без изменения пользовательского интерфейса.

Реализованные точки привязки:

- `domain.SolutionComponent` хранит входной компонент, финансовые поля, метрики, `metadata`, предупреждения и признаки допуска;
- `SolutionComponentNormalizationService` нормализует компонент без изменения исходного payload, разделяет `validation_warnings` и `blocking_errors`;
- `CandidateConfigurationService.from_solution_components` подключает готовые компоненты к существующему формату альтернатив;
- сервис нормализации отдаёт `CostModel`, energy input и snapshot для `DecisionReport`;
- записи с `_legacy_sandbox=true` или префиксом `legacy_infrastructure:*` не включаются в строгую аналитику автоматически.

После C3 runtime-хранилище и fixture-данные реализованы отдельной секцией `entities.solution_components` и описаны в `solution_component_runtime_storage.md`. UI-редактор остаётся задачей следующего этапа roadmap.
