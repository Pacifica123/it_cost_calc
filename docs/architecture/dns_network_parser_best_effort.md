# Best-effort парсер сетевых характеристик DNS-Shop

## Назначение

Этот документ фиксирует реализацию Патча 9 из roadmap переработки оптимизационных блоков. Цель патча — связать уже добавленные технические метрики ТО с каталогом оборудования без превращения парсинга в обязательную часть runtime GUI.

Реализация остаётся минимальной и безопасной: парсер работает только как вспомогательный инструмент `tools/catalog_parser/`, разбирает сохранённые example-снимки и сохраняет результат в нормализованном каталоге. Если формат названия неизвестен, инструмент возвращает предупреждения, а не падает.

## Что добавлено

Добавлен модуль:

- `tools/catalog_parser/dns_network_metrics.py`.

Он извлекает из названия сетевого устройства:

| Поле | Смысл |
|---|---|
| `lan_ports` | число LAN-портов из фрагментов вида `1 LAN` |
| `lan_speed_mbps` | скорость LAN, best-effort по найденным значениям `Мбит/с` |
| `wifi_standards` | список стандартов `802.11n/ac/ax/be` |
| `wifi_generation_rank` | условный ранг поколения Wi-Fi: 4/5/6/7 |
| `wifi_total_mbps` | заявленная суммарная скорость из фрагмента `Wi-Fi N Мбит/с` |
| `ipv6_support` | наличие `IPv6` в названии |
| `parse_warnings` | предупреждения о неполных/неоднозначных данных |
| `confidence` | грубая уверенность распознавания `0..1` |

## Пример

Исходное название:

```text
Wi-Fi роутер TP-Link Archer Air R5 [1 LAN, 1000 Мбит/с, 100 Мбит/с, 4 (802.11n), 5 (802.11ac), 6 (802.11ax), Wi-Fi 2976 Мбит/с, IPv6]
```

Из него извлекается:

```json
{
  "lan_ports": 1,
  "lan_speed_mbps": 1000,
  "wifi_standards": ["ac", "ax", "n"],
  "wifi_generation_rank": 6,
  "wifi_total_mbps": 2976,
  "ipv6_support": true
}
```

Так как в названии есть несколько не-Wi-Fi скоростей (`1000 Мбит/с` и `100 Мбит/с`), parser добавляет предупреждение и берёт максимум как `lan_speed_mbps`. Это лучше, чем молча выбрать неверное поле.

## Приоритет ручных данных

Распознанные поля не должны перетирать вручную заполненные характеристики из `specs`.

Правило merge:

1. сначала берутся ручные `specs`;
2. затем добавляются распознанные поля, которых ещё нет;
3. если ручное поле уже есть, оно сохраняется;
4. в `parse_warnings` добавляется пояснение, что ручное значение имеет приоритет.

Это важно для будущего сценария, где пользователь или нормализатор каталога сможет исправить ошибку парсинга.

## Граница ответственности

Парсер не является обязательным источником истины для оптимизации. Он только подготавливает признаки, которые затем могут попасть в каталог и runtime-нормализацию.

Не сделано намеренно:

- не добавлен HTML-скрейпинг DNS-Shop в GUI;
- не вводится сложная модель доверия;
- не выполняется автоматическое подтверждение низкоуверенных характеристик;
- не делается полный парсер всех товарных категорий.

## Связь с Патчем 8

Патч 8 добавил runtime-поля `lan_ports`, `lan_speed_mbps`, `wifi_total_mbps`, `ipv6_support` и предупреждения по неполным метрикам. Патч 9 добавляет best-effort путь наполнения этих полей из каталожных названий.

Итоговая схема остаётся простой:

```text
DNS/example snapshot -> tools/catalog_parser -> normalized catalog -> ручная проверка/импорт -> ПО/ТО аналитика
```
