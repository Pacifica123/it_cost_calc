# Roadmap devctl-патчей для переработки аналитических блоков и критериев

## Назначение документа

Документ собирает предложенную очередность будущих devctl-патчей после текущего документационного планирования. Он нужен, чтобы не смешивать в одном изменении интерфейс, методику критериев, парсинг характеристик, экспорт и гибридную оценку.

Главное правило последовательности: сначала исправить видимую архитектуру блоков, затем упростить критерии, затем аккуратно расширять метрики.

---

## Общий целевой результат

После выполнения roadmap пользователь должен видеть внутри каждой рабочей области ПО/ТО четыре независимых аналитических блока:

| Блок | Роль |
|---|---|
| GA | поиск допустимых кандидатных конфигураций |
| AHP | экспертное ранжирование общего пула альтернатив |
| Pareto | поиск компромиссных недоминируемых вариантов |
| Гибридная оценка | сводная рекомендация и проверка согласованности методов |

Критерии и ограничения должны браться из общего профиля области анализа. Покрытие категорий и клиентские места должны перейти из мягких весов в жёсткие фильтры/конструктор ограничений. Стоимость должна быть объяснена как бюджетный фильтр и, при необходимости, аккуратный tie-breaker, а не безусловный основной критерий.

## Патч 1. Документационная фиксация методической реформы

Статус текущего патча.

Что входит:

- план перераспределения GA/AHP/Pareto/Hybrid;
- план отдельной гибридной панели;
- план унификации критериев и ограничений;
- план конкретных метрик оборудования и связи с парсингом;
- общая очередность будущих devctl-патчей.

Что не входит:

- изменение Python-кода;
- изменение демо-данных;
- изменение расчётных формул.

Проверки:

- markdown-файлы читаются как UTF-8;
- code fences сбалансированы;
- patch.zip содержит только документационные файлы.

## Патч 5. Конструктор категорий и отказ от coverage как soft-критерия

Статус: реализовано базовым MVP. `category_policy` уже работает в UI как цикл `пусто → + → − → пусто`, а `category_coverage` удалён из soft-критериев GA.

Цель: заменить `category_coverage` на понятный hard-filter `+ / - / без разницы`.

Изменить:

- `AnalysisScopeProfileService`;
- UI ограничений в GA и, возможно, AHP/Pareto;
- demo-профили;
- документацию `analysis_scope_profiles.md`.

Сделать:

- добавить `category_policy`;
- сохранить совместимость чекбоксов через преобразование в `required`;
- поддержать `excluded` категории;
- убрать вес `Покрытие категорий` из базовой формы;
- убрать `category_coverage` из мягких критериев профиля.

Не делать:

- не удалять старые поля из JSON резко;
- не ломать сохранённые демо-данные;
- не превращать конструктор категорий в сложный rule engine.

Проверки:

- обязательные категории работают как раньше;
- исключённая категория не попадает в кандидата;
- пустая категория не влияет на фильтр;
- старые тесты GA required behavior проходят или обновлены осмысленно.

## Патч 6. Минимум мест/лицензий как фильтр, а не вес

Цель: убрать спорный вес клиентских мест из базовой оптимизации.

Изменить:

- профиль ТО и ПО;
- UI весов;
- constraints builder;
- объяснения результата.

Сделать:

- `client_capacity >= target` для ТО;
- `software_license_quantity >= target` для ПО;
- убрать вес `Клиентские места` из базового режима;
- в отчёте показывать, что минимум выполнен или не выполнен;
- оставить будущий расширенный критерий `запас масштабируемости` только как идею.

Не делать:

- не пытаться одновременно вводить сложную capacity planning модель;
- не считать периферию рабочим местом;
- не смешивать лицензии ПО и клиентские ПК.

Проверки:

- МФУ не закрывает клиентское место;
- ПК/рабочая станция закрывает `client_seats`;
- ПО-лицензии считаются отдельно;
- варианты ниже минимума отфильтровываются.

## Патч 7. Роль стоимости: бюджет и tie-breaker

Цель: убрать методическое дублирование стоимости как одновременно бюджета и главного критерия.

Изменить:

- профиль критериев;
- UI весов;
- GA/AHP/Pareto объяснения;
- DecisionReport.

Сделать:

- оставить бюджет как hard constraint;
- убрать вес стоимости из базового режима или перевести его в опцию `предпочитать дешевле при близком качестве`;
- в отчёте отдельно показывать экономию бюджета;
- описать, что экономия не должна уничтожать качество решения.

Не делать:

- не удалять стоимость из отчётов;
- не запрещать расширенный режим со стоимостью как критерием;
- не скрывать от пользователя, что более дешёвое решение может быть хуже по характеристикам.

Проверки:

- варианты выше бюджета не проходят;
- допустимые варианты сравниваются по качественным критериям;
- при tie-breaker более дешёвый вариант выигрывает только при близкой основной оценке.

## Рекомендуемая граница MVP

Минимально достаточный результат для защиты после первых реализационных патчей:

- внутри ПО/ТО есть четыре понятных аналитических панели;
- GA не показывает AHP как собственную вкладку;
- AHP ранжирует общий пул альтернатив;
- гибридная панель отдельно показывает согласованность;
- coverage и клиентские места больше не являются спорными весами в базовом режиме;
- стоимость объяснена как бюджетный фильтр и опциональная экономия;
- отчёт честно показывает, какие методы рассчитаны, а какие нет.

Этого достаточно, чтобы система выглядела методически связной, даже если расширенные метрики оборудования и парсер будут развиваться следующими отдельными патчами.
