# DecisionReport: единый итоговый отчёт выбора

## Назначение

`DecisionReport` завершает цепочку концептуальной связанности проекта: отдельные расчёты CAPEX, OPEX, электроэнергии, GA, AHP, Pareto, Hybrid и NPV собираются в один воспроизводимый артефакт выбора ИТ-решения.

После рефакторинга аналитических блоков отчёт фиксирует не смешанный сценарий `GA + AHP`, а новую модель ответственности:

```text
GA формирует общий пул альтернатив → AHP ранжирует этот пул → Pareto показывает компромиссы → Hybrid собирает сводную рекомендацию
```

До этого этапа приложение уже умело:

- нормализовать сущности через `scope` и `component_type`;
- применять профили анализа ПО/ТО;
- приводить альтернативы к `CandidateConfiguration`;
- рассчитывать TCO и передавать финансовую базу в NPV;
- хранить результаты методов внутри рабочих областей `ПО` и `ТО`.

`DecisionReport` не запускает методы заново. Он фиксирует состояние данных на момент экспорта, показывает источник общего пула и объясняет итоговый выбор через отдельные результаты методов.

## Структура отчёта

| Блок | Содержание | Назначение |
|---|---|---|
| `project` | исходный проект выбора, цель, дата формирования | показывает, для какого решения сформирован отчёт |
| `components` | текущие runtime-компоненты из CAPEX/OPEX/ПО/ТО | связывает вывод с исходными пользовательскими данными |
| `cost_model` | итоги CAPEX/OPEX/электроэнергии и блок `tco` | показывает стоимость владения и финансовую базу |
| `candidate_configurations` | общий пул альтернатив с `candidate_pool_source` | даёт единый объект сравнения для GA, AHP, Pareto, Hybrid и отчёта |
| `solution_component_report` | отдельное объяснение компонентов редактора | показывает валидные компоненты, черновики, warnings и связь с альтернативами |
| `catalog_data_quality` | provenance и полнота каталожных компонентов | показывает заполненные/отсутствующие метрики, parser confidence, ручные исправления и warnings |
| `analysis_results` | отдельные результаты GA, AHP, Pareto, Hybrid и legacy-совместимости | хранит машинно-воспроизводимые результаты методов |
| `npv_interpretation` | NPV, денежные потоки, предупреждения | объясняет финансовую часть выбора |
| `winner_explanation` | лидеры методов, согласованность, рекомендуемый вариант | превращает набор результатов в итоговое объяснение |
| `risks` и `warnings` | ограничения, неполные данные, методические предупреждения | не скрывает слабые места вывода |
| `metadata` | версия схемы, тип отчёта, источник экспорта | поддерживает машинный экспорт и будущую совместимость |

## Машинный и пользовательский экспорт

Этап разделяет два уровня представления.

### JSON

JSON считается полным машинным артефактом. Он сохраняет:

- исходные компоненты;
- полный пул `CandidateConfiguration`;
- источник пула и метод его формирования;
- отдельный блок `solution_components` для строк редактора;
- отдельный блок `catalog_data_quality` для происхождения и полноты импортированных карточек;
- признаки допуска, warnings, origin и связь компонентов с альтернативами;
- отдельные результаты GA, AHP, Pareto и Hybrid;
- legacy-блок `genetic_ahp`, если он был передан старым сценарием;
- TCO и NPV;
- предупреждения;
- metadata схемы.

Файл предназначен для повторного анализа, диагностики и будущей автоматической генерации дипломных материалов.

### Markdown

Markdown является пользовательским и защитным представлением. Он не пытается показать все debug-поля, а отвечает на вопросы:

- какое решение рассматривается;
- какой вариант рекомендован;
- откуда взялся общий пул альтернатив;
- какие методы рассчитаны отдельно;
- почему методы пришли к такому выводу;
- какая стоимость владения получилась;
- какие `SolutionComponent` участвовали в выборе;
- какие черновики или исключённые компоненты не влияли на выбор и почему;
- какие технические метрики каталожных компонентов заполнены и какие отсутствуют;
- что показывает NPV;
- какие ограничения и риски остаются.

### CSV

CSV кандидатов остаётся табличным срезом пула альтернатив. Он удобен для просмотра в таблицах и не заменяет JSON-отчёт. В нём фиксируются `candidate_pool_source`, `candidate_pool_method`, агрегированные `technical_metrics` и `metric_warnings`, чтобы было видно происхождение альтернативы и ограничения исходных характеристик.

## Качество данных каталога

Для строк с `origin = catalog` отчёт формирует отдельный срез качества. Он переносит `source`, `source_product_id`, время наблюдения цены, `field_provenance`, `parse_source`, `confidence`, список ручных исправлений и технические метрики. Полнота проверяется по `component_type`: рабочие станции и серверы требуют RAM/CPU/storage/power, сетевые устройства — power и сетевые признаки.

Отсутствующие поля не превращаются в скрытые нули. Они перечисляются в `missing_metrics`, а сводное предупреждение попадает в `warnings` и `risks` DecisionReport. JSON сохраняет весь блок, Markdown показывает компактную таблицу, CSV альтернатив — агрегированные метрики и предупреждения кандидата.

## Компоненты редактора SolutionComponent

`DecisionReport` отдельно объясняет строки редактора компонентов. Валидные компоненты показываются с `scope`, `component_type`, происхождением, статусом допуска, финансовым вкладом, использованными метриками и связью с `CandidateConfiguration`. Черновики и исключённые компоненты выводятся отдельной таблицей с причинами, поэтому неполные записи не выглядят как скрытые расчётные данные.

Это особенно важно для защиты: можно показать, что advanced-редактор не создаёт параллельный формат альтернатив, а отдаёт данные в ту же цепочку `component → CandidateConfiguration → TCO → GA/AHP/Pareto/Hybrid → DecisionReport`.

## Объяснение победителя

`DecisionReportService` собирает лидеров из доступных методов:

- GA как источника найденных допустимых кандидатов;
- AHP как самостоятельного экспертного ранжирования общего пула;
- Pareto/критериального анализа как проверки компромиссов;
- Hybrid как сводной витрины согласованности;
- legacy `genetic_ahp`, если отчёт строится по старому машинному payload.

Если лидеры совпали, отчёт помечает вывод как согласованный. Если лидеры различаются, отчёт не скрывает конфликт: рекомендуемый вариант выбирается по приоритету Hybrid/AHP-оценки, а причины расхождения остаются в отчёте.

## Защитный сценарий

Для дипломной защиты поток можно объяснить так:

1. Пользователь вводит компоненты ИТ-решения.
2. Прикладной слой нормализует их по области анализа и типу компонента.
3. GA или demo-сценарий формирует общий пул альтернатив `CandidateConfiguration` для `ПО` и `ТО`.
4. TCO связывает CAPEX, OPEX и электроэнергию.
5. AHP ранжирует тот же пул альтернатив.
6. Pareto показывает компромиссные и недоминируемые варианты.
7. Hybrid собирает последние результаты GA, AHP и Pareto в аккуратную сводку.
8. NPV использует рассчитанную финансовую базу.
9. `DecisionReport` показывает валидные компоненты редактора, исключённые черновики, источник пула, результаты методов, победителя, риски и ограничения.

Главная мысль: проект больше не является набором отдельных калькуляторов и больше не смешивает AHP внутри GA. Он работает как связанная система поддержки выбора ИТ-решения по формуле `(GA + AHP + Pareto + Hybrid) × ПО/ТО`.

## Роль C9-демонстрации

Финальный C9-сценарий использует отдельную fixture `data/fixtures/solution_component_c9_demo_dataset.json`, чтобы показать именно редактор компонентов. В отчёте должны быть видны:

- mixed-комплект «сервер + гипервизор + базовая настройка» как валидный компонент с TCO-вкладом;
- программная подписка как ПО-компонент без энергетического участия;
- МФУ как техническая периферия с `client_seats = 0`;
- черновик как исключённая запись с причинами.

Это делает `DecisionReport` не только экспортом расчёта, но и защитным материалом: он показывает, какие данные участвовали в выборе, какие были исключены и почему.

## Границы отчёта

`DecisionReport` не заменяет старый CSV-экспорт затрат и не требует полного прохождения всех аналитических вкладок перед экспортом. Если часть методов ещё не запускалась, отчёт формируется как частичный и явно добавляет предупреждения.

Такой подход безопасен для текущего GUI: пользователь может получить отчёт уже после ввода затрат, а по мере запуска GA, AHP, Pareto, Hybrid и NPV отчёт становится богаче.
