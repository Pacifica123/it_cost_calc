# Итоговая сводка выполнения roadmap концептуальной связанности

## Назначение документа

Этот документ фиксирует итоговую сверку после завершения Этапа 9 из `conceptual_cohesion_roadmap.md`. Он нужен как короткий вход в текущее состояние проекта: что было сделано, какая цепочка данных считается основной и какие идеи оставлены на будущее.

## Итоговый статус

Roadmap доведён до базового целевого состояния: проект больше не описывается как набор независимых вкладок. Основная модель теперь читается как связанный сценарий поддержки выбора ИТ-решения:

```text
runtime-записи → scope/component_type → AnalysisScopeProfile → CandidateConfiguration → TCO/NPV → DecisionReport
```

Это не отменяет старые пользовательские данные и legacy-слои. Они сохранены через необязательные поля, переходную нормализацию, совместимые адаптеры и отдельную роль `ИТ-песочницы`.

## Что закрыто сверх нумерованных этапов

### Общие правила roadmap

Общие правила можно считать закрытыми в текущей базовой реализации:

- обратная совместимость поддержана через необязательные поля и legacy-нормализацию;
- предметная логика ПО/ТО вынесена из UI в application/domain;
- изменения оформлялись малыми devctl-патчами с отдельным смыслом;
- новые сущности не создавались как изолированные форматы, а подключались к общей цепочке.

### Контрольные вопросы

Контрольные вопросы перед патчами также закрыты итоговой сверкой. Для каждого крупного изменения теперь можно ответить:

- какая сущность стала связнее;
- какие данные берутся из runtime-модели;
- какие данные возвращаются другим модулям;
- почему не возник новый ad-hoc-формат альтернатив;
- почему UI не стал владельцем предметной логики;
- как сохраняются старые пользовательские данные;
- как объяснить изменение на защите;
- какая проверка подтверждает отсутствие грубой регрессии.

## Текущая роль основных сущностей

| Сущность | Роль в проекте |
|---|---|
| `scope` | Показывает область анализа: ТО, ПО, внедрение или общий контур. |
| `component_type` | Уточняет тип компонента: сервер, рабочее место, периферия, лицензия, подписка, сервис и т.д. |
| `AnalysisScopeProfile` | Хранит критерии, ограничения и объяснения для ПО/ТО без переноса этой логики в UI. |
| `CandidateConfiguration` | Общий формат альтернатив для AHP, GA, GA + AHP, анализа важности и отчёта. |
| `CostModel` / TCO | Собирает стоимость владения из CAPEX, OPEX, внедрения, сопровождения и электроэнергии. |
| `DecisionReport` | Финальный артефакт выбора, связывающий компоненты, стоимость, альтернативы, методы анализа, NPV, риски и предупреждения. |

## Что не считается закрытым кодом

Вариант C из Этапа 7 — универсальный редактор `SolutionComponent` — не реализован как runtime-функция. Он описан как будущая идея в `future_solution_component_editor_variant_c.md`. Это сознательная граница: текущий roadmap завершает базовую связанность, а не превращает старую вкладку в новый редактор компонентов.

## Академическая формулировка результата

После выполнения roadmap приложение можно описывать так:

> Проект представляет собой прототип системы поддержки выбора ИТ-решения, в котором исходные компоненты инфраструктуры и программного обеспечения нормализуются по области анализа и типу, преобразуются в сопоставимые кандидатные конфигурации, получают стоимость владения, оцениваются несколькими аналитическими методами и сводятся в единый итоговый отчёт выбора.

Такая формулировка показывает, что GA, AHP, анализ важности критериев и NPV не являются разрозненными вкладками. Они решают разные части одной задачи: генерация альтернатив, многокритериальное ранжирование, объяснение влияния критериев, финансовая оценка и итоговая интерпретация.

## Дальнейшее развитие после roadmap

Ближайшие направления развития лучше вести уже как новые roadmap-документы, а не как продолжение текущего списка:

1. универсальный редактор `SolutionComponent` по варианту C;
2. более строгий контракт импортируемых каталогов и пользовательских справочников;
3. расширение `DecisionReport` до полноценного дипломного отчёта с шаблонами разделов;
4. визуальная панель сравнения методов, показывающая согласованность GA, AHP, гибридной оценки и NPV;
5. постепенное удаление старого совместимого namespace после подтверждения, что внешние импорты больше им не пользуются.
