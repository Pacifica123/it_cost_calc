# Единый формат кандидатных альтернатив

## Назначение

`CandidateConfiguration` — общий формат альтернативы, которую можно сравнивать в GA, AHP, Pareto, Hybrid и итоговом `DecisionReport`.

После рефакторинга этот формат стал особенно важным: GA больше не содержит внутри себя AHP-сценарий, а формирует общий пул кандидатов. AHP ранжирует этот пул, Pareto проверяет компромиссы, Hybrid собирает сводку согласованности, а `DecisionReport` фиксирует всё в одном отчёте.

Раньше в проекте одновременно существовали несколько близких, но разных представлений:

- AHP-конфигурация с полями `id`, `devices`, `meta`;
- GA `candidate_solution` с `selected_items`, `totals`, `score` и диагностикой;
- альтернатива анализа важности критериев с `id`, `name` и таблицей оценок;
- демо-конфигурации, собранные из runtime-записей CAPEX/OPEX.

Общий контракт и адаптеры позволяют старым вкладкам и экспортам продолжать работать, но новые сценарии опираются на один тип альтернативы.

## Контракт `CandidateConfiguration`

| Поле | Смысл |
|---|---|
| `id` | стабильный идентификатор альтернативы |
| `name` | человекочитаемое название |
| `scope` | область анализа: `technical`, `software`, `implementation` или `common` |
| `components` | исходные компоненты альтернативы без потери исходных полей |
| `totals` | агрегированные затраты, TCO и итоговые числовые суммы |
| `metrics` | предметные метрики: ОЗУ, CPU, накопитель, мощность, лицензии, функциональность, сопровождение |
| `source` | источник альтернативы: `manual`, `demo`, `ga`, `catalog`, `imported` |
| `metadata` | служебная информация: источник пула, rank, parse warnings, критерии и ограничения |

Ключевое разделение:

- `components` отвечают на вопрос **из чего состоит альтернатива**;
- `totals` отвечают на вопрос **сколько стоит альтернатива**;
- `metrics` отвечают на вопрос **какими свойствами обладает альтернатива**;
- `metadata` объясняет **откуда она появилась и как была получена**.

## Допустимые источники

| Источник | Когда используется |
|---|---|
| `manual` | пользователь вручную собрал текущий набор runtime-компонентов |
| `demo` | альтернатива создана демонстрационным сценарием |
| `ga` | альтернатива найдена генетическим алгоритмом |
| `catalog` | альтернатива построена из внешнего каталога или набора каталожных позиций |
| `imported` | альтернатива импортирована из внешнего файла |

## Адаптеры текущего переходного периода

### AHP → `CandidateConfiguration`

Демо-AHP по-прежнему строит legacy-конфигурации с `devices` и `meta`, но рядом формируется `candidate_configurations`. Это позволяет UI продолжать открывать прежний сценарий, а отчёту уже получать общий формат.

### GA → `CandidateConfiguration`

`GeneticOptimizationService` оставляет `candidate_solutions` для совместимости, но дополнительно формирует `candidate_configurations`. В них:

- `components` берутся из `selected_items`;
- `totals` берутся из итогов GA-кандидата;
- `metrics` сохраняют `ga_score`, raw/directed/normalized scores по профильным критериям;
- `metadata` хранит rank, mask, группировку по категориям, критерии и ограничения.

### GA → общий пул → AHP/Pareto/Hybrid

Основной UI-сценарий после рефакторинга выглядит так:

```text
GA top-N → scoped candidate pool → AHP ranking / Pareto case-data → Hybrid assessment → DecisionReport
```

При передаче top-N в общий пул рабочей области каждому кандидату добавляется metadata источника. Это нужно, чтобы отчёт и защита могли показать: AHP и Hybrid работают не с отдельным скрытым набором, а с теми же альтернативами.

### Legacy GA+AHP

`GeneticAhpRankingService` сохранён как совместимый машинный слой для старых payload и тестов. Он не является основной пользовательской формулой и не должен снова превращать GA-панель в смешанный блок.

### Анализ важности критериев / Pareto → общий пул

`DecisionDemoDataService` добавляет `candidate_configurations` в `criteria_case`, а `run_importance_pipeline` возвращает этот пул в отчёте. Сам алгоритм отношений важности не меняется: он по-прежнему работает с критериями, оценками и отношениями, но теперь результат можно связать с теми же альтернативами, что AHP и GA.

### Runtime CAPEX/OPEX → компоненты конфигурации

`CandidateConfigurationService.from_runtime_entities()` собирает текущие runtime-записи в компоненты одной конфигурации. Это переходный адаптер для связи существующих CAPEX/OPEX-записей с `DecisionReport`.

## Правила совместимости

1. Legacy-форматы не удаляются.
2. `configurations`, `candidate_solutions`, `alternatives` и `scores` остаются рабочими.
3. Новый общий пул добавляется рядом как `candidate_configurations`.
4. AHP-агрегация умеет читать как старое поле `devices`, так и новое поле `components`.
5. Экспорт `DecisionReport` показывает источник пула и отдельные результаты GA, AHP, Pareto, Hybrid.

## Что становится возможным дальше

- TCO считается на уровне одной альтернативы через `TCOModelService`;
- NPV получает финансовые потоки от TCO выбранной конфигурации или текущих итогов затрат;
- `DecisionReport` собирает компоненты, стоимость, метрики, GA/AHP/Pareto/Hybrid, ограничения и предупреждения;
- добавление нового способа генерации альтернатив не требует нового ad-hoc-формата.

## Ограничения текущего этапа

- UI ещё может показывать старые поля, потому что часть экранов сохраняет совместимость.
- `CandidateConfiguration` пока не является единственным входным форматом всех алгоритмов: адаптеры сохраняют старые сценарии.
- История нескольких пулов пока не хранится; фиксируется последний пул текущей области.
