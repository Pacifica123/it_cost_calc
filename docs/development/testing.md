# Тестирование проекта

## Зачем это нужно

Репозиторий содержит не только GUI-код, но и самостоятельные расчётные модули, связанные с предметной логикой проекта.
Поэтому минимальный набор тестов нужен не ради формальности, а чтобы безопасно развивать математику, экспорт и сценарии загрузки данных.

## Какие уровни тестов есть

### Unit
Проверяют отдельные функции и сервисы:
- NPV-расчёт;
- агрегацию затрат;
- AHP-математику и фильтрацию ограничений;
- модели предметной области;
- сервисы работы с сущностями и энергопотреблением.

### Integration
Проверяют сквозные сценарии поверх нескольких слоёв сразу:
- загрузку демонстрационного JSON-набора в runtime-репозиторий;
- подготовку сводки затрат и экспорт в CSV;
- запуск AHP-анализа на примере и сохранение результата в JSON;
- построение каталога оборудования из example-снимков.

### Regression
Фиксируют ожидаемое поведение эталонных кейсов:
- победителя и ранжирование в AHP-кейсе;
- победителя, веса и кратности критериев в кейсе анализа важности критериев.

## Как запускать

Запуск всего набора:

```bash
pytest
```

Только unit:

```bash
pytest tests/unit
```

Только integration:

```bash
pytest tests/integration
```

Только regression:

```bash
pytest tests/regression
```

## Что особенно важно

- если меняется формула, правило фильтрации или схема весов, нужно обновлять не только код, но и регрессионные ожидания;
- runtime-артефакты тестов не должны возвращаться в репозиторий, поэтому нужно использовать `tmp_path`, а не реальные файлы из `data/generated/`;
- для новых алгоритмических модулей желательно сразу добавлять хотя бы один unit-тест и один регрессионный пример.
