# Исследование: парсинг каталога оборудования

## Зачем эта зона есть в проекте

Для более серьёзного подбора конфигураций проекту нужен каталог оборудования:
- с ценами;
- с характеристиками;
- с признаками совместимости;
- с параметрами энергопотребления;
- с привязкой к реально существующим позициям магазина.

Именно поэтому в репозитории присутствует отдельная исследовательская зона, посвящённая парсингу и нормализации данных о товарах.

## Как это устроено

Архитектура построена так:
- `src/` — расчётное и аналитическое приложение;
- `tools/catalog_parser/` — отдельный инструмент обновления каталога оборудования;
- `data/generated/catalog/equipment_catalog.json` — нормализованный результат работы инструмента.

Главная идея проста:
- парсер **обновляет каталог**;
- приложение **загружает каталог в staging**;
- пользователь подтверждает пригодные позиции;
- только подтверждённые позиции становятся альтернативами ТО;
- сетевой сбор и HTML-разбор не тащатся в runtime GUI.

## Что показали накопленные материалы

По ранее собранным рабочим заметкам и по содержимому `tools/catalog_parser/legacy/` видно следующее:

- парсинг DNS сталкивался с защитой и challenge-механиками;
- часть сценариев упиралась в ограничения автоматизации и риск блокировок;
- по Ozon и другим площадкам тоже возникали препятствия вроде капчи;
- селекторы и извлечение характеристик работали нестабильно и требовали доработки.


## Best-effort извлечение сетевых характеристик

В рамках развития метрик ТО добавлен ограниченный parser для названий сетевого оборудования DNS-Shop. Он не заменяет ручной ввод и не является частью runtime GUI, но может подготовить поля `lan_ports`, `lan_speed_mbps`, `wifi_standards`, `wifi_generation_rank`, `wifi_total_mbps`, `ipv6_support`.

Для прозрачности вместе с распознанными полями сохраняются `parsed_metrics`, `parse_warnings`, `confidence` и `parse_source`. Если ручные `specs` уже содержат такое же поле, ручное значение не перезаписывается.

## Практический вывод

Парсер следует рассматривать как:
- исследовательскую наработку;
- технический инструмент наполнения справочника;
- отдельный CLI-процесс;
- задел на будущее для сценария «обновить каталог оборудования».

Сетевой парсер не является частью основного пользовательского runtime. При этом
runtime теперь содержит безопасный экран импорта уже подготовленных JSON, CSV
и XLSX. Это закрывает прежний разрыв между созданием `equipment_catalog.json`
и фактическим наполнением вариантов ТО.

Перед импортом пользователь может исправить назначение, цену, количество,
рабочие места и технические метрики. Исходная карточка при этом сохраняется
отдельно, поэтому отчёт различает данные источника и ручные исправления.

## Контракт каталога v2

Вторая версия схемы разделяет:

- идентичность модели: бренд, модель, MPN и GTIN;
- коммерческое предложение: цена, валюта, наличие, регион и время наблюдения;
- технические характеристики;
- происхождение полей;
- статус ручной проверки.

Старые schema v1 каталоги продолжают приниматься staging-сервисом.

## Почему это полезно оставить в репозитории

Даже в незрелом виде этот раздел показывает:
- как проект подходил к наполнению базы оборудования;
- какие технические ограничения были выявлены;
- почему справочник оборудования пока не автоматизирован полностью.
