# Профили области анализа ПО/ТО

## Назначение

Документ фиксирует результат Этапа 3 roadmap концептуальной связанности: различия между программным обеспечением и техническим обеспечением больше не должны жить только во вкладках интерфейса. Для этого введён прикладной профиль области анализа — `AnalysisScopeProfile`.

Профиль описывает, какие категории, типы компонентов, критерии, ограничения, веса и пояснения относятся к конкретной области анализа. UI теперь выбирает область и передаёт пользовательские параметры, а прикладной слой возвращает согласованный профиль и готовит предметные правила для аналитических сценариев.

---

## 1. Общая структура профиля

`AnalysisScopeProfile` хранится в `src/application/services/analysis_scope_profile_service.py` и включает следующие поля.

| Поле | Смысл |
|---|---|
| `scope` | Машинный идентификатор области анализа: `technical` или `software` |
| `label` | Короткая подпись для UI: `ТО`, `ПО` |
| `title` | Полное человекочитаемое название области |
| `capital_categories` | CAPEX-категории, из которых собираются кандидаты анализа |
| `operational_categories` | OPEX-категории, относящиеся к области анализа или будущему TCO |
| `component_types` | Допустимые `component_type` для области |
| `criteria` | Критерии GA/AHP-сценариев: идентификатор, подпись, направление оптимизации, метрика |
| `constraints` | Ограничения области: бюджет, обязательные категории, лимиты и минимальные значения |
| `default_weights` | Базовые веса критериев в порядке профиля |
| `metric_extractors` | Описание того, как извлекаются метрики из runtime-записей |
| `explanation_rules` | Текстовые правила для подсказок и объяснений результата |

Профиль остаётся прикладным контрактом. Он не заменяет domain-модель и не требует немедленного изменения формата runtime-JSON.

## 2. Профиль технического обеспечения

Профиль ТО описывает физическую и инфраструктурную часть решения.

| Блок | Значение |
|---|---|
| `scope` | `technical` |
| Метка | `ТО` |
| CAPEX-категории | `server`, `client`, `network` |
| OPEX-категории | `server_rental`, `backup`, `server_administration` |
| Компоненты | `server`, `workstation`, `peripheral`, `network_device`, `support_service`, `backup_service` |

### Критерии ТО

| Критерий | Направление | Метрика | Пояснение |
|---|---:|---|---|
| ОЗУ, ГБ | max | `total_ram_gb` | Суммирует явно заданный объём оперативной памяти |
| CPU, ядра | max | `total_cpu_cores` | Суммирует явно заданное число вычислительных ядер |
| Накопитель, ГБ | max | `total_storage_gb` | Суммирует объём постоянной/внешней памяти |
| Энергопотребление | min | `total_power_watts` | Считает суммарную мощность выбранных компонентов |
| Стоимость | min | `capital_cost` | Считает капитальную стоимость выбранного набора как мягкий tie-breaker внутри бюджета |

### Ограничения ТО

| Ограничение | Тип | Применение |
|---|---|---|
| Бюджет | общий лимит | Обрабатывается универсальным GA-сервисом через `max_budget` |
| Обязательный серверный контур | обязательная категория | `required_category_server` |
| Обязательный клиентский контур | обязательная категория | `required_category_client` |
| Обязательный сетевой контур | обязательная категория | `required_category_network` |
| Лимит мощности | числовой лимит | `power_limit`, если пользователь задал максимальную мощность |
| Минимум клиентских мест | числовой минимум | `client_capacity_required`, если пользователь задал целевое количество мест |

Ключевое правило: `client` не равен рабочему месту автоматически. Рабочее место определяется через `client_seats` или `component_type=workstation`. Поэтому МФУ и другая периферия могут оставаться в клиентском контуре, но не закрывают ограничение по пользователям.

`category_coverage` больше не является soft-критерием ТО. Наличие серверного, клиентского и сетевого контуров задаётся только фильтром категорий: `+` — обязательно включить, `−` — исключить, пусто — не учитывать в ограничениях.

## 3. Профиль программного обеспечения

Профиль ПО описывает лицензии, подписки, программные сервисы, внедрение и сопровождение.

| Блок | Значение |
|---|---|
| `scope` | `software` |
| Метка | `ПО` |
| CAPEX-категории | `licenses` |
| OPEX-категории | `subscription_licenses`, `migration`, `testing`, `labor_costs` |
| Компоненты | `software_license`, `software_subscription`, `software_service`, `implementation_service`, `support_service` |

### Критерии ПО

| Критерий | Направление | Метрика | Пояснение |
|---|---:|---|---|
| Функциональная пригодность | max | `functionality_score` | Средняя функциональная оценка ПО, если она явно задана в данных |
| Сопровождение | max | `support_score` | Средняя оценка поддержки/сопровождения, если она явно задана |
| Стоимость владения ПО | min | `capital_cost` | Считает стоимость выбранных программных компонентов как мягкий tie-breaker внутри бюджета |

### Ограничения ПО

| Ограничение | Тип | Применение |
|---|---|---|
| Бюджет | общий лимит | Обрабатывается универсальным GA-сервисом через `max_budget` |
| Обязательный лицензионный контур | обязательная категория | `required_category_licenses` |
| Минимум лицензий | числовой минимум | `software_license_quantity_required`, если пользователь задал минимум пользователей/лицензий |
| Лимит подписки | будущий TCO-лимит | зарезервирован для развития OPEX/TCO |
| Минимум функционального покрытия | будущий критерий качества | зарезервирован для развития профиля ПО |
| Наличие сопровождения | будущая проверка поддержки | зарезервировано для связи с OPEX-сопровождением |

Ключевое правило: программный профиль не требует физический сервер, не требует сетевой контур и не использует мощность как обязательный критерий. Клиентские устройства не считаются лицензиями.

Минимум лицензий/пользователей в ПО также трактуется как фильтр, а не как мягкий вес: конфигурация либо закрывает требуемую ёмкость, либо не допускается к сравнению.

## 4. Подключение профилей к аналитическим вкладкам

После Этапа 3 профили подключены к следующим местам.

| Место | Что изменилось |
|---|---|
| `GeneticOptimizationTab` | Переключатель ПО/ТО, категории, критерии, ограничения, веса и подсказки берутся из `AnalysisScopeProfileService` |
| `GeneticOptimizationService` | Может принимать `analysis_scope`/`analysis_profile`, строить критерии и ограничения из профиля и добавлять профиль в результат/export payload |
| `GeneticAhpRankingService` | Получает те же GA-опции и поэтому ранжирует кандидатный пул, подготовленный по профилю области |
| `ConfigurationSelectionTab` | Использует общий сервис профилей для подписей областей анализа |
| `CriteriaImportanceTab` | Использует общий сервис профилей для подписей областей анализа |
| `DecisionDemoDataService` | Формирует scoped payloads по зарегистрированным профилям и вкладывает metadata профиля в демонстрационные данные |
| Экспорт GA/AHP | В JSON payload добавляется `analysis_profile`, чтобы отчёт было проще объяснять и проверять |

UI всё ещё отвечает за ввод значений, отображение таблиц и пользовательские подсказки. Но источник правил «чем отличается ПО от ТО» перенесён в прикладной слой.

## 5. Будущий сводный профиль ИТ-решения

В коде зарезервирован `IT_SOLUTION_PROFILE`. Он пока не подключён к UI как полноценный сценарий, потому что сначала нужно унифицировать формат кандидатных альтернатив на Этапе 4.

Целевой смысл сводного профиля:

```text
ИТ-решение = ТО + ПО + внедрение + сопровождение + стоимость владения + аналитический отчёт
```

После появления общей сущности кандидатной альтернативы сводный профиль сможет объединять технические, программные, внедренческие и общие расходы без копирования логики между вкладками.

## 6. Риски и переходные ограничения

| Риск | Как снижен в Этапе 3 |
|---|---|
| Поломка старых JSON | Runtime-формат не изменён; профили работают поверх `scope`/`component_type` и legacy-категорий |
| Слишком резкий перенос логики из UI | UI переведён на сервис профилей постепенно, без изменения внешнего сценария пользователя |
| Дублирование критериев GA/AHP | Критерии GA строятся из профиля; AHP и Pareto-вкладки используют общий список профилей и scoped payloads |
| Неполный профиль ПО | Базовые критерии ПО реализованы; более глубокие TCO/OPEX-ограничения зарезервированы для следующих этапов |
| Смешение ТО и ПО | Профиль ПО не включает мощность и обязательные технические категории; профиль ТО не считает лицензии рабочими местами |

---

## 7. Критерий завершения этапа

Этап 3 считается выполненным, потому что:

- различия ПО/ТО описаны отдельной прикладной сущностью `AnalysisScopeProfile`;
- профиль можно получить и проверить отдельно через `AnalysisScopeProfileService`;
- GA и GA+AHP получают категории, критерии и ограничения из профиля;
- AHP и анализ важности критериев используют общий сервис профилей для выбора области;
- экспорт аналитического результата получает metadata профиля;
- добавление нового профиля больше не требует копировать базовые списки категорий и критериев между вкладками.
