# Общий пул альтернатив ПО/ТО после GA

## Назначение изменения

Патч реализует следующий шаг roadmap переработки аналитических блоков: внутри каждой рабочей области ПО/ТО появляется простой общий пул кандидатных альтернатив. Он нужен, чтобы GA, AHP и Pareto не выглядели как несвязанные острова и при этом не сливались обратно в старую странную схему `GA + AHP` внутри одной панели.

Главное ограничение реализации: не переусложнять. Пул хранит только последний явно переданный набор top-N кандидатов текущего scope и не пытается быть базой данных, журналом запусков или механизмом автоматического пересчёта.

## Как это работает в интерфейсе

1. Пользователь запускает GA в панели `GA: поиск кандидатных конфигураций`.
2. GA показывает лучший найденный набор и количество top-N кандидатов, доступных для передачи.
3. Пользователь нажимает `Передать top-N в общий пул`.
4. Рабочая область текущего scope обновляет AHP и Pareto:
   - AHP получает те же кандидаты как таблицу конфигураций;
   - Pareto получает кейс с теми же альтернативами и оценками по критериям профиля;
   - обе панели показывают текстовый статус источника пула.

Так сохраняется понятная цепочка:

`GA ищет кандидатов → общий пул фиксирует альтернативы → AHP/Pareto анализируют тот же набор`.

## Почему пул не обновляется автоматически

Автоматический пересчёт всех методов при каждом запуске GA выглядел бы удобно, но для текущего этапа это вредно:

- пользователь хуже понимает, какой набор именно анализируется;
- AHP и Pareto могут внезапно перезаписать ручные правки;
- сложнее объяснить на защите, где закончился поиск GA и где началось независимое ранжирование.

Поэтому передача сделана явным действием. Это чуть менее автоматизировано, зато намного понятнее методически.

## Как строится Pareto-кейс из общего пула

Старый модуль анализа важности критериев ожидает, что все критерии заданы как оценки, где больше — лучше. Профиль ПО/ТО уже содержит направления критериев (`max` или `min`), поэтому сервис пула приводит значения к простой шкале `1..5`:

- для `max`-критерия большее значение получает больший score;
- для `min`-критерия меньшее значение получает больший score;
- если у всех альтернатив значение одинаковое, ставится нейтральная оценка `3.0`.

Это не заменяет полноценную будущую модель Pareto, но уже убирает главный методический перекос: стоимость и мощность не трактуются как `больше — лучше`.

## Ограничения MVP

- Пул не сохраняется между запусками приложения.
- Пул хранит только последний явно переданный набор текущей области.
- История запусков GA не ведётся.
- AHP и Pareto получают общий набор альтернатив, но не запускаются автоматически.
- Гибридная панель пока остаётся отдельным способом сводной оценки и не читает этот in-memory пул напрямую.

## Проверки

Добавлены unit-тесты для общего пула:

- AHP-конфигурации получают `people` из `client_seats`;
- ПО-кандидаты получают `people` из `software_license_quantity`;
- Pareto-кейс строится из тех же альтернатив;
- `min`-критерии в Pareto-кейсе инвертируются на шкале полезности.
