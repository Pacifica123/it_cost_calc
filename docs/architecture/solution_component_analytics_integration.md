# Аналитическая интеграция `SolutionComponent`

Документ фиксирует результат этапа C6 из `solution_component_editor_roadmap.md`: компоненты редактора подключены к GA, AHP, Pareto и Hybrid через существующий формат `CandidateConfiguration`, без отдельной параллельной модели альтернатив.

## Что изменилось

После C6 строгий компонент редактора может попасть в аналитический контур так же, как обычный элемент предметной модели, но только после профильной проверки. После финального рефакторинга новая цепочка выглядит так:

```text
entities.solution_components[]
  → SolutionComponentNormalizationService
  → SolutionComponentAnalyticsIntegrationService
  → CandidateConfiguration
  → GA → общий пул → AHP / Pareto → Hybrid → DecisionReport
```

Задача интеграции — связать редактор с аналитическими методами и доказать, что данные проходят через общий контракт, а не через отдельный скрытый формат.

## Правила включения в альтернативы

Компонент редактора не включается в аналитику автоматически только потому, что он сохранён в runtime-данных.

| Состояние компонента | Поведение |
| --- | --- |
| `strict_analysis_participation=true`, валидный `scope`, валидный `component_type` | может стать компонентом альтернативы `CandidateConfiguration` |
| `draft`, `blocked` или неполные обязательные поля | остаётся в исключённых компонентах с причиной |
| `mixed` / `bundle` без отдельного профильного разбиения | исключается из профильного пула, чтобы не получить скрытое преимущество |
| компонент другого профиля | не попадает в текущий профильный запуск, но может попасть в запуск своего профиля |
| legacy sandbox-строка | не участвует без ручной конвертации |

Так сохраняется правило: неполные данные дают предупреждение или причину исключения, а не неявные нули в рейтинге.

## Профильные метрики

`SolutionComponentAnalyticsIntegrationService` обогащает компонент только теми метриками, которые нужны существующим профилям ПО/ТО.

Для технического профиля используются:

- `source_category` / `category` для hard-фильтра категорий;
- `client_seats` для проверки минимального числа рабочих мест;
- `ram_gb`, `cpu_cores`, `storage_gb`, `max_power_watts` и сетевые метрики, если они заданы;
- `purchase_cost` + разовые + годовые регулярные расходы для TCO и бюджетного фильтра;
- предупреждения по неполным характеристикам.

Для программного профиля используются:

- `license_units` или `quantity` для фильтра минимального числа лицензий;
- `functionality_score` и `support_score` для качественных критериев;
- стоимость владения программным компонентом;
- мощность не требуется и не подставляется искусственно.

Если профильная метрика отсутствует, сервис добавляет warning. Он не пытается угадать качество, производительность или надёжность.

## AHP

Для AHP компоненты редактора сначала превращаются в `CandidateConfiguration`, а затем уже в существующий AHP-формат через `CandidateConfigurationService.to_ahp_configurations`.

Это означает, что форма редактора не становится новым источником AHP-альтернатив. AHP получает тот же общий пул альтернатив, что и Pareto, Hybrid и итоговый отчёт.

Контрольный учебный пример AHP остаётся отдельным и не меняет результат. Smoke-сценарии отдельно проверяют, что базовые AHP-примеры остаются воспроизводимыми.

## GA

`GeneticOptimizationService` умеет добавлять строгие `SolutionComponent`-строки в пул GA-кандидатов для выбранного профиля.

Правила сохранены:

- бюджет применяется как hard-фильтр общей стоимости item-ов;
- `client_seats` отвечает за рабочие места, поэтому периферия не превращается в рабочее место;
- `license_units` отвечает за программную ёмкость;
- `max_power_watts` используется только в техническом профиле и связан с энергопотреблением;
- seed и параметры запуска остаются в `parameters.ga` для воспроизводимости.

Результаты GA по-прежнему возвращают `candidate_solutions` и `candidate_configurations`. Если в выбранный набор попал компонент редактора, он сохраняет признаки `source_format=solution_component` и `source_component_id`.

## Общий пул, Pareto и Hybrid

Основной пользовательский сценарий после рефакторинга:

```text
GA-кандидаты или demo-пул → scoped candidate pool → AHP / Pareto → Hybrid
```

Pareto не становится скрытым весом гибридной оценки: он показывает компромиссный статус и предупреждения. Hybrid не запускает GA или AHP заново, а собирает последние результаты методов и фиксируется в `DecisionReport` отдельным блоком.

Это закрывает ключевой риск варианта C: компонент редактора не получает скрытого преимущества из-за того, что он был введён через другую вкладку.

## Проверка

Для headless-проверки добавлен smoke-сценарий `scripts/run_solution_component_analytics_demo.py`. Он не использует `pytest` и проверяет, что:

- технические и программные компоненты редактора проходят профильную фильтрацию;
- draft и mixed-компоненты не попадают в профильные альтернативы без причин;
- AHP получает компоненты редактора через `CandidateConfiguration`;
- контрольный AHP-пример не меняет победителя;
- GA может выбрать item, пришедший из `SolutionComponent`;
- общий пул сохраняет тот же набор `CandidateConfiguration` и маркеры происхождения компонентов.
