# Финансовая интеграция `SolutionComponent`

Документ фиксирует результат этапа C5 из `solution_component_editor_roadmap.md`: компоненты редактора подключены к существующей финансовой цепочке `стоимость → TCO → NPV` без отдельного параллельного калькулятора.

## Что изменилось

После C5 строгие компоненты редактора могут участвовать в финансовой модели так же, как обычные данные проекта. Черновики и заблокированные строки остаются видимыми, но не попадают в строгие суммы.

Новая цепочка выглядит так:

```text
solution_components[]
  → SolutionComponentRuntimeService
  → SolutionComponentFinancialIntegrationService
  → CostModel
  → TCOModelService
  → NPV financial_basis
  → NPVReportService / DecisionReport
```

## Правила стоимости

Финансовые поля больше не сворачиваются в одну неопределённую сумму. Они передаются в `CostModel` по смыслу.

| Поле компонента | Финансовая роль |
| --- | --- |
| `purchase_cost` | стартовый CAPEX |
| `implementation_cost` | разовые расходы внедрения |
| `migration_cost` | разовые расходы миграции |
| `testing_cost` | разовые расходы тестирования и пилота |
| `monthly_cost` | регулярный ежемесячный OPEX |
| `annual_cost` | ежегодный OPEX, пересчитанный в месячный эквивалент для TCO |
| `energy_cost` | ручная месячная энергетическая стоимость, если энергия применима к типу компонента |

Пользовательский эффект, ожидаемая экономия или рост выручки не смешиваются с расходами. Для NPV они остаются отдельным параметром `annual_effect`.

## Энергетическая часть

Энергия подключается только к применимым компонентам.

| Тип компонента | Поведение |
| --- | --- |
| `server`, `workstation`, `network_device`, `peripheral` | энергия может быть рассчитана по мощности, количеству, часам работы, рабочим дням и тарифу |
| `software_license`, `software_subscription`, `software_service` | мощность не требуется, энергетический расчёт не применяется |
| `mixed` | энергия включается только при явном признаке технической части или выделенной технической метрике |

Для каждого финансового payload теперь сохраняется источник энергетической части:

- `derived_ready` — можно рассчитать от мощности и профиля работы;
- `manual` — пользователь ввёл месячную сумму вручную;
- `missing_usage_profile` — компонент технически применим, но не хватает профиля работы;
- `not_applicable` — энергетический расчёт не относится к типу компонента.

## TCO и глобальные итоги

`PrepareCostSummaryUseCase` получил опциональную связку с `SolutionComponentRuntimeService`. Если в runtime есть строгие компоненты редактора, их вклад добавляется к общим итогам:

- CAPEX компонента добавляется к `total_capital`;
- разовые внедренческие расходы добавляются к `total_operational_one_time`;
- регулярная часть без энергии добавляется к `total_operational_monthly`;
- энергетическая часть добавляется к `electricity_costs`;
- после объединения пересчитывается общий `tco`.

При этом в итогах сохраняется отдельный блок `solution_component_financial_chain`, чтобы можно было объяснить, какой вклад пришёл именно из редактора компонентов.

## NPV-база

C5 не делает NPV искусственно положительным. Из TCO формируется финансовая база:

- `investment` берётся из стартовых расходов компонента;
- `cash_flows` формируются как `annual_effect - annual_opex`;
- если `annual_effect = 0`, NPV получает предупреждение, что расчёт отражает только оттоки;
- если расходы есть, а эффект не задан, вывод остаётся отрицательным или нулевым без подмены результата.

Так сохраняется методическая корректность: расходы передаются автоматически, а эффект пользователь должен задать отдельно и обосновать.

## Предупреждения

Финансовая модель теперь явно предупреждает о типовых проблемах:

- все расходы компонента указаны одной суммой;
- `annual_cost` пересчитан в месячный эквивалент для TCO;
- энергия указана вручную и не связана с мощностью;
- энергия задана для типа, где она не применима;
- `mixed`-компонент имеет энергетическую часть без выделения технической части;
- NPV получил расходы, но не получил отдельный эффект или экономию.

Эти предупреждения попадают в runtime-снимок, финансовую цепочку и `DecisionReport`.

## Проверка

Для headless-проверки добавлен smoke-сценарий `scripts/run_solution_component_financial_demo.py`. Он не использует `pytest` и проверяет, что:

- минимум три строгих компонента из demo-fixture проходят финансовую цепочку;
- `purchase_cost`, внедрение, тестирование, ежемесячные и ежегодные расходы попадают в TCO;
- технический компонент даёт производную энергетическую стоимость;
- программный компонент не требует энергетического расчёта;
- NPV получает investment из TCO;
- при нулевом `annual_effect` NPV показывает предупреждение;
- глобальные итоги стоимости содержат отдельный блок `solution_component_financial_chain`.
