# Контракт демо-данных и контрольных сценариев

## Назначение

После этапа 8 демонстрационные данные больше не рассматриваются как отдельная смысловая реальность. Они должны показывать тот же путь, который проходит пользовательский набор данных:

```text
data/fixtures/demo_dataset.json
  → RuntimeEntityNormalizationService
  → DecisionDemoDataService
  → CandidateConfigurationService
  → TCOModelService
  → DecisionReportService
```

Цель контракта — разделить три роли данных и сделать демонстрационный сценарий воспроизводимым без запуска `pytest` и без записи cache-файлов.

## Роли данных

| Роль | Где лежит | Для чего используется | Что нельзя делать |
|---|---|---|---|
| Учебные данные | `data/examples/ahp/`, `scripts/run_ga_demo.py`, `data/examples/optimization/` | Пояснение отдельной методики: AHP, GA или малый синтетический пример | Нельзя выдавать их за полноценный пользовательский сценарий выбора ИТ-решения |
| Демонстрационные данные | `data/fixtures/demo_dataset.json` | Сквозной показ: CAPEX/OPEX/энергия → альтернативы → TCO → DecisionReport | Нельзя хранить строки без `scope` и `component_type` |
| Регрессионные инварианты | `data/fixtures/regression/demo_control_invariants.json` | Проверка, что демонстрационный сценарий не распался после изменений | Нельзя использовать как runtime-данные пользователя |

Реестр ролей закреплён в `data/fixtures/demo_scenarios.json`. Он нужен не для приложения, а для разработчика и проверок: по нему видно, какой файл является учебным примером, какой — пользовательским демо, а какой — контрольным набором инвариантов.

## Требования к `demo_dataset.json`

Демонстрационный набор содержит верхнеуровневые metadata-поля и блок `entities`. Загрузчик runtime-данных по-прежнему берёт только `entities`, поэтому обратная совместимость сохранена.

Каждая строка в известных CAPEX/OPEX-категориях должна иметь:

- `scope` — `technical`, `software`, `implementation` или `common`;
- `component_type` — доменный тип компонента;
- стоимость в формате своей категории:
  - `quantity` + `price` для CAPEX;
  - `one_time_cost` для разовых работ;
  - `monthly_cost` для подписок и сопровождения;
- дополнительные метрики, если они нужны следующему слою анализа:
  - `max_power` для технических компонентов;
  - `client_seats` для рабочих мест;
  - `client_seats=0` для периферии.

Такой набор позволяет показать не только покупку оборудования, но и внедрение, тестирование, резервное копирование и сопровождение. Благодаря этому `TCOModelService` получает не пустой OPEX, а реальную финансовую структуру.

## Контрольный сценарий

`DemoControlScenarioService` собирает контрольный payload без записи файлов:

1. нормализует `entities`;
2. строит профильные payload для ТО и ПО;
3. формирует общий пул `candidate_configurations`;
4. проверяет наличие `totals.tco` у каждой альтернативы;
5. запускает анализ важности критериев по тем же альтернативам;
6. собирает частичный `DecisionReport`;
7. возвращает инварианты и ошибки валидации.

CLI-проверка:

```bash
python -B scripts/run_demo_control_scenario.py --check-only
```

Команда ничего не пишет в `data/generated/` и не требует `pytest`. В devctl-патче её можно использовать как smoke-проверку.

## Воспроизводимость

Для демонстрационного сценария фиксируются:

| Параметр | Значение / источник |
|---|---|
| GA seed | `42` |
| GA init mode | `heuristic_init` |
| AHP weights | `AnalysisScopeProfileService.default_weights` |
| Пул альтернатив | `DecisionDemoDataService.scoped_payloads[*].candidate_configurations` |
| Отчёт | `DecisionReportService`, без повторного запуска методов внутри экспорта |

Также формируется `input_fingerprint`: SHA-256 от нормализованных сущностей и контракта воспроизводимости. Он нужен для будущего сравнения снапшотов, если появится отдельный артефакт сохранённого контрольного отчёта.

## Границы этапа 8

Этап 8 не меняет сами алгоритмы AHP, GA, GA + AHP и NPV. Он укрепляет входные данные и контрольный сценарий вокруг уже реализованной общей модели.

В результате демо-данные становятся не набором случайных строк для заполнения вкладок, а коротким воспроизводимым примером всей предметной цепочки проекта.
