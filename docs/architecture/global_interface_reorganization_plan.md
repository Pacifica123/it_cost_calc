# План глобальной реорганизации интерфейса

## 1. Назначение документа

Этот документ фиксирует план крупной переработки интерфейса после этапа синхронизации рабочего пространства. План не предлагает косметически переименовать вкладки, а описывает новую структуру пользовательского сценария: пользователь сначала выбирает предметную область решения, затем внутри неё работает с затратами, критериями и аналитикой.

Главная идея: `Программное обеспечение` и `Техническое обеспечение` становятся не простыми таблицами ввода, а полноценными рабочими областями. Внутрь них постепенно переезжают сценарии, которые сейчас вынесены в отдельные верхнеуровневые вкладки: генетический подбор, AHP-анализ конфигураций и обоснование выбора ИТ-решения.

Документ нужен как переходный архитектурный контракт перед кодовыми патчами. Он также специально оставляет место для следующей большой работы: пересборки критериев и демонстрационных данных после завершения интерфейсной реформы.

## 2. Текущее состояние, от которого отталкивается план

В текущем настольном приложении верхний `ttk.Notebook` собирается в `src/app.py`. Сейчас пользователь видит несколько независимых вкладок:

| Текущая вкладка | Текущая роль | Проблема для будущего интерфейса |
|---|---|---|
| `ТО` | ввод технических CAPEX-позиций: серверы, клиенты, сеть | слишком короткое название и слишком узкая роль для будущей рабочей области ТО |
| `ПО` | ввод программных CAPEX-позиций: лицензии | слишком узкая роль для будущей рабочей области ПО |
| `Операционные затраты` | ввод подписок, аренды, миграции, тестирования, резервирования, труда и администрирования | разрывает финансовую природу затрат с предметной областью ПО/ТО |
| `Редактор компонентов` | строгий ввод `SolutionComponent` | должен остаться самостоятельным, но должен лучше связываться с ПО/ТО и экспортом |
| `ИТ-песочница (архив)` | legacy-ввод и ручная конвертация старых записей | не должна оставаться верхнеуровневой вкладкой после стабилизации редактора компонентов |
| `Электроэнергия` | расчёт энергопотребления по техническим компонентам | должна остаться отдельной, но явно объяснять связь только с ТО |
| `Экспорт` | CSV и DecisionReport | должна стать живой сводкой, а не только кнопками выгрузки |
| `NPV-анализ` | финансовая оценка сценария | должна работать от TCO/CAPEX/OPEX-сводки и предупреждать о ручных допущениях |
| `Генетический подбор` | GA и GA + AHP | должен быть профильным блоком внутри ПО/ТО |
| `AHP-анализ конфигураций` | ранжирование альтернатив | должен быть профильным блоком внутри ПО/ТО |
| `Обоснование выбора ИТ-решения` | анализ важности критериев | должен быть профильным блоком внутри ПО/ТО |

Существующие документы уже дают основу для такой реформы: `current_subject_schema.md` фиксирует связь текущих CAPEX/OPEX-категорий с будущими `scope` и `component_type`, `analysis_scope_profiles.md` описывает разные профили ПО/ТО, `candidate_configurations.md` задаёт общий формат альтернатив, а `tco_npv_bridge.md` связывает CAPEX, OPEX, электроэнергию и NPV.

## 3. Целевая верхнеуровневая структура

После реорганизации верхнеуровневый интерфейс должен быть существенно короче и логичнее:

| Целевая вкладка | Назначение | Что входит внутрь |
|---|---|---|
| `Программное обеспечение` | рабочая область программных решений | программные CAPEX, программные OPEX, программные критерии, GA/AHP/Парето для ПО, пояснения выбора ПО |
| `Техническое обеспечение` | рабочая область оборудования и технической инфраструктуры | технические CAPEX, технические OPEX, технические критерии, GA/AHP/Парето для ТО, пояснения выбора ТО |
| `Редактор компонентов` | универсальный ввод сложных компонентов решения | строгие компоненты, черновики, предпросмотр нормализации, ручная конвертация legacy-песочницы |
| `Электроэнергия` | профильный расчёт энергопотребления | расчёт только для энергоёмких технических компонентов и передача результата в TCO/NPV |
| `NPV-анализ` | инвестиционная оценка сценария | TCO-входы, ручные эффекты, денежные потоки, предупреждения по полноте модели |
| `Экспорт` | живая итоговая витрина и выгрузка | сводные итоги, предупреждения, полный Markdown/JSON/CSV-отчёт и частичные экспортные срезы |

Верхнеуровневые вкладки `Генетический подбор`, `AHP-анализ конфигураций`, `Обоснование выбора ИТ-решения`, `Операционные затраты` и `ИТ-песочница (архив)` после реформы не должны оставаться отдельными пунктами главного меню. Их функции должны быть перенесены внутрь профильных рабочих областей или редактора компонентов.

## 4. Принцип новой навигации: рабочая область и раскрываемые блоки

Из-за переноса аналитики внутрь ПО/ТО простая вертикальная прокрутка станет недостаточной. Нужна концепция иерархических блоков, которые можно скрывать, раскрывать и по возможности менять по размеру.

### 4.1. Базовая единица интерфейса

Нужно ввести переиспользуемый виджет условного типа `CollapsibleSection` или `WorkspacePanel`.

| Элемент блока | Назначение |
|---|---|
| Заголовок | название блока: например `CAPEX`, `OPEX`, `Критерии`, `GA`, `AHP`, `Парето`, `Сводка` |
| Строка статуса | короткое состояние: `заполнено`, `есть предупреждения`, `результат устарел`, `не рассчитано` |
| Кнопка свернуть/развернуть | уменьшает визуальный шум без удаления данных |
| Основные действия | запуск расчёта, обновление, экспорт среза, переход к связанному блоку |
| Тело блока | таблица, форма, график, дерево результатов или Markdown-превью |
| Нижняя подсказка | поясняет, какие данные участвуют в общей модели и какие не участвуют |

Такой блок должен быть простым и безопасным для Tkinter: на первом этапе достаточно сворачивания тела и обновления scrollregion. Полноценная док-панель в стиле Adobe может быть эволюционным улучшением, а не обязательным условием первого патча.

### 4.2. Панельная компоновка

Для вкладок `Программное обеспечение` и `Техническое обеспечение` целесообразно использовать двухуровневую схему:

| Уровень | Возможная реализация | Назначение |
|---|---|---|
| Верхний уровень | `ttk.PanedWindow` или несколько крупных вертикальных областей | разделить ввод данных, аналитику и сводку |
| Средний уровень | раскрываемые блоки | скрывать редко используемые секции |
| Нижний уровень | таблицы, формы, вложенные результаты | не перегружать главный экран деталями |

Ссылка на Adobe здесь должна пониматься как UX-референс: панели можно сворачивать, область можно перераспределять по размеру, пользователь не обязан видеть всё сразу. Не требуется копировать внешний вид Adobe или делать сложную drag-and-drop док-систему на первом этапе.

### 4.3. Сохранение состояния интерфейса

После появления скрываемых панелей желательно сохранять пользовательское состояние:

| Состояние | Где хранить | Зачем |
|---|---|---|
| свернутые/развернутые блоки | `data/generated/ui_layout.json` или отдельный UI-settings storage | пользователь не раскрывает одно и то же каждый запуск |
| размеры панелей | тот же UI-settings storage | рабочая область остаётся удобной на разных мониторах |
| последний выбранный профиль/срез | runtime UI-state | экспорт понимает контекст последней работы |

На первом кодовом этапе допустимо не сохранять размеры, но важно заложить API так, чтобы позднее это не потребовало переписывать все вкладки.

## 5. Соотношение ПО/ТО и CAPEX/OPEX

CAPEX/OPEX и ПО/ТО не должны конкурировать как две разные классификации. Это две независимые оси:

| Ось | Отвечает на вопрос | Пример |
|---|---|---|
| ПО/ТО | к какой предметной области относится компонент | лицензия, подписка, сервер, ПК, сеть |
| CAPEX/OPEX | как затрата ведёт себя финансово | покупка один раз, ежемесячная подписка, ежегодное сопровождение, разовая миграция |

Поэтому интерфейс не должен выбирать между вкладками `ПО/ТО` и `CAPEX/OPEX`. Он должен показывать CAPEX/OPEX внутри каждой предметной области.

### 5.1. Матрица классификации

| Предметная область | CAPEX: покупка или стартовая инвестиция | OPEX: регулярные или операционные расходы | Особые случаи |
|---|---|---|---|
| ПО | бессрочные лицензии, разовые коробочные лицензии, покупка программного модуля | SaaS, подписки, ежегодное продление, программная поддержка | внедрение ПО, миграция данных и обучение не являются самой лицензией, но могут быть привязаны к программному сценарию |
| ТО | покупка серверов, ПК, сетевого оборудования, периферии | аренда серверов, обслуживание, администрирование, гарантийное сопровождение, облачная инфраструктура как ресурс | амортизация не является платежом сама по себе, но может использоваться как аналитическое распределение стоимости владения |
| Общий контур | стартовые проектные расходы, которые нельзя честно отнести только к ПО или только к ТО | регулярное сопровождение всего решения, резервное копирование, мониторинг | нужно разрешить явную привязку: `software`, `technical`, `common`, `implementation` |

### 5.2. Разовые лицензии и подписки

Бессрочная лицензия похожа на CAPEX в модели проекта, потому что она увеличивает стартовую инвестицию и дальше не создаёт регулярный денежный отток. Подписка почти всегда является OPEX, потому что её надо оплачивать ежемесячно или ежегодно. Поэтому в интерфейсе программной вкладки должны быть минимум два блока:

| Блок в `Программное обеспечение` | Текущие категории | Финансовая логика |
|---|---|---|
| `ПО / CAPEX` | `licenses` | покупка или единовременная стоимость лицензии |
| `ПО / OPEX` | `subscription_licenses`, часть `backup`, часть `labor_costs` | регулярная подписка, SaaS, поддержка, сопровождение |

Важное ограничение: эта классификация является инженерно-финансовой для TCO/NPV, а не полноценным бухгалтерским заключением. Для дипломного проекта важнее прозрачность денежных потоков: что платится один раз, а что повторяется.

### 5.3. Серверы, аренда и амортизация

Покупка физического сервера или ПК относится к техническому CAPEX: это стартовая инвестиция в оборудование. Аренда сервера или облачного ресурса относится к техническому OPEX: физического актива у пользователя нет, есть периодический платёж за ресурс.

Амортизацию не стоит смешивать с оплатой аренды. В интерфейсе её лучше трактовать как аналитический параметр TCO: она может распределять стоимость купленного оборудования по годам, но не является отдельным исходящим денежным платежом, если пользователь уже внёс покупку в CAPEX.

### 5.4. Миграция, тестирование и внедрение

Миграция и тестирование могут относиться к программному или техническому проекту, но по смыслу это не сама лицензия и не само оборудование. Поэтому их лучше вынести в блок `Внедрение и переход`, который может находиться внутри ПО/ТО как привязанный к области, либо в общем контуре внутри редактора компонентов.

| Вид расхода | Рекомендуемая модель | Почему |
|---|---|---|
| миграция данных на новую систему | one-time implementation cost, scope=`implementation` или привязка к ПО | это разовая работа, а не постоянная подписка |
| монтаж и настройка оборудования | one-time implementation cost, привязка к ТО | это стартовая работа вокруг технического решения |
| тестирование внедрения | one-time testing cost | влияет на инвестиционный сценарий и риски, но не должно становиться регулярным OPEX без причины |
| сопровождение после запуска | recurring support cost | это уже операционное обслуживание |

## 6. Целевая структура вкладки `Программное обеспечение`

Вкладка должна стать программной рабочей областью с блоками:

| Блок | Содержимое | Источник текущего интерфейса |
|---|---|---|
| `Сводка ПО` | количество программных компонентов, CAPEX, OPEX, предупреждения, статус последней аналитики | новый агрегирующий блок |
| `ПО / CAPEX` | бессрочные лицензии и разовые программные покупки | текущая вкладка `ПО` |
| `ПО / OPEX` | подписки, SaaS, программная поддержка, ежегодные платежи | часть текущей вкладки `Операционные затраты` |
| `Внедрение ПО` | миграция данных, настройка программной среды, тестирование программного сценария | часть текущей вкладки `Операционные затраты` и будущие `SolutionComponent` |
| `Критерии ПО` | функциональность, лицензионная ёмкость, поддержка, совместимость, стоимость владения | текущий профиль `software` из `AnalysisScopeProfileService` |
| `Парето ПО` | быстрый отбор альтернатив по стоимости, функциональности и сопровождению | будущий/существующий аналитический контур альтернатив |
| `GA для ПО` | генерация программных конфигураций без технических ограничений мощности и рабочих мест | текущая вкладка `Генетический подбор` с `scope=software` |
| `AHP для ПО` | ранжирование программных альтернатив | текущая вкладка `AHP-анализ конфигураций` с `scope=software` |
| `Обоснование выбора ПО` | веса критериев и объяснение выбора программного решения | текущая вкладка `Обоснование выбора ИТ-решения` с `scope=software` |
| `Локальный экспорт ПО` | экспорт только программного среза с предупреждениями | новая возможность export-конструктора |

Ключевое правило: программная вкладка не должна требовать серверы, мощность, клиентские ПК или сеть как обязательные технические ограничения. Если программный сценарий зависит от серверной инфраструктуры, эта зависимость должна описываться связью с ТО или общим сценарием, а не неявным смешиванием критериев.

## 7. Целевая структура вкладки `Техническое обеспечение`

Вкладка должна стать технической рабочей областью с блоками:

| Блок | Содержимое | Источник текущего интерфейса |
|---|---|---|
| `Сводка ТО` | количество технических компонентов, CAPEX, OPEX, мощность, рабочие места, предупреждения | новый агрегирующий блок |
| `ТО / CAPEX` | серверы, клиентские устройства, сеть, периферия | текущая вкладка `ТО` |
| `ТО / OPEX` | аренда серверов, администрирование, обслуживание, техническая поддержка | часть текущей вкладки `Операционные затраты` |
| `Внедрение ТО` | монтаж, настройка, пусконаладка, техническое тестирование | часть текущей вкладки `Операционные затраты` и будущие `SolutionComponent` |
| `Энергетическая связка` | статус передачи технических компонентов во вкладку электроэнергии | связь с вкладкой `Электроэнергия` |
| `Критерии ТО` | стоимость, мощность, рабочие места, покрытие контуров, надёжность, производительность | текущий профиль `technical` из `AnalysisScopeProfileService` |
| `Парето ТО` | быстрый отбор альтернатив по стоимости, мощности, рабочим местам и покрытию | будущий/существующий аналитический контур альтернатив |
| `GA для ТО` | генерация технических конфигураций с ограничениями сервер/клиент/сеть/бюджет | текущая вкладка `Генетический подбор` с `scope=technical` |
| `AHP для ТО` | ранжирование технических альтернатив | текущая вкладка `AHP-анализ конфигураций` с `scope=technical` |
| `Обоснование выбора ТО` | веса критериев и объяснение выбора технического решения | текущая вкладка `Обоснование выбора ИТ-решения` с `scope=technical` |
| `Локальный экспорт ТО` | экспорт только технического среза с предупреждениями | новая возможность export-конструктора |

Ключевое правило: периферия, МФУ и другие клиентские устройства не должны автоматически превращаться в рабочие места. Рабочие места должны продолжать задаваться через явные метрики вроде `client_seats`.

## 8. Роль редактора компонентов после реформы

`Редактор компонентов` должен остаться самостоятельной вкладкой, потому что он решает не задачу анализа, а задачу нормализованного ввода сложных компонентов. Однако его нужно сильнее связать с рабочими областями ПО/ТО.

| Блок редактора | Назначение |
|---|---|
| `Строгие компоненты` | компоненты, которые уже могут участвовать в TCO, NPV, AHP, GA и экспорте |
| `Черновики и исключённые` | неполные компоненты с причинами исключения |
| `Предпросмотр нормализации` | показывает `scope`, `component_type`, финансовую модель и аналитическую готовность |
| `Связи с ПО/ТО` | показывает, в какую рабочую область попадёт компонент |
| `Legacy-песочница` | бывшая вкладка `ИТ-песочница (архив)` как скрываемый блок ручной конвертации, а не отдельная верхнеуровневая вкладка |

После этого legacy-песочница не исчезает резко, но перестаёт занимать место на верхнем уровне и становится вспомогательным инструментом миграции старых записей.

## 9. Электроэнергия и NPV в новой структуре

### 9.1. Электроэнергия

Вкладка `Электроэнергия` должна остаться отдельной, потому что энергетический расчёт имеет собственные параметры: мощность, часы работы, рабочие дни и тариф. Но она должна явно показывать, что основной источник данных — технические компоненты.

| Правило | Обоснование |
|---|---|
| программные лицензии не попадают в расчёт мощности | ПО само по себе не имеет `max_power` |
| серверы, ПК, сеть и часть периферии могут попадать в расчёт | это технические компоненты с энергопотреблением |
| результат энергии передаётся в TCO/NPV | электроэнергия является регулярным операционным расходом |
| если технические данные изменились после расчёта энергии, нужен статус `устарело` | экспорт не должен молча использовать старый результат |

### 9.2. NPV

Вкладка `NPV-анализ` должна стать финансовым сценарием поверх TCO, а не изолированным калькулятором.

| Источник | Как попадает в NPV |
|---|---|
| ПО / CAPEX | стартовая инвестиция |
| ТО / CAPEX | стартовая инвестиция |
| внедрение, миграция, тестирование | стартовая инвестиция или разовый расход первого периода |
| ПО / OPEX | ежегодный или ежемесячный операционный отток |
| ТО / OPEX | ежегодный или ежемесячный операционный отток |
| электроэнергия | регулярный операционный отток |
| ожидаемый эффект | отдельный пользовательский ввод, потому что затраты сами по себе не доказывают выгоду |

NPV-блок должен показывать предупреждение, если пользователь экспортирует финансовый результат без заданного положительного эффекта, без актуального расчёта энергии или без результатов аналитического выбора.

## 10. Умная вкладка `Экспорт`

Вкладка `Экспорт` должна стать не только местом кнопок, но и живой итоговой витриной. Она должна показывать укрупнённые итоги в реальном времени и одновременно давать управляемую выгрузку полного или частичного отчёта.

### 10.1. Живая витрина

| Блок экспорта | Что показывает |
|---|---|
| `Общая сводка` | общий CAPEX, OPEX, TCO, энергия, NPV, количество компонентов и предупреждения |
| `Сводка ПО` | программные CAPEX/OPEX, последний GA/AHP/Парето-результат ПО, статус актуальности |
| `Сводка ТО` | технические CAPEX/OPEX, энергия, последний GA/AHP/Парето-результат ТО, статус актуальности |
| `Согласованность методов` | совпадают ли лидеры GA, AHP, гибридной оценки и Парето |
| `Финансовая полнота` | хватает ли данных для честного TCO/NPV-вывода |
| `Предупреждения` | устаревшие расчёты, отсутствующие критерии, неполные компоненты, экспорт только части модели |

Для тяжёлых методов не нужно пересчитывать GA/AHP каждую секунду. Достаточно хранить последний результат и помечать его как устаревший, если изменились исходные данные, критерии или профиль.

### 10.2. Полный экспорт

Полный экспорт должен формировать комплексный Markdown-отчёт и машинные JSON/CSV-срезы. В него должны входить:

| Раздел полного отчёта | Содержимое |
|---|---|
| `Исходные данные` | ПО, ТО, компоненты редактора, исключённые черновики |
| `Финансовая модель` | CAPEX, OPEX, энергия, TCO, входы NPV |
| `Аналитика ПО` | Парето, GA, AHP, GA + AHP, критерии и объяснение выбора ПО |
| `Аналитика ТО` | Парето, GA, AHP, GA + AHP, критерии и объяснение выбора ТО |
| `Сводный выбор` | где методы согласны, где конфликтуют, какая конфигурация рекомендуется и почему |
| `NPV` | инвестиции, потоки, ставка, горизонт, итоговый NPV и ограничения интерпретации |
| `Предупреждения и ограничения` | что не учтено, что устарело, где нужен ручной ввод |
| `Методическая справка` | короткое объяснение, что CAPEX/OPEX и ПО/ТО являются разными осями классификации |

### 10.3. Частичный экспорт

Пользователь должен иметь возможность экспортировать только один срез, но с обязательным контекстом и предупреждениями.

| Частичный экспорт | Что включать обязательно | Типовое предупреждение |
|---|---|---|
| только CAPEX | стартовые покупки по выбранному scope или по всему проекту | CAPEX не показывает регулярные расходы и не доказывает лучшую конфигурацию по GA/AHP |
| только OPEX | регулярные и разовые операционные расходы | OPEX не показывает стартовые инвестиции и не равен полной стоимости владения |
| только ПО | программные компоненты, программные критерии и программная аналитика | результат не учитывает техническую совместимость, энергию и оборудование |
| только ТО | технические компоненты, технические критерии, энергию и техническую аналитику | результат не учитывает программные лицензии и программные подписки |
| только GA/AHP | результаты последнего аналитического запуска | результат может быть устаревшим, если данные менялись после запуска |
| только NPV | финансовый сценарий и предупреждения | NPV зависит от заданного эффекта и может не учитывать несогласованные аналитические результаты |

Такой частичный экспорт полезен для защиты и отладки, но он не должен создавать ложного впечатления, что один срез является полной рекомендацией.

## 11. Состояние результатов и устаревание расчётов

После реорганизации интерфейса появится больше взаимосвязей. Поэтому нужен единый принцип актуальности результатов.

| Событие | Что должно устаревать |
|---|---|
| изменение ПО-компонентов | GA/AHP/Парето ПО, программная сводка экспорта, TCO/NPV |
| изменение ТО-компонентов | GA/AHP/Парето ТО, энергия, техническая сводка экспорта, TCO/NPV |
| изменение OPEX | TCO/NPV, экспорт, связанные профильные сводки |
| изменение энергетического расчёта | TCO/NPV и экспорт |
| изменение критериев | соответствующие GA/AHP/Парето-результаты и обоснование выбора |
| изменение NPV-эффекта или ставки | NPV и полный финансовый вывод |

На UI-уровне это можно показывать статусами `актуально`, `требует пересчёта`, `не хватает данных`, `есть предупреждения`. Это особенно важно перед будущей переработкой критериев и демо-данных.

## 12. Предлагаемая очередность devctl-патчей

### Патч 1. Документационный контракт интерфейсной реформы

Цель: зафиксировать целевую структуру вкладок, матрицу ПО/ТО × CAPEX/OPEX и правила экспорта.

Состав:

- добавить этот документ;
- обновить `docs/README.md`;
- не менять runtime-поведение.

Критерий готовности: у команды есть согласованный план, от которого можно делать кодовые патчи без спора о структуре.

### Патч 2. Базовые UI-блоки и панельная инфраструктура

Цель: ввести безопасные переиспользуемые элементы для сворачиваемых панелей.

Состав:

- добавить `CollapsibleSection` или аналог в `src/ui/widgets`;
- добавить лёгкий контейнер рабочей области;
- сохранить совместимость с `BaseScrollableTab`;
- покрыть синтаксической проверкой без `pytest` и без `__pycache__`.

Критерий готовности: существующие вкладки можно начать переносить в блоки без изменения бизнес-логики.

### Патч 3. Каркас новых верхнеуровневых вкладок

Цель: заменить короткие `ПО`/`ТО` на полноценные рабочие области и убрать будущий долг по навигации.

Состав:

- переименовать верхние вкладки в `Программное обеспечение` и `Техническое обеспечение`;
- внутри каждой вкладки создать блоки `Сводка`, `CAPEX`, `OPEX`, `Критерии`, `Аналитика`, `Локальный экспорт`;
- пока оставить старые аналитические вкладки как совместимый fallback, если перенос ещё не завершён.

Критерий готовности: пользователь видит новую структуру, но старые сценарии ещё не сломаны.

### Патч 4. Перенос OPEX внутрь ПО/ТО

Цель: убрать отдельную вкладку `Операционные затраты` с верхнего уровня и распределить её содержимое по областям.

Состав:

- программные подписки перенести в блок `ПО / OPEX`;
- аренду серверов, администрирование и обслуживание перенести в `ТО / OPEX`;
- миграцию и тестирование вынести в блок `Внедрение` с явной привязкой к scope;
- сохранить старые runtime-категории через нормализацию, чтобы не ломать данные.

Критерий готовности: CAPEX/OPEX видны внутри ПО/ТО, а финансовая сводка остаётся прежней или становится более объяснимой.

### Патч 5. Перенос GA/AHP/обоснования внутрь ПО/ТО

Цель: сделать аналитику профильной, а не отдельной от данных.

Состав:

- создать аналитический блок для `scope=software` внутри программной вкладки;
- создать аналитический блок для `scope=technical` внутри технической вкладки;
- перенести запуск GA, AHP, GA + AHP и анализ важности критериев в соответствующие блоки;
- сохранить единый `AnalysisScopeProfileService` как источник правил;
- показывать в блоках последний результат и статус устаревания.

Критерий готовности: пользователь запускает аналитику из той же области, где вводит данные, а не ищет отдельные вкладки.

### Патч 6. Пересборка редактора компонентов и legacy-песочницы

Цель: оставить редактор самостоятельным, но убрать архивную песочницу с верхнего уровня.

Состав:

- добавить в редактор скрываемый блок legacy-конвертации;
- показать связи компонента с ПО/ТО;
- улучшить предпросмотр финансовой и аналитической готовности;
- убрать `ИТ-песочница (архив)` из верхнего `Notebook`, когда конвертация доступна внутри редактора.

Критерий готовности: все старые записи можно обработать, но верхнее меню больше не перегружено архивной вкладкой.

### Патч 7. Умная вкладка экспорта

Цель: превратить экспорт в живую итоговую витрину и конструктор отчётов.

Состав:

- добавить блоки сводки ПО, ТО, TCO, NPV и предупреждений;
- добавить полный Markdown/JSON/CSV-экспорт всего решения;
- добавить частичные экспорты с обязательными контекстными предупреждениями;
- помечать устаревшие GA/AHP/Парето-результаты;
- не пересчитывать тяжёлые методы автоматически без действия пользователя.

Критерий готовности: экспорт объясняет не только цифры, но и полноту/ограниченность выбранного среза.

### Патч 8. Финальная чистка навигации и подготовка к критериям/демо-данным

Цель: закрыть интерфейсную реформу и подготовить следующую большую волну изменений.

Состав:

- удалить верхнеуровневые аналитические вкладки после переноса функций;
- проверить, что целевой список вкладок совпадает с планом;
- обновить screenshots/demo docs;
- составить отдельный план переработки критериев и демо-данных на основе новой структуры.

Критерий готовности: интерфейс стабилизирован, и следующая работа может фокусироваться на содержании критериев, а не на расположении экранов.

## 13. Что важно не делать слишком рано

| Риск | Почему опасно | Как снизить |
|---|---|---|
| сразу удалить старые вкладки GA/AHP | можно потерять рабочие сценарии до завершения переноса | сначала сделать блоки-дубликаты, затем удалить старый верхний уровень |
| смешать ПО и ТО в один общий анализ | нарушится уже введённая логика профилей | запускать аналитику всегда с явным `scope` |
| считать CAPEX/OPEX заменой ПО/ТО | это разные оси классификации | показывать CAPEX/OPEX внутри ПО/ТО |
| автоматически пересчитывать GA на каждое изменение | интерфейс станет медленным и непредсказуемым | хранить последний результат и статус устаревания |
| назвать миграцию обычным OPEX без пояснений | разовые внедренческие работы будут путаться с регулярными платежами | использовать отдельный блок внедрения и TCO-пояснения |
| экспортировать частичный срез без предупреждения | пользователь может принять неполный отчёт за полную рекомендацию | каждый частичный экспорт должен иметь контекстный дисклеймер |

## 14. Связь с будущей переработкой критериев и демо-данных

После интерфейсной реформы критерии и демо-данные нужно перерабатывать уже под новую структуру.

| Будущая работа | Что должна получить от новой UI-архитектуры |
|---|---|
| критерии ПО | отдельный блок программных критериев, где можно объяснить функциональность, поддержку, лицензии, совместимость и стоимость владения |
| критерии ТО | отдельный блок технических критериев, где можно объяснить серверы, сеть, рабочие места, мощность, надёжность и производительность |
| демо-данные ПО | программные CAPEX/OPEX, подписки, несколько альтернатив и понятный программный победитель |
| демо-данные ТО | оборудование, аренда/администрирование, энергия, несколько альтернатив и понятный технический победитель |
| комплексный сценарий | связка ПО + ТО + TCO + NPV + DecisionReport, где видно, что частные победители не всегда равны общему лучшему решению |
| защита диплома | экранная демонстрация должна идти по логике: данные → критерии → методы → финансы → экспорт |

Важно: интерфейсная реорганизация должна подготовить места для новых критериев, но не обязана в том же патче менять сами критерии. Иначе смешаются две большие реформы, и будет сложнее проверить, что именно сломало результат.

## 15. Итоговое целевое правило

После завершения реформы пользователь должен мыслить не набором разрозненных вкладок, а двумя основными рабочими областями:

- `Программное обеспечение`: что покупается или оплачивается как программная часть, какие критерии важны для ПО, какие программные альтернативы предлагают GA/AHP/Парето.
- `Техническое обеспечение`: какое оборудование или технические ресурсы используются, какие расходы они создают, как они влияют на энергию, рабочие места и техническую оптимизацию.

`Редактор компонентов`, `Электроэнергия`, `NPV-анализ` и `Экспорт` остаются отдельными, потому что это не конкурирующие предметные области, а сквозные инструменты ввода, расчёта и объяснения итогового решения.
