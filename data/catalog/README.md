# Каталог оборудования

Этот каталог описывает **стабильную границу** между основным приложением и вспомогательным инструментом парсинга.

## Идея

В проекте принята следующая схема:
- основное приложение **не запускает** парсер напрямую;
- парсер живёт отдельно в `tools/catalog_parser/`;
- результат его работы — нормализованный JSON-каталог оборудования.

## Где лежат данные

- примеры сырых снимков парсинга: `data/examples/parser/`
- пример нормализованного каталога: `data/examples/catalog/normalized_dns_catalog.json`
- рабочий выходной файл инструмента: `data/generated/catalog/equipment_catalog.json`

## Для чего это нужно

Такая схема заранее готовит проект к будущей интеграции кнопки вида **«Загрузить данные»**:
1. GUI или отдельный use case запускает инструмент обновления каталога;
2. парсер собирает и нормализует данные;
3. приложение затем читает уже готовый каталог, а не тянет сетевой парсинг в свой runtime.

## Почему это важно

Это позволяет не смешивать:
- расчётно-аналитическое приложение;
- сетевой сбор данных;
- исследовательские эксперименты по парсингу.
