# Будущая идея: вариант C — универсальный редактор SolutionComponent

Статус документа: **идея будущего развития после завершения conceptual cohesion roadmap**.  
Документ не отменяет решение Этапа 7, где старая вкладка `ИТ-инфраструктура` стабилизирована как вспомогательная песочница. Здесь зафиксирован более сильный вариант развития — **вариант C: превратить старую вкладку в универсальный редактор `SolutionComponent`**, но рассматривать его только как отдельную будущую инициативу после того, как вся дорожная карта концептуальной связанности уже доведена до рабочего состояния.

Детальный пошаговый roadmap реализации этой инициативы вынесен в `solution_component_editor_roadmap.md`.

## Исходная позиция

На Этапе 7 выбран безопасный вариант A: старая вкладка остаётся песочницей для свободных пользовательских статей. Это решение снижает риск появления второй скрытой модели данных рядом с нормализованными вкладками `ПО`, `ТО`, `OPEX`, `Электроэнергия`, `NPV`, `AHP`, `GA`, `GA + AHP` и `Экспорт`.

Вариант C предлагает более амбициозное продолжение: не просто хранить свободные статьи, а превратить вкладку в универсальный редактор компонентов решения. Такой шаг имеет смысл не сейчас, а после того, как проект уже обладает единой цепочкой:

```text
runtime-компоненты
  → scope / component_type
  → AnalysisScopeProfile
  → CandidateConfiguration
  → CostModel / TCO
  → NPV
  → AHP / GA / GA + AHP
  → DecisionReport
```

То есть вариант C должен опираться не на старую свободную таблицу, а на уже выстроенную общую предметную модель.

## Главная идея варианта C

Вариант C превращает вкладку `ИТ-песочница` в редактор нормализованных компонентов решения. Пользователь получает не просто возможность записать произвольную статью, а возможность создать полноценный элемент будущей конфигурации:

```text
SolutionComponent = предметный компонент ИТ-решения,
который имеет область, тип, стоимость, метрики,
ограничения, источник происхождения и правила участия в аналитике.
```

В таком виде вкладка становится не конкурентом `ПО` и `ТО`, а надстройкой над ними. Она нужна для случаев, когда компонент не помещается в стандартную форму или когда пользователь хочет собрать смешанный элемент: например, комплект оборудования с лицензией, сервисом внедрения и эксплуатационными расходами.

## Чем SolutionComponent отличается от текущей sandbox-записи

| Признак | Sandbox-запись варианта A | `SolutionComponent` варианта C |
|---|---|---|
| Назначение | Черновая пользовательская статья | Нормализованный компонент решения |
| Участие в аналитике | По умолчанию не участвует | Может участвовать после валидации |
| `scope` | Не обязателен | Обязателен или выводится из шаблона |
| `component_type` | Не обязателен | Обязателен или выводится из шаблона |
| Стоимость | Свободное поле статьи | Разделена на CAPEX, OPEX, внедрение, сопровождение, энергию |
| Метрики | Не структурированы | Привязаны к профилю ПО/ТО/сводному профилю |
| TCO | Не считается напрямую | Считается через `CostModel` |
| DecisionReport | Показывается как вспомогательная строка | Показывается как полноценный компонент альтернативы |
| Риск | Может быть смысловой заметкой | Должен проходить нормализацию и проверки |

Главное отличие: `SolutionComponent` не должен быть просто более сложной строкой формы. Это связующее звено между пользовательским вводом и общей системой выбора ИТ-решения.

## Предполагаемая модель компонента

На уровне идеи компонент может включать следующие группы полей.

### 1. Идентификация и происхождение

| Поле | Смысл |
|---|---|
| `id` | стабильный идентификатор компонента |
| `name` | человекочитаемое название |
| `description` | краткое пояснение назначения |
| `origin` | источник: manual, demo, catalog, imported, converted_sandbox |
| `source_category` | исходная категория runtime-данных, если компонент создан из старого формата |
| `tags` | дополнительные пометки для поиска и группировки |

### 2. Предметная классификация

| Поле | Смысл |
|---|---|
| `scope` | область применения: software, hardware, mixed, service, infrastructure |
| `component_type` | тип компонента: server, workstation, network, license, subscription, service, migration, support и т.п. |
| `lifecycle_role` | роль в жизненном цикле: purchase, implementation, operation, renewal, support |
| `analysis_profile` | профиль анализа, если компонент должен участвовать в ПО/ТО/сводной аналитике |

### 3. Стоимость и финансовая роль

| Поле | Смысл |
|---|---|
| `purchase_cost` | стоимость покупки |
| `implementation_cost` | стоимость внедрения |
| `migration_cost` | стоимость миграции |
| `testing_cost` | стоимость тестирования |
| `monthly_cost` | регулярные ежемесячные расходы |
| `annual_cost` | регулярные годовые расходы |
| `energy_cost` | производная или заданная стоимость электроэнергии |
| `cost_assumptions` | пояснения к расчёту стоимости |

Эти поля должны попадать в `CostModel` и TCO, а не жить отдельно внутри формы.

### 4. Технические и пользовательские метрики

| Поле | Смысл |
|---|---|
| `client_seats` | число пользовательских рабочих мест, если применимо |
| `performance_score` | условная производительность или качество |
| `reliability_score` | надёжность или устойчивость |
| `maintainability_score` | удобство сопровождения |
| `security_score` | уровень защищённости |
| `power_watts` | мощность, если компонент технический |
| `usage_profile` | режим использования для энергетического расчёта |

Набор метрик не должен быть одинаковым для всех компонентов. Он должен интерпретироваться через `AnalysisScopeProfile`, иначе универсальная форма быстро превратится в перегруженную таблицу.

### 5. Участие в аналитике

| Поле | Смысл |
|---|---|
| `strict_analysis_participation` | может ли компонент участвовать в строгой аналитике |
| `candidate_eligible` | может ли компонент входить в `CandidateConfiguration` |
| `tco_eligible` | учитывается ли компонент в TCO |
| `energy_eligible` | участвует ли компонент в расчёте электроэнергии |
| `npv_eligible` | может ли компонент попадать в финансовый мост NPV |
| `validation_warnings` | предупреждения нормализации |

Эти признаки нужны для защиты от ситуации, когда пользователь ввёл красивую, но методически неполную строку, а приложение автоматически использовало её как строгий компонент решения.

## Роль вкладок ПО и ТО после варианта C

Вариант C не должен уничтожать вкладки `ПО` и `ТО`. Они остаются основными быстрыми формами для типовых сценариев.

| Вкладка | Роль после варианта C |
|---|---|
| `ПО` | быстрый ввод типовых программных компонентов и лицензий |
| `ТО` | быстрый ввод технических компонентов и оборудования |
| `OPEX` | ввод регулярных услуг, сопровождения, подписок и эксплуатационных расходов |
| `Электроэнергия` | расчёт энергетической части для технических компонентов |
| `Редактор компонентов` | расширенный режим для нестандартных, смешанных и составных компонентов |

То есть вариант C не должен превращать проект в одну огромную универсальную форму. Лучше рассматривать его как **advanced-режим**: обычный пользователь работает с привычными вкладками, а сложные случаи дооформляет в редакторе компонентов.

## Как вариант C ложится на модель после завершения roadmap

### После Этапа 1–2

У проекта уже есть явные `scope` и `component_type`. Значит, редактор может не угадывать смысл компонента только по названию категории, а требовать или предлагать явную классификацию.

### После Этапа 3

Существуют профили области анализа. Значит, редактор может показывать пользователю, какие поля обязательны для ПО, какие для ТО, а какие нужны только в сводном сценарии.

### После Этапа 4

Появляется общий `CandidateConfiguration`. Значит, компонент редактора можно безопасно превращать в часть альтернативы, если он прошёл нормализацию.

### После Этапа 5

Есть TCO-модель и мост к NPV. Значит, компонент редактора должен отдавать стоимость в `CostModel`, а не хранить её как независимую сумму.

### После Этапа 6

Есть `DecisionReport`. Значит, все компоненты, созданные через редактор, должны иметь объяснимое происхождение и попадать в отчёт вместе с предупреждениями и допущениями.

### После Этапа 8

Демо-данные и контрольные сценарии уже отражают общую модель. Значит, вариант C можно показывать на демонстрационных данных без ощущения, что это отдельный экспериментальный режим.

### После Этапа 9

Документация синхронизирована с фактическим кодом. Значит, внедрение варианта C можно делать как осознанное расширение архитектуры, а не как попытку закрыть старый долг.

## Предлагаемый план реализации варианта C

### Патч C1. Документационный контракт SolutionComponent

Статус: выполнен отдельным документационным контрактом `solution_component_contract.md` без изменения runtime-кода.

Цель: зафиксировать контракт компонента до изменения кода.

- описать обязательные и необязательные поля `SolutionComponent`;
- определить связь с `CandidateConfiguration`;
- определить связь с `CostModel`;
- определить связь с `AnalysisScopeProfile`;
- описать, какие sandbox-записи можно конвертировать;
- явно указать, что вариант C не заменяет вкладки `ПО` и `ТО`.

Критерий завершения: по документу понятно, какие поля нужны компоненту, какие поля вычисляются, а какие остаются пользовательскими комментариями.

### Будущий патч C2. Доменная модель и прикладная нормализация

Цель: добавить модель и сервисы без изменения пользовательского интерфейса.

- добавить доменный `SolutionComponent`;
- добавить `SolutionComponentNormalizationService`;
- добавить проверку обязательных полей по `scope` и `component_type`;
- добавить преобразование `SolutionComponent → CandidateConfiguration`;
- добавить преобразование `SolutionComponent → CostModel`;
- сохранить совместимость с текущими runtime-структурами.

Критерий завершения: компонент можно создать программно, провалидировать и включить в общий аналитический поток без ручных обходных адаптеров.

### Будущий патч C3. UI-редактор компонентов в режиме MVP

Цель: дать пользователю форму, но не перегрузить её всеми возможными полями сразу.

- переименовать `ИТ-песочницу` в `Редактор компонентов` или `Компоненты решения`;
- добавить выбор `scope`;
- добавить выбор `component_type`;
- показывать разные группы полей по выбранному профилю;
- отдельно показывать стоимость, метрики и пояснения;
- показывать предупреждения нормализации до сохранения;
- оставить режим черновика для неполных компонентов.

Критерий завершения: пользователь может создать компонент, который явно отличается от свободной заметки и может быть безопасно использован в аналитике.

### Будущий патч C4. Интеграция с аналитикой

Цель: подключить компоненты редактора к уже существующей цепочке выбора.

- разрешить валидным компонентам попадать в `CandidateConfiguration`;
- учитывать стоимость компонента в TCO;
- учитывать энергетические поля в расчёте электроэнергии;
- передавать финансовую часть в NPV;
- отображать компоненты в `DecisionReport`;
- показывать, какие компоненты были исключены из анализа и почему.

Критерий завершения: компонент, созданный через редактор, проходит путь от UI до итогового отчёта без отдельной параллельной модели данных.

### Будущий патч C5. Конвертация старых sandbox-записей

Цель: аккуратно связать старые свободные статьи с новой моделью.

- добавить действие «Преобразовать в компонент»;
- предложить пользователю выбрать `scope` и `component_type`;
- перенести старую сумму в подходящее финансовое поле;
- сохранить исходное название статьи в `origin` / `source_category`;
- оставить неполные записи черновиками;
- не конвертировать всё автоматически без подтверждения.

Критерий завершения: старые записи можно постепенно переводить в строгую модель, не ломая пользовательские данные.

### Будущий патч C6. Обновление демонстрации и защитных материалов

Цель: сделать вариант C объяснимым для диплома и для будущей разработки.

- обновить сценарий демонстрации;
- показать пример составного компонента;
- показать отличие обычных вкладок от расширенного редактора;
- обновить `DecisionReport`-пример;
- добавить раздел в документацию о том, почему редактор не создаёт вторую модель данных.

Критерий завершения: вариант C можно объяснить как развитие уже связанной архитектуры, а не как очередную отдельную вкладку.

## Пример пользовательского сценария

```text
1. Пользователь открывает расширенный редактор компонентов.
2. Создаёт компонент «Комплект сервер + гипервизор + базовая настройка».
3. Выбирает scope = mixed.
4. Выбирает component_type = infrastructure_bundle.
5. Указывает CAPEX, внедрение, сопровождение и мощность.
6. Система показывает предупреждение: для AHP нужны оценки надёжности и сопровождаемости.
7. Пользователь заполняет недостающие метрики.
8. Компонент становится candidate_eligible = true.
9. Компонент попадает в CandidateConfiguration.
10. TCO учитывает покупку, внедрение, сопровождение и электроэнергию.
11. GA/AHP сравнивают конфигурацию с другими вариантами.
12. DecisionReport показывает компонент, его происхождение, стоимость, метрики и ограничения.
```

Такой сценарий полезен, когда реальное ИТ-решение состоит не только из отдельных строк оборудования и ПО, а из связанного комплекта.

## Риски идеи

| Риск | Почему важен | Как снижать |
|---|---|---|
| Перегрузка интерфейса | Универсальная форма может стать слишком сложной | Делить поля на базовые и расширенные |
| Повторение вкладок ПО/ТО | Редактор может дублировать уже понятные формы | Оставить ПО/ТО быстрыми сценариями, а редактор — advanced-режимом |
| Некорректное участие в аналитике | Неполный компонент может исказить GA/AHP/TCO | Вводить `candidate_eligible`, warnings и строгую валидацию |
| Смешение CAPEX/OPEX | Пользователь может записать все расходы одной суммой | Разделять финансовые поля и показывать подсказки |
| Нарушение обратной совместимости | Старые sandbox-записи могут не соответствовать новой модели | Использовать ручную конвертацию и режим черновика |
| Размывание дипломной логики | Проект может выглядеть как конструктор всего подряд | Подчёркивать связь с выбором ИТ-решения и DecisionReport |

## Что не входит в вариант C

Вариант C не должен превращаться в полноценную CMDB, ERP-учёт, складскую систему или каталог закупок. Его граница уже:

```text
редактор компонентов для задачи выбора и обоснования ИТ-решения
```

Он не обязан хранить весь жизненный цикл активов после внедрения. Для дипломного проекта достаточно, чтобы компонент мог участвовать в расчётах стоимости, сравнении альтернатив и итоговом объяснении выбора.

## Критерии готовности будущей реализации

Вариант C можно считать реализованным, если выполняются следующие условия:

- пользователь может создать нормализованный компонент с `scope` и `component_type`;
- компонент проходит валидацию по профилю анализа;
- стоимость компонента попадает в TCO;
- технические параметры могут участвовать в энергетическом расчёте;
- компонент может быть частью `CandidateConfiguration`;
- GA/AHP/GA + AHP работают с ним через общий пул альтернатив;
- NPV получает финансовые потоки не вручную, а через общий финансовый мост;
- `DecisionReport` показывает компонент, его происхождение, вклад в стоимость, ограничения и предупреждения;
- старые sandbox-записи можно конвертировать вручную и безопасно;
- вкладки `ПО` и `ТО` остаются понятными быстрыми сценариями, а не исчезают под универсальной формой.

## Короткая формулировка для будущей защиты

Вариант C можно объяснять так:

> После стабилизации общей предметной модели старая песочница может быть развита в универсальный редактор компонентов решения. Такой редактор не создаёт отдельную модель данных, а использует уже существующие `scope`, `component_type`, профили анализа, `CandidateConfiguration`, TCO, NPV и `DecisionReport`. Благодаря этому нестандартные или составные элементы ИТ-решения можно вводить гибко, но всё равно проводить через общую цепочку расчётов и объяснения выбора.

## Итог

Вариант C — логичное будущее развитие, но только после завершения основной roadmap. Если реализовать его слишком рано, он рискует вернуть проект к разрозненным формам и свободным статьям. Если реализовать его после стабилизации общей модели, он становится сильным расширением: позволяет описывать нестандартные компоненты, не разрушая связанность данных, аналитики и итогового отчёта.
