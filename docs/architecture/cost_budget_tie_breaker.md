# Стоимость как бюджетный фильтр и tie-breaker

## Что изменено

Базовая оптимизация больше не использует `capital_cost` как обычный мягкий критерий с весом в профилях `ПО` и `ТО`. Стоимость остаётся важной, но её роль разделена на три понятных уровня:

1. **Бюджет** — жёсткий фильтр `max_budget`: варианты выше бюджета не допускаются.
2. **Отчётность** — стоимость сохраняется в `totals`, таблицах результата, экспорте и будущих TCO/NPV-блоках.
3. **Экономия** — GA может предпочесть более дешёвый вариант только среди решений, близких по основной нормализованной оценке.

## Зачем это нужно

До патча стоимость могла одновременно работать как ограничение бюджета и как отдельный вес критерия. Методически это спорно: пользователь уже ограничил допустимую область бюджетом, поэтому повторное сильное наказание за стоимость может скрыто вытеснять более качественные решения.

Новая модель проще объясняется на защите:

```text
сначала отсекаем всё дороже бюджета;
затем сравниваем допустимые варианты по профильным критериям качества;
если качество почти одинаковое — выбираем более дешёвый.
```

## Как работает tie-breaker

После завершения GA сервис смотрит на `top_solutions`. Если несколько вариантов находятся в пределах малого разрыва основной оценки, выбирается вариант с меньшей `capital_cost`. По умолчанию используется порог `0.02` на нормализованной шкале `0..1`.

Это означает, что дешёвый вариант не может победить заметно более качественное решение только за счёт цены. Экономия применяется только как вторичный фактор.

## Что осталось неизменным

- Поле `Бюджет` в UI остаётся.
- Варианты выше бюджета по-прежнему нарушают hard constraint.
- Стоимость отображается в составе найденного решения.
- Старые расширенные/ручные сценарии, где пользователь явно передал `capital_cost` как критерий, не запрещаются.

## Основные файлы

| Файл | Роль |
|---|---|
| `src/application/services/analysis_scope_profile_service.py` | `capital_cost` удалён из базовых критериев ПО/ТО, но оставлен в ограничении бюджета и metric extractors |
| `src/application/services/genetic_optimization_service.py` | применяет cost tie-breaker после GA top-N |
| `src/ui/tabs/genetic_optimization_tab.py` | убирает поле веса стоимости и показывает настройку «предпочитать дешевле при близком качестве» |
| `src/application/services/scoped_candidate_pool_service.py` | Pareto-кейс строится по профильным критериям без стоимости как soft-критерия |

## Ограничения

- Tie-breaker работает по top-N решениям, которые вернул GA, а не по полной базе всех теоретически возможных вариантов.
- Порог близости пока фиксирован в прикладном слое как простой MVP-параметр.
- Полноценный расширенный режим «стоимость как отдельный критерий» можно добавить позже, но он должен быть явно включаемым, а не базовым поведением.
