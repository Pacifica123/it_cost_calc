# План поэтапной замены Tkinter/ttkbootstrap UI на PySide6/Qt

## 1. Вердикт и цель

Цель миграции — полностью заменить текущий интерфейс Tkinter/ttkbootstrap на новый PySide6/Qt UI, не переписывая расчётную и прикладную логику проекта.

Итог последнего патча серии должен быть однозначным: основной запуск приложения открывает новый Qt-интерфейс, а старый Tkinter UI удалён из runtime-пути либо оставлен только как архивный legacy-слой, не используемый приложением.

Основание для такого пути:

- текущая логика уже отделена от UI через `domain/`, `application/services/`, `application/use_cases/`, `infrastructure/storage/`, `infrastructure/exporters/`;
- самые проблемные места находятся именно в `src/app.py`, `src/ui/**` и Tk-зависимых адаптерах;
- пример `anime_rerank_tournament` показывает подходящий стек PySide6: `QMainWindow`, `QWidget`, layout-ы, `QTableWidget`, `QSS`, отдельные окна и переиспользуемые виджеты;
- Qt лучше подходит для сложного desktop-интерфейса: темы, таблицы, подсказки, адаптивные layout-ы, hotkeys, состояния, валидаторы, модальные окна и быстрые перерисовки.

## 2. Нельзя нарушать при проектировании UI

### 2.1. Плотность интерфейса

Видимый интерфейс должен быть намеренно разреженным.

Обязательные правила:

- ни один видимый label не должен быть длиннее 10 слов;
- 10 слов — абсолютный максимум, а не целевая длина;
- предпочтение — 1–4 слова на label;
- длинные объяснения уходят в tooltip, справку, боковую подсказку или раскрываемый блок;
- необязательные параметры скрываются до момента, когда они нужны;
- кнопок на экране должно быть мало;
- главный action экрана должен быть визуально очевиден;
- secondary-actions уходят в меню `Ещё`, toolbar, раскрываемый блок или context-menu;
- пользователь не должен выбирать между несколькими одинаково важными кнопками.

### 2.2. Последовательность действий

Интерфейс должен подсказывать оптимальный порядок работы.

Базовая логика для вкладок ПО/ТО:

1. Данные.
2. Кандидаты GA.
3. AHP-оценка.
4. Pareto-анализ.
5. Гибридный итог.
6. Экспорт/отчёт.

Эта последовательность должна быть видна как маршрут, а не как набор случайных панелей.

Для каждого экрана:

- сверху — название текущего шага;
- рядом — короткий статус;
- справа или снизу — главный следующий шаг;
- недоступные шаги объясняют причину, но коротко;
- сложные параметры открываются только через `Настроить`, `Параметры`, `Подробнее`.

### 2.3. Формат экрана

Возвращаемся к моноэкранной модели.

Запрещаем как базовый подход:

- постоянные split-панели;
- несколько равноправных областей на одном экране;
- Adobe-подобные двигающиеся панели;
- интерфейс, где пользователь должен вручную растягивать смысловые зоны;
- длинные одновременно видимые инструкции.

Разрешаем:

- один большой экран под одну смысловую задачу;
- многоуровневую навигацию;
- скрываемые панели настроек;
- компактные summary-карточки;
- таблицы только там, где без них нельзя;
- пустые состояния с короткой подсказкой;
- wizard/stepper-подход внутри ПО/ТО.

### 2.4. Темы, стиль, шрифты

Основная тема остаётся светлой.

Обязательные темы:

- светлая мягкая;
- тёмная мягкая.

Обе темы не должны резать глаза.

Стиль:

- flat design как основа;
- лёгкий неоморфизм только через форму, радиусы, мягкие поверхности;
- без тяжёлых теней;
- без декоративной перегрузки;
- спокойный accent-color;
- высокий контраст текста;
- одинаковые отступы;
- предсказуемые размеры кнопок.

Рекомендуемый базовый шрифт:

- Windows: `Segoe UI`;
- fallback: `Arial`;
- основной кегль: 10–11 pt для desktop;
- заголовки: 14–18 pt;
- таблицы: 9–10 pt;
- статусные подписи: 9–10 pt.

### 2.5. Адаптивность

Интерфейс должен нормально жить при изменении размера окна.

Правила:

- root navigation не ломается при сужении;
- вторичные действия сворачиваются в `Ещё`;
- длинные таблицы получают горизонтальный/вертикальный scroll;
- summary-карточки перетекают в 1–2 колонки;
- правые вспомогательные блоки уходят вниз или скрываются;
- необязательные элементы могут превращаться в иконки с tooltip;
- критически важные действия не скрываются.

### 2.6. Производительность

Новый UI должен стартовать быстро и без видимых пролагов.

Правила:

- не строить все тяжёлые экраны на старте;
- использовать lazy-loading вкладок;
- не пересчитывать GA/AHP/Pareto при каждом переключении;
- не заполнять большие таблицы синхронно без необходимости;
- тяжёлые расчёты выносить из GUI-потока;
- показывать короткий статус выполнения;
- избегать повторной полной перерисовки экранов.

## 4. Главный UX-маршрут будущего приложения

Корневые разделы остаются:

1. `Программное обеспечение`.
2. `Техническое обеспечение`.
3. `Редактор компонентов`.
4. `Электроэнергия`.
5. `NPV-анализ`.
6. `Экспорт`.
7. `Настройки` — новый компактный раздел или кнопка в верхней панели.

Внутри ПО/ТО вместо множества раскрытых панелей должен быть один маршрут:

```text
Данные → GA → AHP → Pareto → Гибрид
```

Каждый шаг занимает один основной экран.

Пример для `Техническое обеспечение`:

- экран `Данные`: активы, OPEX, краткий итог, кнопка `К GA`;
- экран `GA`: минимум параметров, запуск, список кандидатов, кнопка `Передать в AHP`;
- экран `AHP`: ранжирование кандидатов, проверка согласованности, кнопка `К Pareto`;
- экран `Pareto`: доминирование, критерии, кнопка `К гибриду`;
- экран `Гибрид`: итоговая витрина, объяснение лидера, кнопка `В экспорт`.

Программное обеспечение использует тот же маршрут, но с профилем ПО.

## Патч 01 — `qt-migration-contract`

### Цель

Зафиксировать правила миграции и защитить проект от повторения проблем старого UI.

### Что меняется

- Добавляется документ с UX-контрактом миграции.
- Добавляется чек-лист UI-ограничений.
- Добавляется простой статический сканер видимых текстов.
- Добавляются правила для будущих Qt-экранов:
  - label не длиннее 10 слов;
  - длинный текст только в tooltip/help;
  - один экран — одна смысловая задача;
  - root-разделы сохраняются;
  - расчётная логика не переносится в UI.

Минимальные. Патч документационный и контрольный.

### Проверки

- `python -B tools/check_ui_text_density.py`

### Критерий готовности

## Патч 02 — `qt-bootstrap-shell`

### Цель

Добавить PySide6 как параллельный UI-стек, не ломая текущий Tkinter-запуск.

### Что меняется

- В `pyproject.toml` добавляется зависимость `PySide6`.
- Создаётся минимальный Qt bootstrap.
- Создаётся пустое главное окно Qt.
- Старый Tkinter UI остаётся основным.
- Новый Qt UI запускается отдельной командой или флагом.

### Основные файлы

- `pyproject.toml`
- `src/ui_qt/app.py`
- `src/ui_qt/main_window.py`
- `scripts/run_qt_app.py`
- `src/bootstrap.py`

### Риски

- Увеличится размер окружения из-за PySide6.
- На машинах без GUI Qt-запуск может требовать offscreen-режим для тестов.

### Проверки

- импорт `PySide6`;
- создание `QApplication` в smoke-режиме;
- старый `scripts/run_app.py` продолжает запускаться;
- `pytest` по логике проходит без необходимости открывать GUI.

### Критерий готовности

В проекте есть работающий пустой Qt-контур, но функциональность ещё не перенесена.

## 6. Что важно не забыть при реализации

### 6.1. Состояния экранов

Для каждого экрана нужны состояния:

- пусто;
- данные есть;
- расчёт идёт;
- расчёт завершён;
- ошибка;
- результат устарел;
- нужен предыдущий шаг.

Это лучше, чем длинные инструкции на экране.

### 6.2. Ошибки и подсказки

Ошибки должны быть короткими и действенными.

Плохо:

```text
Не удалось выполнить расчёт из-за отсутствия набора данных, необходимого для построения кандидатных конфигураций.
```

Лучше:

```text
Нет исходных данных.
```

А рядом action:

```text
Добавить данные
```

Подробности — в `Подробнее`.

### 6.3. Горячие клавиши

Нужны не сразу, но ближе к финалу:

- `Ctrl+L` — загрузить демо;
- `Ctrl+S` — сохранить/экспортировать текущий результат, где применимо;
- `Ctrl+Enter` — запустить основной расчёт экрана;
- `Alt+Left/Right` — шаг назад/вперёд в маршруте;
- `F1` — помощь по текущему экрану;
- `Esc` — закрыть диалог/панель.

### 6.4. Настройки

Минимальные настройки:

- тема: светлая/тёмная/системная, если получится;
- масштаб интерфейса;
- стартовый раздел;
- показывать расширенные параметры по умолчанию: да/нет.

Настройки не должны быть отдельным сложным приложением внутри приложения.

### 6.5. Проверка текстовой плотности

В каждом UI-патче должен быть обязательный чек:

```text
Нет видимых label длиннее 10 слов.
```

Но лучше проверять не только label, а все видимые тексты:

- labels;
- buttons;
- group titles;
- tab titles;
- empty states;
- status strips;
- placeholders.

Исключения допустимы только для:

- tooltip/help;
- detailed dialog;
- экспортируемых отчётов;
- логов;
- документации.

### 6.6. Проверка производительности

Нужны хотя бы простые метрики:

- время импорта Qt UI;
- время создания main window;
- время открытия первого экрана;
- время загрузки demo dataset;
- отсутствие повторного тяжёлого построения скрытых экранов.

Цель: пользователь не должен видеть долгую прорисовку и зависания.

## 7. Итоговая стратегия

Правильный путь — не пытаться чинить старый Tkinter UI бесконечно, а строить новый Qt UI параллельно.

Рекомендуемая стратегия:

1. Сначала контракт, дизайн-система и Qt-каркас.
2. Потом перенос простых экранов: компоненты, энергия, NPV, экспорт.
3. Потом перенос сложного маршрута ПО/ТО.
4. Потом performance/responsive/settings.
5. Потом переключение Qt как основного UI.
6. Потом удаление Tkinter legacy.

Так проект не превращается в хаотичный большой переписанный коммит, а каждый devctl-патч остаётся проверяемым и откатываемым.
