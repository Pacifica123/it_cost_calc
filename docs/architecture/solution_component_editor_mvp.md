# MVP пользовательского редактора компонентов решения

Документ фиксирует результат этапа C4 из `solution_component_editor_roadmap.md`: в приложении появился минимальный UI-редактор для `SolutionComponent`, который работает поверх runtime-хранилища C3 и не заменяет существующие вкладки ТО/ПО/OPEX.

## Что добавлено

В интерфейсе появляется отдельная вкладка `Редактор компонентов`. Старая `ИТ-песочница` не удаляется и переименована в `ИТ-песочница (архив)`, чтобы пользователь по-прежнему мог открыть старые вспомогательные статьи, но не путал их с новой структурированной моделью.

Новая вкладка предназначена для advanced-сценария: пользователь описывает часть будущего ИТ-решения как компонент, заранее видит статус нормализации и понимает, попадёт ли компонент в строгую аналитику.

## Минимальная форма

MVP-форма содержит только поля, необходимые для безопасного перехода от свободной записи к аналитической сущности:

- ID компонента, который может быть сгенерирован автоматически из названия;
- название компонента;
- назначение или краткое описание;
- `scope` компонента;
- `component_type` компонента;
- количество;
- стоимость покупки;
- стоимость внедрения, миграции и тестирования;
- ежемесячные, ежегодные и энергетические расходы;
- профильные метрики;
- допущения и комментарии;
- флаг участия в строгой аналитике.

Если флаг строгой аналитики снят, компонент сохраняется как черновик. Такой компонент виден в runtime-хранилище и отчёте, но не становится расчётной строкой для `CandidateConfiguration`.

## Профильные поля

Форма меняется в зависимости от выбранного профиля.

| Профиль | Что показывает MVP |
| --- | --- |
| `technical` | мощность, рабочие места, производительность, надёжность, часы работы и рабочие дни |
| `software` | лицензии/пользователи, функциональность, совместимость и поддержка |
| `implementation` | трудозатраты, сложность миграции, покрытие тестированием и длительность |
| `mixed` | размер пакета, риск интеграции и приоритет разбиения |

Для `mixed` интерфейс дополнительно показывает предупреждение: такой компонент лучше разложить на отдельные части ТО, ПО и внедрения, иначе он может стать слишком крупной и неоднозначной аналитической единицей.

## Предпросмотр нормализации

До сохранения пользователь может нажать `Проверить нормализацию`. Предпросмотр показывает:

- текущее состояние нормализации;
- попадёт ли компонент в `CandidateConfiguration`;
- попадёт ли стоимость в TCO;
- участвует ли компонент в энергетическом расчёте;
- готов ли компонент для NPV;
- блокирующие причины и предупреждения.

Это снижает риск, что пользователь заполнит форму, сохранит компонент и только потом узнает, что он не участвует в аналитике.

## Хранение

UI не пишет данные в отдельную параллельную структуру. Все строки сохраняются через `SolutionComponentRuntimeService` в runtime-секцию `solution_components`. Поэтому C4 не ломает контракт C3: строгие компоненты, черновики и исключённые строки по-прежнему разделяются сервисным слоем.

## Границы MVP

Этап C4 не подключает компоненты редактора к полному финансовому циклу TCO/NPV сверх уже подготовленных адаптеров C2-C3. Глубокая финансовая интеграция относится к C5. Также C4 не выполняет массовую конвертацию старых sandbox-записей — это отдельный будущий этап C8.

## Проверка

Для headless-проверки добавлен smoke-сценарий `scripts/run_solution_component_editor_mvp_demo.py`. Он не использует `pytest`, не требует GUI-дисплея и проверяет, что:

- UI-подобные значения собираются в payload `SolutionComponent`;
- строгий технический компонент становится готовым к аналитике;
- программный компонент, сохранённый без строгого участия, остаётся черновиком;
- runtime-снимок правильно считает строгие компоненты и черновики;
- предпросмотр нормализации явно показывает участие или неучастие в `CandidateConfiguration`.
